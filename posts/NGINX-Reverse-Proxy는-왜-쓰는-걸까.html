[NGINX] Reverse Proxy는 왜 쓰는 걸까?

2026.05.17 Velog 원문

서버를 배포하다 보면 Nginx라는 말을 자주 듣게 된다.

특히 이런 구조를 많이 본다.

사용자
  ↓
Nginx
  ↓
Frontend / Backend

처음에는 이런 생각이 든다.

“그냥 사용자가 백엔드 서버로 바로 접속하면 되는 거 아닌가?”

예를 들어 백엔드가 8080번 포트에서 떠 있다면 그냥 이렇게 접속하면 될 것 같다.

http://서버IP:8080

프론트가 3000번 포트에서 떠 있다면 이렇게 접속하면 될 것 같다.

http://서버IP:3000

그런데 실무에서는 보통 이렇게 하지 않는다.

대부분 앞단에 Nginx 같은 Reverse Proxy를 둔다.

Reverse Proxy란?

Reverse Proxy를 한 문장으로 설명하면 이렇다.

클라이언트 요청을 대신 받아서, 내부 서버로 전달하고 관리하기 위해 사용하는 서버

즉 사용자는 내부 구조를 모른다.

사용자 입장에서는 그냥 하나의 주소만 보인다.

https://example.com

하지만 실제 서버 안쪽에는 여러 서비스가 나뉘어 있을 수 있다.

Frontend 서버
Backend 서버
API 서버
여러 컨테이너
DB

Reverse Proxy는 사용자의 요청을 먼저 받은 뒤, 요청의 경로나 조건에 따라 알맞은 내부 서버로 전달한다.

Reverse Proxy가 없는 구조

Reverse Proxy가 없다면 사용자가 각 서버의 포트에 직접 접근해야 한다.

사용자
  ├─ frontend:3000 직접 접속
  ├─ backend:8080 직접 접속
  └─ db:5432 직접 접속

이 구조는 단순해 보이지만 문제가 많다.

포트를 여러 개 외부에 열어야 함
서버 내부 구조가 사용자에게 노출됨
URL 관리가 복잡해짐
보안 위험이 커짐
HTTPS 설정을 서비스마다 따로 해야 할 수 있음

예를 들어 사용자가 프론트는 3000번, 백엔드는 8080번, DB는 5432번 포트로 접근할 수 있다면 운영상 굉장히 위험하다.

특히 DB 포트는 외부에 직접 노출되면 안 된다.

Reverse Proxy를 쓰는 구조

Reverse Proxy를 사용하면 사용자는 Nginx만 바라본다.

사용자
  ↓
Nginx
  ├─ /      → frontend
  └─ /api   → backend

사용자는 이렇게 하나의 주소만 사용한다.

https://example.com

내부에서는 Nginx가 요청을 나눠준다.

/ 요청
→ 프론트엔드로 전달

/api 요청
→ 백엔드로 전달

즉 사용자는 내부에 프론트 서버가 따로 있는지, 백엔드 서버가 따로 있는지, 컨테이너가 몇 개 떠 있는지 알 필요가 없다.

Reverse Proxy를 쓰는 이유

1. 내부 서버를 숨기기 위해

가장 큰 이유는 보안이다.

Reverse Proxy를 쓰면 외부에는 Nginx만 공개하고, 내부 서버는 직접 노출하지 않을 수 있다.

인터넷
  ↓
Nginx만 공개
  ↓
backend:8080은 내부에서만 접근
db:5432는 내부에서만 접근

사용자는 백엔드 서버의 실제 포트나 내부 주소를 알 수 없다.

예를 들어 백엔드가 Docker 컨테이너 안에서 8080번 포트로 떠 있어도, 외부 사용자는 그 포트로 직접 접근하지 않는다.

사용자는 오직 Nginx를 통해서만 요청을 보낸다.

사용자 → Nginx → Backend

이렇게 하면 내부 서버를 보호할 수 있다.

2. URL 라우팅을 하기 위해

Reverse Proxy는 요청 경로에 따라 다른 서버로 요청을 보낼 수 있다.

예를 들어 Nginx 설정을 이렇게 할 수 있다.

location / {
    proxy_pass http://frontend;
}

location /api {
    proxy_pass http://backend;
}

이 설정은 이런 의미다.

/ 로 들어온 요청
→ frontend로 전달

/api 로 들어온 요청
→ backend로 전달

사용자는 하나의 도메인만 사용한다.

https://example.com

하지만 내부적으로는 요청이 나뉜다.

https://example.com/
→ 프론트엔드

https://example.com/api/boards
→ 백엔드 API

이게 Reverse Proxy의 대표적인 역할이다.

3. 포트 노출을 줄이기 위해

Reverse Proxy가 없으면 여러 포트를 외부에 열어야 할 수 있다.

3000번 포트: 프론트엔드
8080번 포트: 백엔드
5432번 포트: DB

하지만 Reverse Proxy를 쓰면 외부에는 보통 80번과 443번만 열어도 된다.

80  → HTTP
443 → HTTPS

나머지 서비스는 내부 네트워크에서만 통신하게 만들 수 있다.

외부 공개:
80, 443

내부 통신:
frontend:3000
backend:8080
db:5432

이렇게 하면 서버 구조가 훨씬 깔끔해진다.

4. HTTPS 처리를 한 곳에서 하기 위해

실무에서는 HTTPS 설정이 중요하다.

그런데 프론트엔드 서버, 백엔드 서버 각각에 인증서를 설정하려고 하면 관리가 복잡해진다.

그래서 보통 HTTPS 인증서는 Nginx에 설정한다.

사용자
  ↓ HTTPS
Nginx
  ↓ HTTP
Backend

이런 구조를 SSL Termination이라고 한다.

즉 외부 사용자는 HTTPS로 안전하게 접근한다.

하지만 Nginx 뒤쪽의 내부 서버들은 HTTP로 통신할 수도 있다.

외부 통신: HTTPS
내부 통신: HTTP

이렇게 하면 인증서 관리가 단순해진다.

5. 정적 파일을 빠르게 처리하기 위해

Nginx는 정적 파일을 처리하는 데 강하다.

정적 파일이란 이런 것들이다.

HTML
CSS
JavaScript
이미지 파일
폰트 파일

React 프로젝트를 빌드하면 보통 정적 파일이 나온다.

index.html
main.js
style.css

이 파일들을 Nginx가 직접 서빙할 수 있다.

React build 결과
→ Nginx가 직접 사용자에게 전달

백엔드 서버가 굳이 정적 파일까지 처리하지 않아도 되기 때문에 역할 분리가 깔끔해진다.

6. 로드밸런싱을 하기 위해

백엔드 서버가 하나만 있을 때는 단순하다.

Nginx → backend1

하지만 트래픽이 많아지면 백엔드 서버를 여러 개 띄울 수 있다.

backend1
backend2
backend3

이때 Nginx가 요청을 나눠줄 수 있다.

upstream backend {
    server backend1:8080;
    server backend2:8080;
    server backend3:8080;
}

location /api {
    proxy_pass http://backend;
}

그러면 요청이 한 서버에만 몰리지 않고 여러 백엔드 서버로 분산된다.

사용자 요청
  ↓
Nginx
  ├─ backend1
  ├─ backend2
  └─ backend3

이것이 로드밸런싱이다.

7. 내부 구조가 바뀌어도 사용자에게 영향이 적음

Reverse Proxy를 사용하면 내부 구조를 바꿔도 사용자의 주소는 그대로 유지할 수 있다.

예를 들어 처음에는 백엔드가 8080번 포트에서 떠 있었다고 하자.

backend:8080

나중에 백엔드 포트를 8081로 바꿔도 사용자는 알 필요가 없다.

Nginx 설정만 바꾸면 된다.

location /api {
    proxy_pass http://backend:8081;
}

사용자는 여전히 같은 주소를 사용한다.

https://example.com/api

즉 Reverse Proxy는 외부 URL과 내부 서버 구조를 분리해준다.

이게 운영에서 매우 중요하다.

Nginx Reverse Proxy 예시

예를 들어 프론트엔드와 백엔드가 각각 다른 컨테이너로 떠 있다고 하자.

frontend 컨테이너: 3000
backend 컨테이너: 8080
nginx 컨테이너: 80

Nginx 설정은 대략 이런 식으로 만들 수 있다.

server {
    listen 80;

    location / {
        proxy_pass http://frontend:3000;
    }

    location /api {
        proxy_pass http://backend:8080;
    }
}

이 설정의 의미는 간단하다.

/ 로 들어온 요청
→ frontend:3000으로 전달

/api 로 들어온 요청
→ backend:8080으로 전달

사용자는 Nginx가 열어둔 80번 포트로만 접근한다.

http://example.com

하지만 내부적으로는 Nginx가 프론트와 백엔드로 요청을 나눠준다.

EC2에서 Reverse Proxy를 쓴다면

EC2에 애플리케이션을 배포한다고 해보자.

서버 안에는 이런 컨테이너들이 떠 있을 수 있다.

nginx
frontend
backend
mysql 또는 postgresql

이때 외부에 모든 포트를 열 필요는 없다.

보통은 Nginx만 외부에 공개한다.

외부 공개:
80
443

내부 통신:
frontend:3000
backend:8080
db:5432

구조는 이렇게 된다.

사용자
  ↓
EC2 Public IP 또는 도메인
  ↓
Nginx
  ├─ frontend 컨테이너
  └─ backend 컨테이너
       ↓
      DB 컨테이너

사용자는 EC2 안에 어떤 컨테이너가 있는지 모른다.

그냥 도메인으로 접속한다.

https://example.com

Nginx가 요청을 받아서 적절한 컨테이너로 넘긴다.

Reverse Proxy와 Forward Proxy 차이

Proxy라는 말 때문에 Forward Proxy와 헷갈릴 수 있다.

둘 다 중간에서 요청을 대신 처리한다는 점은 비슷하다.

하지만 위치와 목적이 다르다.

Forward Proxy

Forward Proxy는 클라이언트 앞에 있다.

클라이언트
  ↓
Forward Proxy
  ↓
인터넷

회사나 학교에서 인터넷 접속을 통제할 때 사용하는 프록시가 대표적이다.

클라이언트가 직접 외부 인터넷에 접근하는 것이 아니라, 프록시 서버를 거쳐서 나간다.

즉 Forward Proxy는 클라이언트를 대신해서 외부 서버에 접근하는 서버다.

Reverse Proxy

Reverse Proxy는 서버 앞에 있다.

사용자
  ↓
Reverse Proxy
  ↓
내부 서버들

사용자는 내부 서버에 직접 접근하지 않는다.

Reverse Proxy가 요청을 대신 받아 내부 서버로 전달한다.

즉 Reverse Proxy는 서버를 대신해서 클라이언트 요청을 받는 서버다.

한 번에 정리하면

Forward Proxy
= 클라이언트 앞에 있음
= 클라이언트를 숨김
= 클라이언트 대신 외부 인터넷 접근

Reverse Proxy
= 서버 앞에 있음
= 서버를 숨김
= 사용자 요청을 받아 내부 서버로 전달

면접에서 이렇게 말하면 좋다

면접에서 “Reverse Proxy를 왜 쓰나요?”라고 물어보면 이렇게 답할 수 있다.

Reverse Proxy는 클라이언트 요청을 대신 받아 내부 서버로 전달하는 역할을 합니다.

보통 Nginx를 앞단에 두고 사용자의 요청을 먼저 받은 뒤, 정적 파일 요청은 프론트엔드로 처리하고 API 요청은 백엔드 서버로 라우팅합니다.

이를 통해 백엔드 서버나 DB 포트를 외부에 직접 노출하지 않을 수 있고, HTTPS 처리, 로드밸런싱, URL 라우팅 같은 운영 기능도 한 곳에서 관리할 수 있습니다.

프로젝트 경험과 연결하면 이렇게 말할 수 있다.

EC2 환경에서 Nginx를 Reverse Proxy로 두고 frontend와 backend 컨테이너를 분리해서 운영하는 구조를 실습했습니다.

사용자의 요청은 먼저 Nginx가 받고, / 요청은 프론트엔드로, /api 요청은 백엔드 컨테이너로 전달하는 방식으로 구성했습니다.

이 구조를 통해 사용자는 하나의 도메인으로 접근하고, 내부 서비스 포트는 외부에 직접 노출하지 않도록 만들 수 있었습니다.

핵심 정리

Reverse Proxy는 사용자의 요청을 대신 받아 내부 서버로 전달하는 서버다.

사용자는 하나의 도메인만 바라본다.

https://example.com

하지만 내부에서는 Nginx가 요청을 나눠준다.

/      → frontend
/api   → backend

Reverse Proxy를 쓰는 이유는 다음과 같다.

내부 서버를 숨기기 위해
포트 노출을 줄이기 위해
URL 라우팅을 하기 위해
HTTPS 처리를 한 곳에서 하기 위해
정적 파일을 빠르게 서빙하기 위해
로드밸런싱을 하기 위해
내부 구조 변경이 사용자에게 드러나지 않게 하기 위해

결국 Reverse Proxy는 단순히 요청을 전달하는 서버가 아니다.

운영 관점에서는 보안, 라우팅, HTTPS, 로드밸런싱, 구조 단순화를 담당하는 서버 앞단의 중요한 진입점이다.
