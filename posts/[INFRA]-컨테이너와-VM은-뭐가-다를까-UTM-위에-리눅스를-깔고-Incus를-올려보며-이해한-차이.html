[INFRA] 컨테이너와 VM은 뭐가 다를까? UTM 위에 리눅스를 깔고 Incus를 올려보며 이해한 차이

2026.05.17 Velog 원문

컨테이너와 VM의 차이를 처음 공부하면 보통 이런 식으로 설명을 듣는다.

VM은 OS 전체를 가상화한다.
컨테이너는 애플리케이션 실행 환경만 격리한다.

또는 이런 그림을 본다.

하드웨어
→ Host OS
→ Hypervisor
→ Guest OS
→ App

하드웨어
→ Host OS
→ Container Runtime
→ Container
→ App

그런데 이 설명만 보면 감이 잘 안 온다. “그래서 실제로 뭐가 다른데?”라는 생각이 든다.

나도 UTM으로 리눅스 VM을 띄우고, 그 위에 Incus를 올려서 컨테이너 비슷한 환경을 다뤄본 적이 있는데, 이때 조금 헷갈렸다.

내 맥북
→ UTM
→ Ubuntu VM
→ Incus
→ 컨테이너

이렇게 보면 컨테이너가 VM 안에 들어가 있으니까 그냥 포함 관계처럼 느껴진다. 하지만 컨테이너와 VM의 차이는 단순히 “누가 누구 안에 있냐”가 아니라, 무엇을 가상화하고, 어디까지 격리하느냐의 차이다.

1. VM은 컴퓨터 한 대를 통째로 흉내 낸다

VM은 Virtual Machine, 즉 가상 머신이다.

쉽게 말하면 내 컴퓨터 안에 또 다른 컴퓨터를 하나 만드는 것이다.

예를 들어 맥북에서 UTM으로 Ubuntu VM을 만들면 이런 구조가 된다.

MacBook 하드웨어
→ macOS
→ UTM
→ Ubuntu VM

여기서 Ubuntu VM은 자기만의 운영체제를 가진다.

Ubuntu VM
├─ Ubuntu 커널
├─ systemd
├─ apt
├─ 파일시스템
├─ 네트워크 인터페이스
└─ 애플리케이션

즉, VM은 거의 독립된 컴퓨터처럼 동작한다.

그래서 Ubuntu VM 안에서는 이런 것들이 가능하다.

sudo apt update
sudo systemctl status ssh
ip addr
df -h

마치 진짜 Ubuntu 서버를 하나 받은 것처럼 사용할 수 있다.

VM의 핵심은 이거다.

하드웨어 수준에 가까운 가상화
운영체제 전체를 포함
각 VM마다 자기 커널을 가짐

그래서 VM은 무겁지만 격리 수준이 강하다.

2. 컨테이너는 OS 전체가 아니라 프로세스를 격리한다

컨테이너는 VM처럼 OS 전체를 새로 띄우는 방식이 아니다.

컨테이너는 기본적으로 호스트 OS의 커널을 공유하면서, 프로세스 실행 환경을 격리하는 방식이다.

예를 들어 Ubuntu 서버 위에서 Docker 컨테이너를 실행한다고 해보자.

Ubuntu Host
→ Docker
→ nginx 컨테이너

이때 nginx 컨테이너 안에 Ubuntu 전체 OS가 새로 뜨는 게 아니다. 컨테이너는 호스트의 Linux 커널을 같이 사용한다.

대신 컨테이너는 다음을 분리해서 자기만의 환경처럼 보이게 만든다.

파일시스템
프로세스 목록
네트워크
사용자/권한
마운트
환경변수

그래서 컨테이너 안에 들어가면 마치 독립된 리눅스처럼 보인다.

docker exec -it nginx-container bash

컨테이너 안에서 보면 /bin, /etc, /usr 같은 디렉토리가 있고, 프로세스도 따로 보인다.

하지만 중요한 건 이것이다.

컨테이너는 자기만의 커널을 갖고 있지 않다.
호스트 Linux 커널을 공유한다.

이게 VM과 가장 큰 차이다.

3. 가장 큰 차이: 커널을 따로 갖느냐, 공유하느냐

컨테이너와 VM의 차이를 한 줄로 정리하면 이거다.

VM은 Guest OS와 커널을 따로 가진다.
컨테이너는 Host OS 커널을 공유한다.

구조를 비교하면 이렇다.

VM 구조

하드웨어
→ Host OS
→ Hypervisor
→ Guest OS 1
   └─ App
→ Guest OS 2
   └─ App

각 VM마다 Guest OS가 있다.

반면 컨테이너는 이렇다.

컨테이너 구조

하드웨어
→ Host OS Kernel
→ Container Runtime
→ Container 1
   └─ App
→ Container 2
   └─ App

컨테이너마다 OS 전체가 있는 게 아니라, 호스트 커널 위에서 격리된 프로세스처럼 실행된다.

그래서 컨테이너는 VM보다 빠르게 뜬다.

VM:
운영체제 부팅 필요

컨테이너:
프로세스 실행에 가까움

4. 부팅 속도 차이

VM을 켜면 운영체제가 부팅된다.

BIOS/UEFI 비슷한 단계
커널 로딩
systemd 시작
서비스 시작
로그인 가능 상태

그래서 VM은 켜지는 데 시간이 걸린다.

반면 컨테이너는 보통 애플리케이션 프로세스를 바로 실행한다.

docker run nginx

이 명령을 치면 nginx 프로세스가 컨테이너 격리 환경 안에서 실행된다.

그래서 컨테이너는 훨씬 빠르게 뜬다.

VM:
컴퓨터 한 대를 켜는 느낌

컨테이너:
격리된 프로세스를 실행하는 느낌

5. 무게 차이

VM은 OS 전체를 포함한다.

그래서 VM 하나를 만들면 디스크도 많이 쓰고, 메모리도 꽤 잡아먹는다.

Ubuntu VM
- Guest OS
- 커널
- 기본 패키지
- systemd
- 로그
- 파일시스템

반면 컨테이너 이미지는 필요한 실행 환경만 포함한다.

예를 들어 nginx 컨테이너 이미지는 nginx를 실행하는 데 필요한 파일만 갖고 있다.

nginx 컨테이너
- nginx 바이너리
- 설정 파일
- 필요한 라이브러리
- 정적 파일

그래서 일반적으로 컨테이너는 VM보다 가볍다.

물론 컨테이너 이미지도 크게 만들면 무거워질 수 있다. 하지만 구조적으로 VM보다 가볍게 만들기 쉽다.

6. 격리 수준 차이

VM은 격리 수준이 강하다.

각 VM은 자기 커널을 갖고 있고, 하이퍼바이저가 VM들을 분리한다.

VM A의 커널
VM B의 커널
Host OS

VM A에서 문제가 생겨도 VM B나 Host로 바로 영향을 주기 어렵다.

반면 컨테이너는 호스트 커널을 공유한다.

컨테이너 A
컨테이너 B
공유 Linux 커널

그래서 컨테이너는 VM보다 가볍고 빠르지만, 격리 수준은 VM보다 약하다고 볼 수 있다.

물론 컨테이너도 namespace, cgroups, capabilities, seccomp, AppArmor 같은 기술로 강하게 격리할 수 있다. 하지만 기본 개념상 VM처럼 OS 전체를 분리하는 방식은 아니다.

7. 네가 해본 UTM + Incus 구조로 이해하기

네가 해본 구조는 아마 이런 느낌이었을 거다.

MacBook
→ UTM
→ Ubuntu VM
→ Incus
→ Linux Container

여기서 UTM은 VM을 띄우는 역할이다.

UTM:
맥북 위에 Ubuntu라는 가상 컴퓨터를 하나 만듦

Ubuntu VM은 자기 커널을 가진다.

Ubuntu VM:
진짜 리눅스 서버처럼 동작

그 위에 Incus를 설치하면, Incus는 그 Ubuntu 커널을 사용해서 컨테이너를 만든다.

Incus 컨테이너:
Ubuntu VM의 Linux 커널을 공유하면서 격리된 사용자 공간을 제공

즉, 네 구조에서 컨테이너는 맥북 커널을 직접 공유하는 게 아니라, UTM 안의 Ubuntu VM 커널을 공유한다.

MacBook 하드웨어
→ macOS 커널
→ UTM
→ Ubuntu VM 커널
→ Incus 컨테이너

이게 중요하다.

컨테이너는 항상 “자기가 올라가 있는 Linux 호스트의 커널”을 공유한다. 네 경우에는 그 Linux 호스트가 실제 물리 서버가 아니라 UTM 안의 Ubuntu VM이었던 것.

그래서 “VM 안에 컨테이너가 들어갔다”가 맞긴 하지만, 본질은 이거다.

VM은 Ubuntu 커널까지 포함한 가상 컴퓨터
Incus 컨테이너는 그 Ubuntu 커널을 공유하는 격리 환경

8. Incus 컨테이너는 Docker 컨테이너랑 뭐가 다를까?

Incus는 LXC/LXD 계열에서 이어진 시스템 컨테이너 관리 도구라고 보면 된다.

Docker는 보통 애플리케이션 컨테이너에 가깝다.

Docker:
nginx 하나
mysql 하나
spring app 하나
이런 식으로 프로세스/앱 단위 실행에 초점

Incus:
작은 리눅스 서버처럼 쓰는 시스템 컨테이너에 가까움
Ubuntu 컨테이너 안에서 systemd, ssh, 여러 프로세스를 다룰 수 있음

그래서 Incus 컨테이너를 쓰면 VM처럼 느껴질 수 있다.

예를 들어 Incus로 Ubuntu 컨테이너를 띄우면:

incus launch images:ubuntu/22.04 my-ubuntu
incus shell my-ubuntu

그 안에서 마치 작은 Ubuntu 서버처럼 명령어를 칠 수 있다.

apt update
systemctl status

그래서 헷갈릴 수 있다.

이거 VM이랑 뭐가 다르지?

차이는 여전히 커널이다.

Incus 컨테이너:
호스트 Linux 커널 공유

VM:
자기 커널 보유

겉으로는 둘 다 리눅스 서버처럼 보일 수 있지만, 내부 구조가 다르다.

9. 컨테이너는 왜 “프로세스”라고 하는가?

컨테이너는 결국 호스트 입장에서 보면 하나의 프로세스 또는 프로세스 그룹이다.

예를 들어 nginx 컨테이너를 실행하면 호스트에서 프로세스를 볼 수 있다.

ps aux | grep nginx

컨테이너 안에서는 nginx가 PID 1처럼 보일 수 있다.

컨테이너 내부:
nginx가 메인 프로세스

호스트 입장:
격리된 namespace 안에서 실행 중인 프로세스

즉, 컨테이너는 “작은 컴퓨터”라기보다는 작은 컴퓨터처럼 보이게 격리된 프로세스 실행 환경에 가깝다.

반면 VM은 정말로 가상 하드웨어와 Guest OS를 가진 “가상 컴퓨터”에 가깝다.

10. 언제 VM을 쓰고 언제 컨테이너를 쓸까?

VM은 이런 경우에 좋다.

서로 다른 OS를 실행해야 할 때
커널까지 완전히 분리하고 싶을 때
보안 격리가 강하게 필요할 때
서버 한 대를 통째로 나눠 쓰고 싶을 때
운영체제 단위 실습이 필요할 때

예를 들어 맥북에서 리눅스 실습을 하고 싶다면 UTM으로 Ubuntu VM을 띄우는 게 자연스럽다.

컨테이너는 이런 경우에 좋다.

애플리케이션을 빠르게 띄우고 싶을 때
배포 환경을 일관되게 만들고 싶을 때
여러 서비스를 가볍게 분리하고 싶을 때
CI/CD에서 테스트 환경을 빠르게 만들고 싶을 때
마이크로서비스를 운영하고 싶을 때

예를 들어 Spring Boot, MySQL, Nginx를 각각 컨테이너로 띄우면 이런 구조가 된다.

EC2 또는 VM
├─ nginx 컨테이너
├─ backend 컨테이너
└─ mysql 컨테이너

여기서 각 컨테이너는 독립된 서버처럼 보이지만, 실제로는 같은 호스트 커널을 공유한다.

11. VM과 컨테이너 비교표

구분 VM 컨테이너
가상화 대상 하드웨어/OS 수준 프로세스/실행 환경 수준
OS 각 VM마다 Guest OS 포함 Host OS 커널 공유
커널 VM마다 별도 커널 호스트 커널 공유
부팅 속도 느린 편 빠른 편
무게 무거운 편 가벼운 편
격리 수준 강함 VM보다 약하지만 충분히 격리 가능
사용 느낌 컴퓨터 한 대 격리된 프로세스/작은 실행 환경
예시 UTM Ubuntu VM, EC2 인스턴스 Docker 컨테이너, Incus 컨테이너
주 용도 OS 실습, 강한 격리, 서버 단위 분리 앱 배포, 서비스 분리, 빠른 실행

12. 한 번에 이해하는 비유

VM은 원룸을 하나 새로 만드는 느낌이다.

방
화장실
주방
전기
수도
문
주소

전부 독립적이다. 그래서 안전하고 분리도 잘 되지만, 만드는 데 비용이 많이 든다.

컨테이너는 큰 건물 안에 칸막이 작업실을 만드는 느낌이다.

같은 건물의 기반 시설은 공유
각 작업실은 분리
필요한 도구만 넣고 빠르게 사용

독립된 공간처럼 보이지만, 건물의 기반 시설은 공유한다.

이 비유에서 건물의 기반 시설이 커널이고, 각 작업실이 컨테이너다.

13. UTM + Incus 경험을 포트폴리오식으로 말하면

네 경험은 이렇게 표현하면 좋다.

UTM을 활용해 macOS 위에 Ubuntu VM을 구성하고, 해당 VM 내부에 Incus를 설치하여 Linux 컨테이너 환경을 실습했습니다. 이를 통해 VM은 Guest OS와 커널을 포함한 가상 머신 단위의 격리 환경이고, Incus 컨테이너는 VM 내부의 Linux 커널을 공유하면서 사용자 공간과 프로세스를 격리하는 방식이라는 차이를 이해했습니다.

면접에서 짧게 말하면:

UTM으로 Ubuntu VM을 띄운 뒤 그 안에 Incus를 설치해 컨테이너를 실행해봤습니다. 이때 VM은 Ubuntu 커널까지 포함한 독립된 가상 머신이고, Incus 컨테이너는 그 Ubuntu VM의 커널을 공유하면서 파일시스템, 네트워크, 프로세스 공간을 격리하는 방식이라는 점을 이해했습니다.

정리

컨테이너와 VM은 둘 다 격리된 실행 환경을 제공한다. 하지만 무엇을 격리하느냐가 다르다.

VM:
컴퓨터 한 대 전체를 가상화한다.
Guest OS와 커널을 따로 가진다.

컨테이너:
호스트 OS 커널을 공유한다.
프로세스 실행 환경을 격리한다.

네가 해본 UTM + Incus 구조에서는 이렇게 볼 수 있다.

UTM:
맥북 위에 Ubuntu VM이라는 가상 컴퓨터를 만든다.

Incus:
그 Ubuntu VM의 Linux 커널을 공유하면서 컨테이너를 만든다.

그래서 포함 관계로 보면 컨테이너가 VM 안에 있을 수 있다. 하지만 본질적인 차이는 포함 관계가 아니라 커널을 따로 가지는가, 공유하는가이다.
