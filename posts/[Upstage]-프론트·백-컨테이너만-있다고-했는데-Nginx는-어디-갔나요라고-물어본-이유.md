```text
“서버에 띄워진 컨테이너가 뭐가 있었나요?”
```

나는 당시 기억을 기준으로 이렇게 답했다.

```text
“프론트엔드랑 백엔드 두 개였습니다.”
```

그런데 바로 이어서 이런 질문이 들어왔다.

```text
“그럼 Nginx는 어디 갔나요?”
```

처음에는 이 질문이 조금 당황스러웠다. 프론트랑 백엔드 컨테이너를 띄웠다고 말했는데 왜 갑자기 Nginx를 물어보는지 바로 연결되지 않았다. 그런데 배포 구조를 다시 생각해보면, 이 질문은 아주 자연스러운 질문이었다. 면접관 입장에서는 내가 말한 구조에서 **외부 요청을 누가 받고, 정적 파일을 누가 서빙하고, 백엔드 API로 요청을 누가 넘기는지**가 빠져 있었기 때문이다.

## 내가 말한 답변의 문제

내 답변은 기능 단위로는 어느 정도 맞을 수 있다.

```text
프론트엔드
백엔드
```

하지만 서버 운영 관점에서는 너무 단순했다.

실제 사용자가 서비스에 접속할 때는 이런 질문들이 따라온다.

```text
사용자는 어느 포트로 접속하는가?
80번 포트는 누가 받는가?
프론트 빌드 파일은 누가 서빙하는가?
API 요청은 어떻게 백엔드로 전달되는가?
도메인을 썼다면 도메인 요청은 어디로 들어가는가?
```

내가 “프론트랑 백 두 개요”라고만 말하면 이 흐름이 비어버린다.

```text
사용자
→ ?
→ 프론트
→ 백엔드
```

여기서 `?`에 해당하는 역할이 바로 Nginx일 수 있다.

그래서 면접관은 “Nginx는 어디 갔냐”고 물어본 것이다.

## Nginx는 왜 필요한가?

Nginx는 단순히 “웹서버”라는 말로 끝나는 도구가 아니다. 배포 구조에서 보통 다음 역할을 한다.

```text
정적 파일 서빙
reverse proxy
80/443 포트 진입점
도메인 요청 처리
API 요청 라우팅
```

프론트엔드가 React/Vite 같은 SPA라면 운영 환경에서는 보통 `npm run build`를 해서 정적 파일을 만든다.

```bash
npm run build
```

그러면 다음과 같은 빌드 결과물이 생긴다.

```text
dist/
├─ index.html
├─ assets/
└─ ...
```

이 파일들은 브라우저가 다운로드해서 실행하는 정적 파일이다.
이걸 누군가는 사용자에게 내려줘야 한다.

그 역할을 보통 Nginx가 한다.

```text
사용자
→ Nginx
→ index.html, JS, CSS 응답
```

즉, 프론트엔드 운영 배포에서 Nginx는 꽤 자주 등장한다.

## 프론트 컨테이너라는 말의 함정

면접에서 “프론트 컨테이너”라고 말할 때 조심해야 한다.

프론트 컨테이너라고 하면 두 가지 가능성이 있다.

```text
1. React 개발 서버가 컨테이너 안에서 실행되는 구조
2. Nginx가 프론트 빌드 파일을 서빙하는 구조
```

개발 서버 방식은 보통 이런 형태다.

```bash
npm run dev
```

또는:

```bash
npm start
```

이 방식은 개발 환경에서는 편하지만, 운영 배포에서는 일반적으로 권장되지 않는다.

운영에서는 보통 이렇게 한다.

```text
React/Vite 빌드
→ dist 생성
→ Nginx 이미지에 dist 복사
→ Nginx가 정적 파일 서빙
```

Dockerfile로 보면 이런 형태일 수 있다.

```dockerfile
FROM node:20 AS build
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
EXPOSE 80
```

이 경우 컨테이너 이름은 `frontend`일 수 있다.
하지만 실제 컨테이너 안에서 실행되는 프로세스는 Nginx다.

```text
frontend 컨테이너
└─ nginx
   └─ /usr/share/nginx/html 정적 파일 서빙
```

그래서 면접에서 더 정확히 말하려면 이렇게 해야 한다.

```text
컨테이너 기준으로는 frontend와 backend 두 개였고,
frontend 컨테이너는 React 빌드 결과물을 Nginx가 서빙하는 구조였습니다.
```

이렇게 말하면 “Nginx는 어디 갔냐”는 질문에 답이 된다.

## 가능한 배포 구조 1: Nginx가 프론트 컨테이너 내부에 있는 경우

첫 번째 가능성은 Nginx가 별도 컨테이너가 아니라 프론트 컨테이너 내부에 있는 구조다.

```text
EC2
├─ frontend 컨테이너
│   └─ Nginx
│      └─ React 정적 파일 서빙
│
└─ backend 컨테이너
    └─ Spring Boot / Node API
```

이 경우 서버에 떠 있는 컨테이너는 두 개라고 말할 수 있다.

```text
frontend
backend
```

하지만 Nginx가 사라진 것은 아니다.
Nginx는 frontend 컨테이너 내부에서 정적 파일을 서빙하고 있다.

이 구조를 설명할 때는 이렇게 말해야 한다.

```text
프론트 컨테이너라고 표현했지만, 내부적으로는 Nginx 기반 컨테이너였습니다. React 빌드 산출물을 Nginx가 /usr/share/nginx/html에서 정적 파일로 서빙했습니다.
```

## 가능한 배포 구조 2: Nginx가 별도 컨테이너인 경우

두 번째 가능성은 Nginx가 프론트/백엔드와 별도 컨테이너로 존재하는 구조다.

```text
EC2
├─ nginx 컨테이너
├─ frontend 컨테이너
└─ backend 컨테이너
```

이 경우 Nginx는 외부 요청의 진입점이다.

```text
사용자
→ Nginx 컨테이너
   ├─ /      → frontend
   └─ /api   → backend
```

이 구조라면 컨테이너는 프론트와 백 두 개가 아니라 세 개라고 말해야 한다.

```text
nginx
frontend
backend
```

만약 실제 구조가 이랬는데 “프론트랑 백 두 개”라고만 답했다면, Nginx 컨테이너를 누락한 것이다.

## 가능한 배포 구조 3: Nginx가 EC2에 직접 설치된 경우

세 번째 가능성은 Nginx가 Docker 컨테이너가 아니라 EC2 서버에 직접 설치된 구조다.

```text
EC2
├─ Nginx 직접 설치
├─ frontend 컨테이너
└─ backend 컨테이너
```

이 경우 “컨테이너로 떠 있던 것은 프론트와 백엔드 두 개”라는 말은 맞을 수 있다.
다만 Nginx는 컨테이너가 아니었을 뿐, 서버에 직접 설치되어 요청을 받고 있었던 것이다.

이 구조라면 이렇게 답해야 한다.

```text
컨테이너로 떠 있던 것은 프론트와 백엔드였고, Nginx는 컨테이너가 아니라 EC2 호스트에 직접 설치해서 80번 포트 진입점과 reverse proxy 역할을 했습니다.
```

이 답변이면 면접관의 질문에 정확히 대응할 수 있다.

## 내가 놓친 핵심

면접관이 확인하고 싶었던 것은 단순히 “컨테이너 개수”가 아니었다.

진짜 확인하고 싶었던 건 이거다.

```text
외부 요청이 서버에 들어왔을 때 어떤 흐름으로 서비스까지 도달하는가?
```

예를 들어 이런 흐름을 설명할 수 있어야 한다.

```text
사용자
→ EC2 Public IP 또는 도메인
→ Nginx
→ 프론트 정적 파일 응답
→ /api 요청은 백엔드로 프록시
→ 백엔드는 DB와 통신
```

또는 백엔드만 직접 노출했다면 이렇게 설명해야 한다.

```text
사용자
→ EC2_PUBLIC_IP:8080
→ 백엔드 애플리케이션
```

하지만 프론트엔드까지 있는 웹 서비스라면 보통 Nginx의 역할을 물어보는 게 자연스럽다.

## 앞으로 면접에서 어떻게 답해야 할까?

앞으로 같은 질문을 받으면 “프론트랑 백 두 개요”라고만 답하지 않는 게 좋다.

대신 구조를 이렇게 풀어야 한다.

```text
컨테이너 기준으로는 frontend와 backend가 있었고,
frontend 컨테이너는 React 빌드 결과물을 Nginx가 서빙하는 구조였습니다.
외부 요청은 Nginx가 받고, 정적 파일을 응답하거나 API 요청을 백엔드로 전달하는 역할을 했습니다.
```

만약 Nginx가 EC2에 직접 설치된 구조라면 이렇게 말한다.

```text
컨테이너로는 frontend와 backend가 있었고,
Nginx는 컨테이너가 아니라 EC2 호스트에 직접 설치되어 있었습니다.
Nginx가 80번 포트로 들어온 요청을 받고, 필요한 경우 백엔드 포트로 reverse proxy했습니다.
```

만약 Nginx가 별도 컨테이너였다면 이렇게 말한다.

```text
정확히는 nginx, frontend, backend 컨테이너가 있었습니다.
Nginx 컨테이너가 외부 요청의 진입점이었고, 프론트 정적 파일 서빙과 백엔드 API 프록시 역할을 했습니다.
```

## 면접 답변 예시

정확한 구조가 기억나지 않을 때는 이렇게 말하는 게 가장 안전하다.

```text
당시 제가 컨테이너를 기능 단위로 기억해서 프론트와 백엔드라고 답했는데, 운영 구조 관점에서는 Nginx 역할을 분리해서 설명했어야 했습니다. 프론트 컨테이너가 Nginx 기반으로 정적 파일을 서빙했거나, Nginx가 별도 컨테이너 또는 EC2 호스트에 직접 설치되어 외부 요청의 진입점 역할을 했어야 합니다. 핵심은 외부 요청을 Nginx가 받고, 프론트 정적 파일을 서빙하거나 백엔드 API로 프록시하는 구조입니다.
```

이렇게 답하면 적어도 “Nginx가 왜 필요한지 모르는 사람”처럼 보이지 않는다.

## 정리

면접관이 “Nginx는 어디 갔냐”고 물어본 이유는 내가 말한 구조에서 외부 요청 진입점이 빠져 있었기 때문이다.

```text
프론트 컨테이너
백엔드 컨테이너
```

이렇게만 말하면 다음 질문이 남는다.

```text
80번 포트는 누가 받는가?
정적 파일은 누가 내려주는가?
API 요청은 누가 백엔드로 넘기는가?
```

이 질문의 답이 보통 Nginx다.

Nginx는 세 가지 형태로 존재할 수 있다.

```text
1. 프론트 컨테이너 내부에서 정적 파일을 서빙하는 Nginx
2. 별도 Nginx 컨테이너
3. EC2 호스트에 직접 설치된 Nginx
```

따라서 “컨테이너가 프론트와 백 두 개였다”라고 말할 때도, Nginx가 어떤 형태로 존재했는지 함께 설명해야 한다.

한 줄로 정리하면 다음과 같다.

```text
프론트와 백엔드만 말하면 배포 구조가 끝나는 게 아니라, 외부 요청을 받는 Nginx의 위치까지 설명해야 한다.
```
