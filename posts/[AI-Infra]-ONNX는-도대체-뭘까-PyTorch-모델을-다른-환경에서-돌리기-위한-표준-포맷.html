[AI Infra] ONNX는 도대체 뭘까? PyTorch 모델을 다른 환경에서 돌리기 위한 표준 포맷

2026.05.07

PyTorch inference를 공부하다 보면 이런 흐름이 나온다.

PyTorch 모델 학습
↓
.pth 파일 저장
↓
PyTorch로 inference

그런데 실무에서는 이런 요구가 생긴다.

PyTorch로 학습한 모델을
C++ 서버에서 돌리고 싶다

PyTorch 없이 가볍게 추론하고 싶다

모바일, 엣지, 브라우저, 다른 런타임에서 실행하고 싶다

TensorRT나 ONNX Runtime으로 최적화하고 싶다

이때 등장하는 개념이 ONNX다.

1. ONNX 한 줄 정의

ONNX는 Open Neural Network Exchange의 약자다.

한 줄로 말하면 이렇다.

ONNX = 머신러닝 모델을 여러 프레임워크와 실행 환경에서 쓸 수 있게 만든 공통 모델 포맷

공식 ONNX 설명에서도 ONNX는 머신러닝 모델을 표현하기 위한 open format이고, 여러 프레임워크·도구·런타임·컴파일러에서 사용할 수 있도록 공통 연산자와 파일 형식을 정의한다고 설명한다. (ONNX)

쉽게 말하면 ONNX는 모델 세계의 공용 번역 포맷이다.

PyTorch 모델
↓
ONNX로 변환
↓
ONNX Runtime / TensorRT / C++ / Java / Edge 환경에서 실행

2. 왜 ONNX가 필요할까?

PyTorch로 학습한 모델은 보통 PyTorch 환경에서 잘 돈다.

model = MyModel()
model.load_state_dict(torch.load("model.pth"))
model.eval()

그런데 운영 환경에서는 꼭 PyTorch만 쓰고 싶지 않을 수 있다.

예를 들어 이런 상황이다.

서버는 C++ 기반이다
모바일에서 모델을 돌려야 한다
브라우저에서 inference를 해야 한다
PyTorch 의존성을 줄이고 싶다
추론 속도를 더 최적화하고 싶다
TensorRT로 변환하고 싶다

이때 모델을 ONNX로 바꾸면 더 다양한 실행 환경으로 옮길 수 있다.

PyTorch 공식 문서도 ONNX를 다양한 하드웨어 플랫폼과 런타임 환경에서 실행할 수 있는 유연한 표준 포맷으로 설명한다. (PyTorch Docs)

3. ONNX는 모델 자체인가?

ONNX 파일은 보통 .onnx 확장자를 가진다.

model.onnx

이 파일 안에는 모델의 계산 그래프가 들어 있다.

예를 들어 PyTorch 모델이 내부적으로 이런 연산을 한다고 하자.

입력
↓
Conv2d
↓
ReLU
↓
MaxPool
↓
Linear
↓
출력

ONNX는 이 구조를 프레임워크 독립적인 그래프 형태로 표현한다.

즉, ONNX 파일은 단순한 가중치 파일이 아니라 다음 정보를 담는다.

모델 연산 그래프
연산자 정보
입력 shape
출력 shape
가중치
데이터 타입

PyTorch의 .pth가 보통 PyTorch 모델 구조 코드와 함께 쓰이는 가중치 저장 방식이라면, .onnx는 실행 가능한 계산 그래프에 더 가깝다.

4. PyTorch .pth와 ONNX .onnx 차이

구분 .pth .onnx
주 사용처 PyTorch 내부 저장 프레임워크 간 모델 교환
내용 주로 state_dict, 가중치 계산 그래프 + 가중치
실행 환경 PyTorch 필요 ONNX Runtime 등에서 실행
모델 구조 코드 필요할 때가 많음 그래프 안에 구조 포함
목적 학습/재학습/저장 배포/추론/최적화

정리하면 이렇다.

.pth = PyTorch 세계 안에서 쓰기 좋음
.onnx = PyTorch 밖으로 모델을 내보내기 좋음

5. ONNX는 Training보다 Inference에 가깝다

ONNX는 주로 inference 배포 쪽에서 많이 등장한다.

흐름은 보통 이렇다.

1. PyTorch로 모델 학습
2. 학습된 가중치 저장
3. 모델을 ONNX로 export
4. ONNX Runtime 또는 TensorRT로 inference
5. API 서버나 엣지 장치에 배포

ONNX GitHub 설명에서도 ONNX는 AI 모델을 위한 오픈소스 포맷이고, 계산 그래프 모델·내장 연산자·표준 데이터 타입을 정의하며 현재는 주로 inferencing 기능에 초점을 둔다고 설명한다. (GitHub)

즉, ONNX는 보통 이런 질문에서 나온다.

모델을 어떻게 더 빠르게 서빙하지?
PyTorch 없이 어떻게 실행하지?
다른 언어/환경에서 어떻게 돌리지?
TensorRT로 어떻게 넘기지?

6. ONNX Runtime이란?

ONNX 파일만 있다고 끝은 아니다.

.onnx 파일을 실제로 실행해주는 엔진이 필요하다.

그 대표적인 엔진이 ONNX Runtime이다.

ONNX = 모델 포맷
ONNX Runtime = ONNX 모델을 실행하는 추론 엔진

ONNX Runtime 공식 문서는 ONNX Runtime을 다양한 하드웨어 특화 라이브러리와 통합할 수 있는 cross-platform 머신러닝 모델 accelerator라고 설명한다. 또한 PyTorch, TensorFlow/Keras, TFLite, scikit-learn 등의 모델을 사용할 수 있다고 설명한다. (ONNX Runtime)

비유하면 이렇다.

ONNX 파일 = 악보
ONNX Runtime = 악보를 실제로 연주하는 연주자

7. 전체 흐름으로 보기

PyTorch 모델을 ONNX로 바꿔서 inference하는 흐름은 대략 이렇다.

PyTorch 학습
↓
model.pth 저장
↓
PyTorch 모델 로드
↓
dummy input 준비
↓
torch.onnx.export()
↓
model.onnx 생성
↓
ONNX Runtime으로 로드
↓
inference 실행

한 문장으로 요약하면 이렇다.

PyTorch에서 만든 모델을 ONNX라는 공통 포맷으로 export한 뒤, ONNX Runtime 같은 엔진에서 실행한다.

8. PyTorch 모델을 ONNX로 Export하는 예시

간단한 예시는 이런 느낌이다.

import torch

device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

model = MyModel()
model.load_state_dict(torch.load("model.pth", map_location=device))
model.to(device)
model.eval()

dummy_input = torch.randn(1, 3, 224, 224).to(device)

torch.onnx.export(
    model,
    dummy_input,
    "model.onnx",
    input_names=["input"],
    output_names=["output"],
    opset_version=17
)

여기서 핵심은 dummy_input이다.

ONNX export는 모델이 어떤 입력 shape으로 실행되는지 알아야 한다.

dummy_input = 모델 구조를 추적하기 위한 예시 입력

예를 들어 이미지 모델이면 보통 이런 shape을 쓴다.

[1, 3, 224, 224]

의미는 다음과 같다.

1 = batch size
3 = channel
224 = height
224 = width

9. 왜 dummy input이 필요할까?

PyTorch 모델은 Python 코드로 동작한다.

즉, forward 함수 안에 여러 연산이 있다.

def forward(self, x):
    x = self.conv(x)
    x = self.relu(x)
    x = self.pool(x)
    x = self.fc(x)
    return x

ONNX로 export할 때는 이 forward 흐름을 따라가면서 계산 그래프를 만든다.

그래서 예시 입력을 한 번 넣어보면서 이런 그래프를 뽑는다.

input
↓
Conv
↓
Relu
↓
MaxPool
↓
Gemm
↓
output

즉, dummy input은 실제 추론용 데이터라기보다 모델 그래프를 추출하기 위한 샘플 입력이다.

10. ONNX Runtime으로 추론하기

ONNX 파일을 만들었다면 ONNX Runtime으로 실행할 수 있다.

import onnxruntime as ort
import numpy as np

session = ort.InferenceSession("model.onnx")

x = np.random.randn(1, 3, 224, 224).astype(np.float32)

outputs = session.run(
    ["output"],
    {"input": x}
)

print(outputs[0])

PyTorch와 차이가 있다.

PyTorch는 보통 tensor를 쓴다.

x = torch.randn(1, 3, 224, 224)

ONNX Runtime Python API에서는 보통 numpy array를 입력으로 넣는다.

x = np.random.randn(1, 3, 224, 224).astype(np.float32)

흐름은 이렇다.

numpy input
↓
ONNX Runtime session
↓
ONNX model 실행
↓
numpy output

11. ONNX를 쓰면 무조건 빨라질까?

무조건은 아니다.

ONNX는 포맷이다.

속도를 내는 것은 ONNX Runtime, TensorRT, OpenVINO 같은 실행 엔진과 최적화 과정이다.

ONNX = 모델을 표현하는 표준 포맷
ONNX Runtime = ONNX 모델을 실행하고 최적화하는 엔진
TensorRT = NVIDIA GPU 추론 최적화 엔진

그래서 정확히는 이렇게 이해해야 한다.

ONNX로 바꾼다
= 모델을 다른 최적화 엔진에서 실행할 수 있는 형태로 만든다

ONNX Runtime/TensorRT로 실행한다
= 실제 inference 성능 최적화를 기대할 수 있다

ONNX Runtime inference 설명에서도 PyTorch, Hugging Face, TensorFlow 모델을 다양한 소프트웨어·하드웨어 스택에서 추론할 수 있게 하고, 하드웨어 가속기와 여러 언어 API를 지원한다고 설명한다. (ONNX Runtime)

12. ONNX와 TensorRT 관계

AI inference 최적화에서 이런 흐름을 자주 본다.

PyTorch
↓
ONNX
↓
TensorRT Engine
↓
NVIDIA GPU에서 고속 추론

여기서 ONNX는 중간 포맷 역할을 한다.

PyTorch 모델을 바로 TensorRT로 넘기기보다
ONNX로 export한 뒤
TensorRT가 ONNX를 읽어서 최적화 엔진을 만드는 방식

TensorRT는 NVIDIA GPU에서 inference를 빠르게 하기 위한 최적화 도구다.

예를 들어 이런 최적화를 할 수 있다.

FP16 변환
INT8 양자화
Layer fusion
Kernel 최적화
메모리 최적화

즉, 관계는 이렇게 보면 된다.

PyTorch = 모델 개발/학습
ONNX = 모델 교환 포맷
TensorRT = NVIDIA GPU 추론 최적화

13. ONNX와 CUDA 관계

CUDA는 NVIDIA GPU에서 병렬 연산을 하기 위한 플랫폼이다.

ONNX는 모델 포맷이다.

둘은 역할이 다르다.

CUDA = GPU 계산 플랫폼
ONNX = 모델 표현 포맷
ONNX Runtime GPU Provider = CUDA를 사용해서 ONNX 모델을 GPU에서 실행

즉, ONNX 자체가 GPU를 의미하는 것은 아니다.

ONNX 모델은 CPU에서도 돌 수 있고, GPU에서도 돌 수 있다.

ONNX + CPU Runtime
ONNX + CUDA Execution Provider
ONNX + TensorRT Execution Provider

이런 식으로 어떤 실행 provider를 쓰느냐에 따라 실행 환경이 달라진다.

14. ONNX에서 자주 나오는 Opset

ONNX export를 하다 보면 opset_version이라는 인자가 나온다.

torch.onnx.export(
    model,
    dummy_input,
    "model.onnx",
    opset_version=17
)

Opset은 ONNX 연산자 집합의 버전이다.

쉽게 말하면 이런 뜻이다.

opset = ONNX가 지원하는 연산자 규칙 버전

모델 안에는 여러 연산이 있다.

Conv
Relu
MatMul
Add
Softmax
Reshape

이 연산자들이 ONNX에서 어떤 방식으로 표현되는지 버전이 있다.

그래서 export할 때 opset version을 맞춰줘야 한다.

너무 낮은 opset을 쓰면 최신 연산이 지원되지 않을 수 있고, 너무 높은 opset을 쓰면 실행 환경이 해당 버전을 지원하지 않을 수 있다.

15. Dynamic Axis는 왜 필요할까?

기본 export는 입력 shape이 고정될 수 있다.

예를 들어 dummy input을 이렇게 줬다면

[1, 3, 224, 224]

batch size가 1로 고정될 수 있다.

그런데 실제 inference에서는 batch size가 바뀔 수 있다.

[1, 3, 224, 224]
[8, 3, 224, 224]
[16, 3, 224, 224]

이럴 때 dynamic_axes를 설정한다.

torch.onnx.export(
    model,
    dummy_input,
    "model.onnx",
    input_names=["input"],
    output_names=["output"],
    dynamic_axes={
        "input": {0: "batch_size"},
        "output": {0: "batch_size"}
    },
    opset_version=17
)

의미는 이렇다.

input의 0번 차원은 batch size라서 바뀔 수 있다
output의 0번 차원도 batch size라서 바뀔 수 있다

즉, dynamic axes는 실제 서비스에서 batch size가 유동적일 때 중요하다.

16. ONNX에서 자주 겪는 문제

ONNX export는 항상 한 번에 깔끔하게 되는 것은 아니다.

자주 만나는 문제는 이런 것들이다.

지원하지 않는 PyTorch 연산이 있음
모델 forward에 Python 제어문이 복잡함
입력 shape이 고정됨
dynamic axes 설정 누락
opset version 문제
PyTorch 결과와 ONNX 결과가 미묘하게 다름
후처리 로직이 ONNX 밖에 있음

특히 PyTorch 모델 안에 Python 로직이 많으면 ONNX로 변환하기 어렵다.

예를 들어 이런 경우다.

if x.sum() > 0:
    ...
else:
    ...

ONNX는 모델을 계산 그래프로 표현해야 하므로, Python 런타임에 의존하는 동적 제어 흐름은 변환이 까다로울 수 있다.

17. Export 후에는 반드시 검증해야 한다

ONNX로 변환했다고 바로 배포하면 안 된다.

PyTorch 결과와 ONNX Runtime 결과가 거의 같은지 확인해야 한다.

import numpy as np
import torch
import onnxruntime as ort

model.eval()

x = torch.randn(1, 3, 224, 224)

with torch.inference_mode():
    torch_output = model(x).detach().numpy()

session = ort.InferenceSession("model.onnx")
onnx_output = session.run(
    ["output"],
    {"input": x.numpy()}
)[0]

np.testing.assert_allclose(
    torch_output,
    onnx_output,
    rtol=1e-03,
    atol=1e-05
)

print("PyTorch output and ONNX output are close")

검증 포인트는 이렇다.

입력 shape 동일
전처리 동일
출력 shape 동일
수치 오차 허용 범위 내
클래스 예측 결과 동일

18. ONNX가 AI DevOps에서 중요한 이유

AI DevOps나 MLOps 관점에서 ONNX는 “모델 배포 포맷”으로 중요하다.

학습 환경과 운영 환경은 다를 수 있다.

학습 환경:
Python
PyTorch
GPU 서버
연구 코드

운영 환경:
FastAPI
C++ 서버
Java 서비스
ONNX Runtime
TensorRT
Kubernetes
모니터링

ONNX를 쓰면 학습 프레임워크와 서빙 환경을 분리하기 쉬워진다.

모델 개발자는 PyTorch로 학습
인프라/서빙 쪽은 ONNX Runtime 또는 TensorRT로 배포

이렇게 되면 운영 환경에서 PyTorch 의존성을 줄이고, 추론 엔진 중심으로 관리할 수 있다.

19. 전체 그림 정리

지금까지 본 개념을 연결하면 이렇게 된다.

Training
= PyTorch로 모델 학습
= loss.backward()
= optimizer.step()
= 가중치 업데이트

Inference
= 학습된 모델로 예측
= model.eval()
= torch.inference_mode()
= forward only

ONNX
= 학습된 모델을 다른 실행 환경으로 옮기기 위한 공통 포맷

ONNX Runtime
= ONNX 모델을 실제로 실행하는 추론 엔진

TensorRT
= NVIDIA GPU에서 inference를 최적화하는 엔진

AI inference 배포 흐름은 이렇게 볼 수 있다.

PyTorch model
↓ export
ONNX model
↓ optimize/run
ONNX Runtime or TensorRT
↓ serve
FastAPI / C++ / Java / Edge / Kubernetes

20. 최종 정리

ONNX는 모델을 학습시키는 도구라기보다는, 학습된 모델을 여러 환경에서 실행할 수 있게 해주는 공통 모델 포맷이다.

ONNX = 모델 교환 포맷
ONNX Runtime = ONNX 모델 실행 엔진
TensorRT = NVIDIA GPU 추론 최적화 엔진

PyTorch만 쓸 때는 .pth로 충분할 수 있다.

하지만 모델을 실제 서비스에 올리고, 다른 언어·다른 런타임·GPU 최적화 엔진에서 돌리고 싶다면 ONNX가 중요해진다.

한 문장으로 정리하면 이렇다.

ONNX는 PyTorch에서 만든 모델을 운영 환경으로 옮기기 위한 중간 표준 포맷이다.

그래서 AI DevOps 관점에서는 ONNX를 단순 파일 형식이 아니라 학습 코드와 추론 서빙 환경을 분리해주는 배포 인터페이스로 이해하면 좋다.
