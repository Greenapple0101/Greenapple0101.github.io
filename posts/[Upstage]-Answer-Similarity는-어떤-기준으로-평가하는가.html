[Upstage] Answer Similarity는 어떤 기준으로 평가하는가

2026.05.30

RAG나 LLM 서비스를 평가할 때 Answer Similarity라는 지표가 자주 등장한다. 이름 그대로 해석하면 “생성된 답변이 기준 답변과 얼마나 비슷한가”를 보는 지표이다.

하지만 여기서 말하는 비슷함은 단순히 글자가 똑같은지를 보는 것이 아니다. RAG 평가에서 중요한 것은 표현이 완전히 같은가가 아니라, 의미가 같은가, 핵심 정보가 일치하는가, 질문에 대해 같은 답을 하고 있는가이다.

예를 들어 기준 답변이 다음과 같다고 하자.

환불은 결제 취소일로부터 영업일 기준 3~5일 이내 처리됩니다.

모델 답변이 다음과 같다면 표현은 다르지만 의미는 거의 같다.

결제를 취소하면 보통 영업일 기준 3일에서 5일 안에 환불됩니다.

이 경우 문자열은 다르지만 Answer Similarity는 높게 나와야 한다.

반대로 모델 답변이 다음과 같다면 일부 단어는 비슷하지만 의미는 다르다.

환불은 결제일로부터 7일 이내에만 신청할 수 있습니다.

이 답변에는 환불, 결제, 7일 같은 관련 단어가 들어 있지만, 기준 답변과 핵심 내용이 다르다. 따라서 Answer Similarity는 낮게 평가되어야 한다.

Answer Similarity가 필요한 이유

LLM 답변은 정답이 하나의 문장으로 고정되지 않는다. 같은 의미를 여러 방식으로 표현할 수 있다.

예를 들어 다음 세 문장은 모두 비슷한 의미를 가진다.

비밀번호는 마이페이지에서 변경할 수 있습니다.
마이페이지에 들어가면 비밀번호 변경 메뉴를 찾을 수 있습니다.
계정 설정의 마이페이지 항목에서 비밀번호를 바꾸면 됩니다.

전통적인 문자열 비교 방식은 이런 답변들을 서로 다르게 볼 수 있다. 하지만 실제 서비스 관점에서는 모두 비슷한 답변이다.

따라서 RAG 평가에서는 단순 문자열 일치보다 의미 기반 평가가 필요하다. Answer Similarity는 이때 사용된다.

Answer Similarity는 무엇과 무엇을 비교하는가

Answer Similarity는 보통 다음 두 가지를 비교한다.

reference answer: 사람이 작성한 기준 답변
generated answer: RAG 또는 LLM이 생성한 답변

예를 들어 평가 데이터셋이 다음과 같이 구성되어 있다고 하자.

question: 연차는 며칠 전에 신청해야 하나요?

reference answer:
연차는 최소 3일 전에 사내 인사 시스템에서 신청해야 합니다.

generated answer:
휴가는 사내 시스템에서 적어도 3일 전까지 신청해야 합니다.

Answer Similarity는 reference answer와 generated answer의 의미가 얼마나 가까운지를 평가한다.

평가 기준 1. 핵심 의미가 같은가

가장 중요한 기준은 핵심 의미가 같은지이다.

기준 답변과 생성 답변이 같은 결론을 말하고 있다면 similarity는 높아야 한다.

기준 답변:
연차는 최소 3일 전에 신청해야 합니다.

생성 답변:
휴가는 적어도 3일 전까지 신청해야 합니다.

이 경우 연차, 휴가라는 표현은 조금 다르지만 핵심 의미는 같다.

핵심 의미:
휴가 신청은 최소 3일 전에 해야 함

따라서 Answer Similarity는 높게 평가할 수 있다.

반대로 다음 답변은 비슷해 보이지만 핵심 의미가 다르다.

기준 답변:
연차는 최소 3일 전에 신청해야 합니다.

생성 답변:
연차는 당일에도 신청할 수 있습니다.

둘 다 연차 신청을 말하지만 결론이 반대이다. 이 경우 similarity는 낮게 평가해야 한다.

평가 기준 2. 핵심 정보가 빠지지 않았는가

Answer Similarity는 단순히 비슷한 주제를 말하는지만 보지 않는다. 기준 답변에 있는 중요한 정보가 생성 답변에 포함되어 있는지도 봐야 한다.

예를 들어 기준 답변이 다음과 같다고 하자.

연차는 최소 3일 전에 사내 인사 시스템에서 신청해야 합니다.

여기에는 중요한 정보가 두 개 있다.

1. 최소 3일 전에 신청
2. 사내 인사 시스템에서 신청

생성 답변이 다음과 같다면 일부 정보가 빠졌다.

연차는 미리 신청해야 합니다.

이 답변은 큰 방향은 맞지만 3일 전과 사내 인사 시스템이라는 핵심 정보가 빠져 있다. 따라서 similarity는 중간 또는 낮은 점수로 평가하는 것이 적절하다.

생성 답변이 다음과 같다면 더 정확하다.

연차는 최소 3일 전에 사내 인사 시스템에서 신청해야 합니다.

기준 답변의 핵심 정보가 모두 포함되어 있으므로 similarity는 높다.

평가 기준 3. 숫자, 날짜, 조건이 일치하는가

Answer Similarity에서 특히 조심해야 하는 것이 숫자, 날짜, 조건이다.

문장 전체 분위기는 비슷해도 숫자가 틀리면 실제로는 오답일 수 있다.

기준 답변:
환불은 영업일 기준 3~5일 이내 처리됩니다.

생성 답변:
환불은 영업일 기준 5~7일 이내 처리됩니다.

두 답변은 매우 비슷해 보인다. 둘 다 환불 처리 기간을 말하고 있고, 문장 구조도 거의 같다. 하지만 핵심 숫자가 다르다.

이 경우 단순 임베딩 기반 similarity는 높게 나올 수 있다. 하지만 실제 평가에서는 낮게 봐야 한다.

OCR, 금융, 계약서, 정책 문서, 고객 지원 RAG에서는 숫자와 날짜가 매우 중요하다.

중요하게 봐야 하는 요소는 다음과 같다.

금액
날짜
기간
비율
수량
버전
계약 조건
제외 조건
자격 요건
마감 기한

Answer Similarity를 사용할 때는 이런 요소가 틀렸는데도 높은 점수를 주지 않도록 별도 검증이 필요하다.

평가 기준 4. 질문에 대한 답이 같은가

Answer Similarity는 기준 답변과 생성 답변만 비교하면 부족할 수 있다. 질문까지 함께 봐야 한다.

예를 들어 질문이 다음과 같다고 하자.

질문:
환불은 며칠 안에 처리되나요?

기준 답변은 다음과 같다.

환불은 영업일 기준 3~5일 이내 처리됩니다.

생성 답변이 다음과 같다면 어떨까.

환불 신청은 결제일로부터 7일 이내에 가능합니다.

이 답변은 환불과 관련된 말이지만 질문에 대한 답은 아니다. 질문은 “처리 기간”을 물었는데, 생성 답변은 “신청 가능 기간”을 말하고 있다.

따라서 Answer Similarity를 제대로 평가하려면 다음을 함께 봐야 한다.

질문이 무엇을 묻고 있는가
기준 답변은 어떤 질문에 대한 답인가
생성 답변이 같은 질문에 답하고 있는가

즉, 같은 주제를 말하는 것과 같은 질문에 답하는 것은 다르다.

평가 기준 5. 표현은 달라도 의미가 같은가

LLM은 같은 내용을 다양한 문장으로 표현한다. 그래서 Answer Similarity는 표현 차이를 어느 정도 허용해야 한다.

다음 답변들은 의미상 비슷하다.

기준 답변:
비밀번호는 마이페이지에서 변경할 수 있습니다.

생성 답변 1:
마이페이지에 들어가면 비밀번호를 바꿀 수 있습니다.

생성 답변 2:
계정의 마이페이지 메뉴에서 비밀번호 변경이 가능합니다.

생성 답변 3:
비밀번호 변경은 마이페이지에서 진행하면 됩니다.

문장 구조나 단어는 다르지만 핵심 의미는 같다.

이런 경우에는 문자열 비교보다 임베딩 기반 similarity나 LLM-as-a-Judge 방식이 더 적합하다.

평가 기준 6. 반대 의미를 말하지 않는가

Answer Similarity에서 특히 위험한 경우가 반대 의미이다.

예를 들어 다음 두 답변은 단어는 많이 겹치지만 의미는 정반대이다.

기준 답변:
관리자 승인 없이 외부 공유는 불가능합니다.

생성 답변:
관리자 승인 없이도 외부 공유가 가능합니다.

두 문장은 거의 같은 단어를 사용하지만 불가능과 가능 때문에 의미가 완전히 달라진다.

임베딩 기반 similarity는 이런 경우에도 높게 나올 수 있다. 왜냐하면 두 문장이 같은 주제와 비슷한 단어를 공유하기 때문이다.

따라서 Answer Similarity를 평가할 때는 부정 표현을 조심해야 한다.

가능 / 불가능
허용 / 금지
포함 / 제외
필수 / 선택
이전 / 이후
이상 / 이하
초과 / 미만

이런 표현이 바뀌면 similarity가 높아도 실제로는 오답이다.

평가 기준 7. 불필요하거나 잘못된 정보가 추가되지 않았는가

생성 답변이 기준 답변의 내용을 포함하더라도, 문서에 없는 잘못된 정보를 추가하면 문제가 된다.

예를 들어 기준 답변이 다음과 같다고 하자.

환불은 영업일 기준 3~5일 이내 처리됩니다.

생성 답변이 다음과 같다면 일부는 맞지만 불필요한 정보가 추가되었다.

환불은 영업일 기준 3~5일 이내 처리되며, VIP 고객은 당일 환불됩니다.

만약 기준 문서에 VIP 고객 당일 환불 내용이 없다면 이 답변은 위험하다.

Answer Similarity는 기준 답변과 비슷한 정도를 보지만, 잘못 추가된 내용까지 완벽히 잡지 못할 수 있다. 그래서 RAG 평가에서는 Answer Similarity와 함께 Faithfulness 또는 Groundedness도 같이 본다.

Answer Similarity:
기준 답변과 의미가 비슷한가

Faithfulness / Groundedness:
생성 답변이 제공된 문서에 근거하는가

즉, 기준 답변과 비슷해도 문서에 없는 내용을 추가하면 좋은 답변이 아니다.

Answer Similarity를 계산하는 방법

Answer Similarity를 계산하는 방법은 크게 세 가지로 나눌 수 있다.

1. 문자열 기반 비교
2. 임베딩 기반 비교
3. LLM-as-a-Judge 기반 비교

1. 문자열 기반 비교

문자열 기반 비교는 기준 답변과 생성 답변의 글자 또는 단어가 얼마나 겹치는지 보는 방식이다.

대표적으로 다음 방식이 있다.

Exact Match
BLEU
ROUGE
Edit Distance

이 방식은 계산이 빠르고 단순하다. 하지만 표현이 조금만 달라져도 낮은 점수가 나올 수 있다.

예를 들어 다음 두 문장은 의미가 같지만 문자열은 다르다.

기준 답변:
환불은 영업일 기준 3~5일 이내 처리됩니다.

생성 답변:
결제 취소 후 보통 3일에서 5일 안에 환불됩니다.

문자열 기반 지표는 이런 답변을 충분히 좋게 평가하지 못할 수 있다.

따라서 RAG 평가에서는 문자열 기반 지표만으로는 부족하다.

2. 임베딩 기반 비교

임베딩 기반 비교는 기준 답변과 생성 답변을 각각 벡터로 변환한 뒤, 두 벡터의 유사도를 계산하는 방식이다.

흐름은 다음과 같다.

reference answer
↓
embedding model
↓
reference vector

generated answer
↓
embedding model
↓
generated vector

reference vector와 generated vector의 cosine similarity 계산

예를 들어 두 답변의 벡터가 의미적으로 가까우면 similarity가 높게 나온다.

기준 답변:
비밀번호는 마이페이지에서 변경할 수 있습니다.

생성 답변:
마이페이지에서 비밀번호 변경이 가능합니다.

이 방식은 표현이 달라도 의미가 비슷하면 높은 점수를 줄 수 있다.

하지만 단점도 있다. 숫자, 부정 표현, 조건 차이를 민감하게 잡지 못할 수 있다.

기준 답변:
환불은 3~5일 이내 처리됩니다.

생성 답변:
환불은 5~7일 이내 처리됩니다.

두 문장은 의미 공간에서는 매우 가까울 수 있다. 하지만 실제 서비스에서는 틀린 답변이다.

따라서 임베딩 기반 Answer Similarity는 편리하지만, 중요한 필드나 조건 검증과 함께 사용해야 한다.

3. LLM-as-a-Judge 기반 비교

LLM-as-a-Judge는 LLM에게 기준 답변과 생성 답변을 비교하게 하는 방식이다.

예를 들어 LLM에게 다음과 같이 평가를 맡긴다.

질문:
환불은 며칠 안에 처리되나요?

기준 답변:
환불은 영업일 기준 3~5일 이내 처리됩니다.

생성 답변:
결제 취소 후 보통 영업일 기준 3일에서 5일 안에 환불됩니다.

평가 기준:
생성 답변이 기준 답변과 의미적으로 같은가?
핵심 숫자, 조건, 제한사항이 일치하는가?
불필요하거나 잘못된 정보가 추가되었는가?
0~1 사이 점수로 평가하라.

LLM-as-a-Judge는 단순 임베딩보다 문맥과 조건을 더 잘 볼 수 있다. 특히 숫자, 조건, 부정 표현을 기준에 넣어 평가할 수 있다.

다만 LLM-as-a-Judge도 완벽하지 않다. 평가 모델 자체가 실수할 수 있고, 비용과 latency가 발생한다. 그래서 중요한 평가셋에는 사람 검수와 함께 사용하는 것이 좋다.

Answer Similarity와 Answer Correctness의 차이

Answer Similarity와 Answer Correctness는 비슷해 보이지만 다르다.

Answer Similarity:
기준 답변과 생성 답변이 의미적으로 얼마나 비슷한가

Answer Correctness:
생성 답변이 질문에 대해 실제로 맞는 답인가

예를 들어 기준 답변 자체가 잘못되어 있다면, 모델 답변이 기준 답변과 비슷해도 correctness는 낮을 수 있다.

또는 기준 답변과 표현은 다르지만 실제로 맞는 답일 수도 있다.

RAG 평가에서는 보통 다음을 함께 본다.

Answer Similarity
Answer Correctness
Faithfulness
Context Relevance
Context Recall

Answer Similarity는 “기준 답변과 비슷한가”를 보는 지표이고, Answer Correctness는 “정답인가”를 보는 지표에 가깝다.

Answer Similarity와 Faithfulness의 차이

Answer Similarity는 기준 답변과 생성 답변을 비교한다.

Faithfulness는 생성 답변이 검색된 문서에 근거하고 있는지를 본다.

예를 들어 다음과 같은 상황이 있다.

기준 답변:
환불은 영업일 기준 3~5일 이내 처리됩니다.

검색된 문서:
환불은 영업일 기준 7일 이내 처리됩니다.

생성 답변:
환불은 영업일 기준 3~5일 이내 처리됩니다.

이 경우 생성 답변은 기준 답변과 비슷하므로 Answer Similarity는 높다. 하지만 검색된 문서에는 7일이라고 되어 있으므로 Faithfulness는 낮을 수 있다.

즉, Answer Similarity가 높다고 항상 RAG 답변이 좋은 것은 아니다. RAG에서는 답변이 기준 답변과 비슷한지도 중요하지만, 검색된 문서에 근거하고 있는지도 중요하다.

Answer Similarity 평가 시 주의할 점

Answer Similarity를 사용할 때는 다음을 주의해야 한다.

숫자와 날짜가 틀려도 similarity가 높게 나올 수 있음
부정 표현이 바뀌어도 비슷하게 평가될 수 있음
중요 조건이 빠져도 문장 전체가 비슷하면 높은 점수가 나올 수 있음
기준 답변이 불완전하면 평가도 흔들림
생성 답변이 불필요한 정보를 추가해도 similarity만으로는 잘 안 잡힐 수 있음

따라서 Answer Similarity는 단독으로 쓰기보다 다른 지표와 함께 사용해야 한다.

Answer Similarity + Answer Correctness
Answer Similarity + Faithfulness
Answer Similarity + Context Recall
Answer Similarity + Critical Field Match

특히 OCR, 계약서, 금융 문서, 정책 문서처럼 숫자와 조건이 중요한 도메인에서는 Critical Field Match 같은 별도 검증이 필요하다.

실무에서 사용할 수 있는 평가 기준 예시

Answer Similarity를 0~1 점수로 평가한다고 하면 다음처럼 기준을 둘 수 있다.

1.0
기준 답변과 의미가 거의 동일하고 핵심 정보가 모두 포함됨

0.8
표현은 다르지만 핵심 의미와 주요 조건이 대부분 일치함

0.6
큰 방향은 맞지만 일부 중요한 정보가 빠짐

0.4
주제는 비슷하지만 질문에 대한 정확한 답이 부족함

0.2
관련 단어는 있으나 핵심 답변이 다름

0.0
기준 답변과 무관하거나 반대 의미임

예를 들어 다음처럼 볼 수 있다.

질문:
연차는 며칠 전에 신청해야 하나요?

기준 답변:
연차는 최소 3일 전에 사내 인사 시스템에서 신청해야 합니다.

생성 답변 A:
연차는 최소 3일 전에 사내 인사 시스템에서 신청해야 합니다.
점수: 1.0

생성 답변 B:
휴가는 적어도 3일 전까지 사내 시스템에서 신청해야 합니다.
점수: 0.9

생성 답변 C:
연차는 미리 신청해야 합니다.
점수: 0.5~0.6

생성 답변 D:
연차는 당일에도 신청할 수 있습니다.
점수: 0.0~0.2

RAG 품질 회귀에서 Answer Similarity를 어떻게 쓰는가

RAG 품질 회귀 탐지에서는 기존 버전과 새 버전의 Answer Similarity를 비교한다.

예를 들어 기준 평가셋 100개에 대해 기존 버전과 새 버전을 각각 실행한다.

기존 버전 평균 Answer Similarity: 0.86
새 버전 평균 Answer Similarity: 0.78

이 경우 새 버전의 답변이 기준 답변과 덜 비슷해졌으므로 품질 회귀 가능성이 있다.

하지만 평균만 보면 안 된다. 질문 유형별, 문서 유형별, 고객사별로 나누어 봐야 한다.

전체 평균: 0.86 → 0.84
정책 질문: 0.88 → 0.87
금액 질문: 0.91 → 0.73
절차 질문: 0.82 → 0.81

전체 평균은 조금만 떨어졌지만 금액 질문에서 크게 떨어졌다면 중요한 품질 회귀이다.

따라서 Answer Similarity는 전체 평균뿐 아니라 카테고리별로 분석해야 한다.

Quality Gate 예시

Answer Similarity를 배포 게이트에 사용할 때는 다음처럼 기준을 둘 수 있다.

전체 평균 Answer Similarity가 기존 대비 0.03 이상 하락하면 배포 차단

중요 카테고리의 Answer Similarity가 0.80 미만이면 배포 차단

숫자·날짜·금액 질문에서 Answer Similarity가 하락하고 Critical Field Match도 실패하면 배포 차단

Answer Similarity는 높지만 Faithfulness가 낮으면 배포 차단

Answer Similarity는 유지되지만 p95 latency가 기준 초과하면 배포 차단

중요한 점은 Answer Similarity 하나만으로 배포 여부를 결정하지 않는 것이다.

실제 운영에서는 similarity, correctness, faithfulness, retrieval metric, latency, cost를 함께 봐야 한다.

정리

Answer Similarity는 생성 답변이 기준 답변과 의미적으로 얼마나 비슷한지를 평가하는 지표이다.

핵심 기준은 다음과 같다.

핵심 의미가 같은가
중요 정보가 빠지지 않았는가
숫자, 날짜, 조건이 일치하는가
질문에 대한 답이 같은가
표현은 달라도 의미가 같은가
반대 의미를 말하지 않는가
잘못된 정보를 추가하지 않았는가

Answer Similarity는 문자열 일치보다 유연하고, LLM 답변 평가에 적합하다. 하지만 단독으로 사용하면 위험하다. 숫자, 날짜, 부정 표현, 조건 차이를 놓칠 수 있기 때문이다.

따라서 RAG나 OCR 기반 문서 AI 평가에서는 Answer Similarity를 다음 지표들과 함께 보는 것이 좋다.

Answer Correctness
Faithfulness
Context Recall
Context Precision
Critical Field Match
p95 Latency
Cost

결국 Answer Similarity는 “답변이 기준 답변과 얼마나 의미적으로 가까운가”를 보는 지표이다. 하지만 운영형 AI 시스템에서는 비슷함만으로 충분하지 않다. 비슷하면서도 정확하고, 문서에 근거하며, 중요한 숫자와 조건까지 맞아야 좋은 답변이라고 볼 수 있다.
