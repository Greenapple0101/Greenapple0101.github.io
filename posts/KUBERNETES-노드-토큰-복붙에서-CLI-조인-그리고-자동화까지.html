[KUBERNETES] 노드 토큰 복붙에서 CLI 조인, 그리고 자동화까지

2026.05.17 Velog 원문

쿠버네티스 클러스터를 구성할 때 가장 헷갈렸던 부분 중 하나가 워커 노드를 마스터 노드에 어떻게 조인시키는가였다.

처음에는 단순히 “워커 노드를 붙인다” 정도로만 생각했는데, 실제로 해보니 그 안에는 꽤 중요한 흐름이 있었다.

내가 기억하는 방식은 이랬다.

마스터 노드에서 노드 토큰을 확인함
↓
그 토큰을 복사함
↓
워커 노드의 YAML 설정 파일에 붙여넣음
↓
워커 노드가 마스터 노드에 조인됨

그때는 이게 그냥 “토큰 복붙해서 연결하는 작업”처럼 느껴졌다.

그런데 나중에 다시 보니, 워커 노드를 조인하는 방식은 꼭 수동 복붙만 있는 게 아니었다. CLI 명령어로 조인할 수도 있고, 더 나아가 자동화할 수도 있었다.

1. 노드 조인이란?

노드 조인은 쉽게 말하면 워커 노드를 기존 쿠버네티스 클러스터에 합류시키는 작업이다.

쿠버네티스 클러스터는 보통 크게 두 종류의 노드로 구성된다.

마스터 노드
= 클러스터를 관리하는 노드

워커 노드
= 실제 애플리케이션 Pod가 실행되는 노드

워커 노드는 혼자 아무렇게나 클러스터에 들어갈 수 없다.

마스터 노드가 관리하는 클러스터에 들어가려면, “이 워커 노드는 이 클러스터에 들어와도 된다”는 인증 정보가 필요하다.

그 역할을 하는 것이 노드 토큰이다.

2. 노드 토큰은 왜 필요할까?

워커 노드가 아무 인증 없이 클러스터에 들어올 수 있다면 위험하다.

아무 서버나 마스터 노드에 접근해서 “나도 워커 노드로 등록해줘”라고 요청할 수 있기 때문이다.

그래서 조인 과정에서는 토큰이 필요하다.

토큰은 일종의 임시 출입증처럼 볼 수 있다.

워커 노드
↓
마스터 노드에 조인 요청
↓
노드 토큰 제출
↓
클러스터가 토큰 확인
↓
정상 토큰이면 워커 노드로 등록

즉, 노드 토큰은 단순한 문자열이 아니라 클러스터에 합류하기 위한 인증 정보다.

3. 내가 처음 했던 방식: 토큰을 YAML 파일에 직접 넣기

내가 기억하는 첫 번째 방식은 노드 토큰을 워커 노드의 YAML 설정 파일에 직접 넣는 방식이었다.

대략 이런 구조였다.

server: https://<MASTER_NODE_IP>:6443
token: <NODE_TOKEN>

여기서 server는 워커 노드가 접속해야 하는 마스터 노드의 주소다.

server: https://<MASTER_NODE_IP>:6443

그리고 token은 마스터 노드에서 가져온 노드 조인용 토큰이다.

token: <NODE_TOKEN>

이 설정을 워커 노드 쪽에 넣어두면, 워커 노드는 해당 마스터 노드를 바라보고 토큰을 이용해 클러스터에 조인을 시도한다.

당시에는 이걸 보면서 이렇게 이해했다.

마스터 노드에서 토큰을 가져온다
↓
워커 노드 설정 파일에 붙여넣는다
↓
워커 노드가 마스터 노드에 붙는다

그래서 내 기억 속에는 “노드 토큰을 워커 노드의 야믈 파일에 붙여넣었다”는 장면이 남아 있었던 것 같다.

4. 이 방식의 장점

토큰을 YAML 파일에 직접 넣는 방식은 처음 배울 때는 이해하기 쉽다.

왜냐하면 워커 노드가 어떤 정보를 가지고 마스터 노드에 붙는지 눈에 보이기 때문이다.

server: https://<MASTER_NODE_IP>:6443
token: <NODE_TOKEN>

이렇게 보면 워커 노드가 조인할 때 필요한 정보가 명확하다.

어디로 접속할 것인가?
→ server

무슨 권한으로 접속할 것인가?
→ token

그래서 학습 단계에서는 꽤 직관적이다.

5. 하지만 수동 복붙 방식에는 문제가 있다

문제는 운영 관점이다.

토큰을 사람이 직접 복사해서 붙여넣는 방식은 실수할 여지가 많다.

예를 들어 토큰이 길면 일부 문자를 빼먹을 수 있다.

토큰 일부 누락
공백 포함
줄바꿈 잘못 입력
잘못된 워커 노드에 붙여넣기
오래된 토큰 사용
마스터 주소 오타

이런 실수가 발생하면 워커 노드 조인이 실패한다.

그리고 워커 노드가 한 대일 때는 괜찮지만, 여러 대가 되면 문제가 더 커진다.

worker-1 설정
worker-2 설정
worker-3 설정
worker-4 설정

이걸 사람이 일일이 들어가서 수정하면 반복 작업이 많아지고, 그만큼 실수 가능성도 커진다.

운영에서 반복 수작업은 항상 위험하다.

6. 두 번째 방식: CLI로 조인하기

나중에 보니 워커 노드를 조인하는 방식은 YAML 파일에 직접 토큰을 넣는 것만 있는 게 아니었다.

CLI 명령어로도 조인할 수 있다.

일반적으로 이런 형태다.

kubeadm join <MASTER_NODE_IP>:6443 \
  --token <TOKEN> \
  --discovery-token-ca-cert-hash sha256:<HASH>

이 명령어는 워커 노드에서 실행한다.

흐름은 이렇다.

마스터 노드에서 조인 명령어 확인
↓
워커 노드에 접속
↓
CLI 명령어 실행
↓
워커 노드가 클러스터에 조인됨

여기서도 핵심은 토큰이다.

--token <TOKEN>

워커 노드는 이 토큰을 가지고 마스터 노드에 조인 요청을 보낸다.

즉, YAML 방식이든 CLI 방식이든 공통점은 같다.

워커 노드는 노드 토큰 없이 클러스터에 조인할 수 없다.

7. YAML 방식과 CLI 방식의 차이

두 방식을 비교하면 이렇게 정리할 수 있다.

방식 설명 특징
YAML 설정 방식 워커 노드 설정 파일에 server, token 값을 직접 입력 설정이 파일에 남아 있어 구조를 보기 쉬움
CLI 조인 방식 워커 노드에서 조인 명령어를 직접 실행 빠르게 조인 가능하고 명령어 중심으로 처리 가능
자동화 방식 YAML 생성이나 CLI 실행을 스크립트/도구로 처리 수동 복붙 실수를 줄일 수 있음

결국 둘 다 “워커 노드가 마스터 노드를 알고, 토큰을 이용해 조인한다”는 점은 같다.

차이는 사람이 그 정보를 어떻게 넣느냐에 있다.

YAML 방식
→ 설정 파일에 직접 넣음

CLI 방식
→ 명령어 인자로 넣음

자동화 방식
→ 사람이 직접 안 넣고 도구가 처리함

8. 여기서 자동화가 왜 필요할까?

처음에는 직접 복붙해도 된다.

학습할 때는 오히려 직접 해보는 게 좋다.

하지만 실무나 운영 환경에서는 다르다.

운영에서는 같은 작업을 여러 번 반복해야 한다.

새 워커 노드 추가
장애 난 노드 교체
테스트 클러스터 재구성
스테이징 환경 재생성
프로덕션 환경 확장

이때마다 사람이 토큰을 복사해서 붙여넣는다면 실수 가능성이 계속 남는다.

그래서 자동화가 필요하다.

자동화의 목적은 단순히 편하려고 하는 것이 아니다.

더 중요한 목적은 재현성과 안정성이다.

누가 해도 같은 방식으로 실행되게 만들기
실수 가능성 줄이기
작업 과정을 코드로 남기기
장애 발생 시 다시 재현 가능하게 만들기

9. 자동화하면 어떻게 바뀔까?

수동 방식은 이렇다.

마스터 노드 접속
↓
토큰 확인
↓
토큰 복사
↓
워커 노드 접속
↓
YAML 파일 수정 또는 CLI 명령어 실행
↓
조인 확인

자동화 방식은 이렇게 바뀐다.

마스터 노드에서 조인 정보 생성
↓
자동화 도구가 워커 노드에 전달
↓
워커 노드 설정 생성 또는 조인 명령어 실행
↓
조인 결과 확인

사람이 직접 긴 토큰을 복붙하지 않도록 만드는 것이다.

10. Shell Script로 자동화하기

가장 단순한 자동화는 Shell Script다.

예를 들어 워커 노드에서 사용할 설정 파일을 스크립트로 생성할 수 있다.

cat <<EOF | sudo tee /etc/example/agent-config.yaml
server: https://<MASTER_NODE_IP>:6443
token: <NODE_TOKEN>
EOF

또는 CLI 조인 명령어 자체를 스크립트로 만들 수도 있다.

sudo kubeadm join <MASTER_NODE_IP>:6443 \
  --token <TOKEN> \
  --discovery-token-ca-cert-hash sha256:<HASH>

이렇게 하면 사람이 직접 편집기로 파일을 열고 토큰을 붙여넣는 과정을 줄일 수 있다.

다만 Shell Script만으로는 토큰 관리, 서버 목록 관리, 실패 처리 같은 부분이 부족할 수 있다.

11. Ansible로 자동화하기

워커 노드가 여러 대라면 Ansible 같은 도구가 더 적합하다.

Ansible은 여러 서버에 같은 작업을 반복 실행할 때 유용하다.

예를 들어 이런 흐름을 만들 수 있다.

워커 노드 목록 정의
↓
각 워커 노드에 접속
↓
필요한 패키지 설치
↓
설정 YAML 생성
↓
서비스 시작
↓
조인 상태 확인

이렇게 하면 워커 노드가 1대든 10대든 같은 방식으로 조인시킬 수 있다.

중요한 점은 작업이 코드로 남는다는 것이다.

사람이 서버에 들어가서 무엇을 했는지 기억에 의존하지 않아도 된다.

12. Terraform과 연결하면 인프라 생성까지 자동화된다

Terraform은 서버, 네트워크, 보안 그룹 같은 인프라 자원을 코드로 만드는 도구다.

Terraform을 사용하면 워커 노드를 만드는 단계부터 자동화할 수 있다.

VPC 생성
Subnet 생성
Security Group 생성
마스터 노드 생성
워커 노드 생성
필요한 포트 오픈
초기 설치 스크립트 실행
워커 노드 조인

이렇게 되면 단순히 “워커 노드를 조인한다”가 아니라, 클러스터를 구성하는 전체 과정이 코드로 관리된다.

여기서 중요한 감각은 이거다.

CLI로 할 수 있는 작업은 스크립트화할 수 있고, 스크립트화할 수 있는 작업은 자동화할 수 있다.

13. LLM까지 붙이면 자연어 자동화로 확장될 수 있다

여기서 더 나아가면 LLM과도 연결할 수 있다.

예를 들어 사람이 이렇게 요청한다고 해보자.

워커 노드 2개를 새로 만들고 기존 마스터 노드에 조인시켜줘

그러면 내부적으로는 이런 작업이 필요하다.

인프라 코드 생성
↓
워커 노드 생성
↓
보안 그룹 확인
↓
필요 패키지 설치
↓
조인 토큰 확인
↓
워커 노드 설정 생성
↓
조인 실행
↓
노드 상태 확인

물론 실제 운영 환경에서는 권한, 보안, 승인, 검증 절차가 반드시 필요하다.

하지만 방향성은 분명하다.

사람이 직접 서버에 들어가 토큰을 복사하고 붙여넣던 작업이, 점점 코드와 자동화의 영역으로 넘어간다.

14. 내가 이 경험에서 이해한 것

처음에는 단순히 이런 경험이었다.

토큰을 복사했다
워커 노드 설정 파일에 붙여넣었다
노드가 붙었다

그런데 나중에 다시 생각해보니 이건 꽤 중요한 운영 흐름이었다.

수동 설정
↓
CLI 명령어
↓
스크립트화
↓
자동화 도구
↓
인프라 코드화
↓
자연어 기반 자동화 가능성

즉, 내가 한 일은 단순한 복붙이 아니라, 인프라 운영이 자동화로 발전하는 흐름의 가장 앞단을 경험한 것이었다.

15. 정리

워커 노드를 쿠버네티스 클러스터에 조인시키려면 노드 토큰이 필요하다.

처음에는 이 토큰을 직접 워커 노드의 YAML 설정 파일에 붙여넣어 조인했다.

server: https://<MASTER_NODE_IP>:6443
token: <NODE_TOKEN>

이 방식은 구조를 이해하기에는 좋지만, 사람이 직접 복붙하기 때문에 실수할 위험이 있다.

이후에는 CLI 명령어로 조인할 수도 있다는 것을 알게 되었다.

kubeadm join <MASTER_NODE_IP>:6443 \
  --token <TOKEN> \
  --discovery-token-ca-cert-hash sha256:<HASH>

그리고 더 나아가면 이 과정은 Shell Script, Ansible, Terraform 같은 도구로 자동화할 수 있다.

토큰 복붙
→ CLI 조인
→ 스크립트화
→ Ansible 자동화
→ Terraform 기반 인프라 자동화
→ LLM 기반 자연어 자동화

이 경험을 통해 느낀 것은 단순하다.

인프라 운영에서 중요한 것은 한 번 명령어를 성공시키는 것이 아니라, 같은 작업을 실수 없이 반복 가능하게 만드는 것이다.

처음에는 노드 토큰을 복사해서 붙여넣는 작은 작업처럼 보였지만, 결국 이 경험은 쿠버네티스 조인 방식과 인프라 자동화의 필요성을 이해하는 출발점이었다.
