[JMETER] JMeter로 API 부하 테스트를 해봤다는 건 무슨 뜻일까?

2026.05.17 Velog 원문

프로젝트를 만들다 보면 단순히 “기능이 동작한다”에서 끝나면 안 된다. 내 로컬에서 버튼 한 번 눌렀을 때 잘 되는 것과, 여러 사용자가 동시에 요청을 보냈을 때도 안정적으로 동작하는 것은 완전히 다른 문제다.

내가 만든 FastAPI Todo 서비스도 마찬가지였다.

이 프로젝트는 단순 Todo API가 아니라, FastAPI 기반 백엔드에 Prometheus, Grafana, cAdvisor, Node Exporter 같은 모니터링 스택을 붙여 운영 가능한 형태로 구성한 프로젝트였다. FastAPI 애플리케이션은 /metrics로 Prometheus 지표를 노출하고, Grafana에서는 컨테이너와 호스트 리소스를 확인할 수 있도록 구성했다.

여기서 JMeter는 “이 API가 여러 요청을 동시에 받으면 어떻게 되는지”를 확인하기 위해 사용했다.

1. JMeter를 왜 사용했나?

처음에는 API가 정상적으로 동작하는지만 보면 충분해 보인다.

예를 들어 Todo 서비스를 만들었다면 이런 것들을 확인한다.

GET /todos
POST /todos
PATCH /todos/{id}
DELETE /todos/{id}

Postman이나 브라우저에서 직접 호출해보고, 응답이 잘 오면 기능적으로는 성공이다.

그런데 운영 관점에서는 질문이 달라진다.

사용자가 1명일 때만 잘 되는가?
사용자가 10명이어도 괜찮은가?
동시에 50명이 요청하면 응답시간이 늘어나는가?
요청이 많아지면 에러가 발생하는가?
CPU나 메모리는 얼마나 올라가는가?

이런 것을 확인하기 위해 JMeter를 사용했다.

즉, JMeter는 단순 API 테스트 도구가 아니라 부하 테스트 도구로 사용했다.

2. 내가 테스트한 서비스 구조

테스트 대상은 FastAPI 기반 Todo 서비스였다.

구조는 대략 다음과 같았다.

사용자 요청
   ↓
FastAPI 애플리케이션
   ↓
Todo API 처리
   ↓
JSON 기반 저장소

그리고 운영 환경에서는 다음과 같은 모니터링 구성도 같이 붙어 있었다.

FastAPI
Prometheus
Grafana
cAdvisor
Node Exporter

FastAPI는 API 요청을 처리하고, Prometheus는 메트릭을 수집하고, Grafana는 그 메트릭을 시각화한다. cAdvisor는 컨테이너 리소스를 보고, Node Exporter는 서버 리소스를 본다.

그래서 JMeter로 부하를 주면서, 동시에 Grafana에서 CPU, 메모리, 컨테이너 상태를 같이 확인할 수 있었다.

3. JMeter 테스트 시나리오

JMeter에서 가장 중요한 건 “어떤 요청을 어떤 흐름으로 보낼 것인가”다.

나는 Todo 서비스의 실제 사용 흐름을 기준으로 시나리오를 잡았다.

1. Todo 목록 조회
2. Todo 생성
3. Todo 상태 변경
4. Todo 삭제

API 기준으로 보면 다음과 같다.

GET /todos
POST /todos
PATCH /todos/{id}
DELETE /todos/{id}

단순히 하나의 API만 계속 때리는 것이 아니라, 사용자가 실제로 서비스를 사용할 때 발생할 법한 흐름을 구성한 것이다.

4. 시나리오 1: Todo 목록 조회

가장 기본적인 요청은 목록 조회다.

GET /todos

사용자가 Todo 페이지에 들어오면 가장 먼저 목록을 조회한다. 그래서 읽기 요청은 가장 자주 발생하는 요청이라고 볼 수 있다.

이 요청을 통해 확인하고 싶었던 것은 다음과 같다.

조회 API의 평균 응답시간
동시 요청 시 응답시간 증가 여부
조회 요청에서 에러가 발생하는지

읽기 요청은 보통 쓰기 요청보다 가벼운 편이다. 그래서 부하 테스트에서 기준점처럼 볼 수 있다.

5. 시나리오 2: Todo 생성

다음은 Todo 생성 요청이다.

POST /todos

이 요청은 사용자가 새로운 Todo를 추가할 때 발생한다.

예를 들면 이런 데이터를 보낸다.

{
  "title": "JMeter 테스트하기",
  "completed": false
}

생성 요청은 단순 조회보다 무겁다. 서버가 요청 바디를 읽고, 데이터를 검증하고, 저장소에 반영해야 하기 때문이다.

이 요청을 통해 확인하고 싶었던 것은 다음과 같다.

쓰기 요청의 응답시간
동시 생성 요청 시 에러 발생 여부
저장소 처리 과정에서 병목이 생기는지

특히 이 프로젝트는 JSON 파일 기반 저장소를 사용했기 때문에, 동시에 여러 쓰기 요청이 들어올 때 안정적으로 처리되는지도 중요한 포인트였다.

6. 시나리오 3: Todo 상태 변경

Todo 서비스에서는 완료 여부를 바꾸는 기능도 있다.

PATCH /todos/{id}

예를 들면 Todo를 완료 처리할 때 이런 요청이 발생한다.

{
  "completed": true
}

PATCH 요청은 전체 데이터를 바꾸는 것이 아니라 일부 필드만 수정하는 요청이다.

이 시나리오에서는 다음을 확인했다.

부분 수정 API가 정상 동작하는지
동시 수정 요청에서 에러가 발생하는지
응답시간이 안정적인지

실제 서비스에서는 사용자가 체크박스를 누를 때마다 PATCH 요청이 발생할 수 있다. 그래서 단순하지만 꽤 자주 발생할 수 있는 요청이다.

7. 시나리오 4: Todo 삭제

마지막은 삭제 요청이다.

DELETE /todos/{id}

Todo를 삭제하는 상황도 실제 사용 흐름에 포함된다.

삭제 요청에서는 다음을 확인했다.

삭제 API의 응답시간
삭제 후 데이터 정합성
존재하지 않는 id 요청 시 예외 처리

부하 테스트에서는 정상 요청뿐만 아니라 실패 가능성이 있는 요청도 중요하다. 예를 들어 이미 삭제된 Todo를 다시 삭제하려고 하면 404 처리가 제대로 되는지도 볼 수 있다.

8. JMeter Thread Group 설정

JMeter에서는 Thread Group을 사용해서 가상의 사용자를 만든다.

쉽게 말하면 Thread 수는 “동시에 접속하는 사용자 수”라고 생각하면 된다.

예를 들어 다음과 같이 설정할 수 있다.

Number of Threads: 10
Ramp-up Period: 10초
Loop Count: 5

이 의미는 다음과 같다.

가상 사용자 10명이
10초 동안 서서히 접속하고
각 사용자가 시나리오를 5번 반복한다

처음부터 100명의 사용자를 한 번에 넣으면 순간 부하가 너무 급격하게 들어간다. 그래서 Ramp-up을 설정해서 사용자가 점점 늘어나는 상황을 만든다.

실제 서비스에서도 사용자가 완전히 동시에 한 번에 몰리기보다는 일정 시간 동안 점진적으로 증가하는 경우가 많기 때문이다.

9. 부하를 늘려가며 테스트

처음부터 큰 부하를 주기보다는 단계적으로 늘리는 방식으로 테스트했다.

예를 들면 이런 식이다.

1단계: 동시 사용자 10명
2단계: 동시 사용자 30명
3단계: 동시 사용자 50명
4단계: 동시 사용자 100명

이렇게 하는 이유는 어느 구간부터 성능이 무너지는지 보기 위해서다.

처음부터 100명으로 테스트하면 “안 된다”는 것만 알 수 있다. 하지만 10명, 30명, 50명, 100명으로 나눠서 보면 더 많은 것을 알 수 있다.

30명까지는 안정적이다
50명부터 응답시간이 증가한다
100명에서는 에러율이 증가한다

이런 식으로 서비스의 한계를 파악할 수 있다.

10. 확인한 지표

JMeter에서 주로 본 지표는 다음과 같다.

평균 응답시간
최대 응답시간
p95 응답시간
에러율
Throughput

각각의 의미는 다음과 같다.

평균 응답시간

전체 요청의 평균 응답시간이다.

요청 하나가 평균적으로 몇 ms 안에 응답되는가

평균 응답시간이 낮으면 전체적으로 빠르게 응답한다고 볼 수 있다.

하지만 평균만 보면 위험하다. 일부 요청이 매우 느려도 평균에 묻힐 수 있기 때문이다.

p95 응답시간

p95는 전체 요청 중 95%가 이 시간 안에 응답했다는 뜻이다.

예를 들어 p95가 800ms라면 다음과 같은 의미다.

전체 요청의 95%는 800ms 안에 응답했다
나머지 5%는 800ms보다 오래 걸렸다

운영에서는 평균보다 p95가 더 중요할 때가 많다. 왜냐하면 사용자가 체감하는 느린 요청은 평균이 아니라 꼬리 지연에서 발생하기 때문이다.

에러율

에러율은 전체 요청 중 실패한 요청의 비율이다.

전체 요청 1000개 중 10개 실패
→ 에러율 1%

부하가 증가했을 때 에러율이 올라간다면 서비스가 안정적으로 요청을 처리하지 못하고 있다는 뜻이다.

Throughput

Throughput은 초당 처리량이다.

초당 몇 개의 요청을 처리하는가

응답시간이 낮고 throughput이 높다면 좋은 상태다. 반대로 사용자가 늘어나는데 throughput은 늘지 않고 응답시간만 증가한다면 병목이 생긴 것이다.

11. JMeter만 본 게 아니라 Grafana도 같이 봤다

JMeter는 클라이언트 입장에서의 결과를 보여준다.

응답이 느린가?
에러가 나는가?
초당 요청을 얼마나 처리하는가?

하지만 이것만으로는 원인을 알기 어렵다.

예를 들어 응답시간이 느려졌다고 해보자.

그 이유는 여러 가지일 수 있다.

FastAPI 애플리케이션 코드가 느린 것일 수도 있고
컨테이너 CPU가 부족한 것일 수도 있고
메모리가 부족한 것일 수도 있고
호스트 서버 자체가 한계에 도달한 것일 수도 있다

그래서 JMeter로 부하를 주는 동시에 Grafana를 확인했다.

Grafana에서는 다음과 같은 것들을 볼 수 있었다.

FastAPI 컨테이너 CPU 사용률
FastAPI 컨테이너 메모리 사용률
컨테이너별 네트워크 사용량
호스트 CPU 사용률
호스트 메모리 사용률

이렇게 보면 단순히 “느리다”에서 끝나지 않고, “왜 느린지”를 추적할 수 있다.

12. 예를 들어 이런 식으로 해석할 수 있다

JMeter에서 응답시간이 증가했다고 해보자.

이때 Grafana에서 CPU도 같이 올라갔다면 다음처럼 해석할 수 있다.

요청량 증가
→ FastAPI 컨테이너 CPU 사용률 증가
→ 요청 처리 지연
→ JMeter 응답시간 증가

반대로 CPU는 여유로운데 응답시간이 늘어난다면 다른 원인을 봐야 한다.

파일 저장소 I/O 문제
동시 쓰기 처리 문제
애플리케이션 내부 로직 문제
네트워크 지연

이렇게 JMeter와 Grafana를 같이 보면 단순 부하 테스트가 아니라 운영 관점의 트러블슈팅으로 이어질 수 있다.

13. 면접에서 말한다면

면접에서 “JMeter를 어떻게 사용했나요?”라고 물어보면 이렇게 답할 수 있다.

FastAPI Todo 서비스의 CRUD API에 대해 JMeter로 부하 테스트를 구성했습니다.
실제 사용 흐름을 기준으로 GET /todos, POST /todos, PATCH /todos/{id}, DELETE /todos/{id} 요청을 하나의 시나리오로 만들었고, Thread 수를 늘려가며 동시 사용자 상황을 재현했습니다.

테스트에서는 평균 응답시간, p95 응답시간, 에러율, throughput을 확인했습니다.
또한 JMeter 결과만 본 것이 아니라 Prometheus와 Grafana를 함께 사용해서 FastAPI 컨테이너의 CPU, 메모리, 호스트 리소스 상태도 같이 관찰했습니다.

이를 통해 단순히 API가 동작하는지뿐만 아니라, 부하가 증가했을 때 어느 지점에서 응답시간이 늘어나고 리소스 병목이 발생하는지 확인했습니다.

이 정도면 “JMeter 써봤다”가 아니라, “운영 관점에서 부하 테스트를 해봤다”로 들린다.

14. 핵심 정리

JMeter를 사용한 이유는 단순 기능 확인이 아니라 부하 상황에서의 API 안정성을 확인하기 위해서였다.

테스트 대상은 FastAPI Todo 서비스였고, 시나리오는 실제 사용 흐름에 맞게 구성했다.

목록 조회
Todo 생성
Todo 상태 변경
Todo 삭제

그리고 Thread 수를 늘려가며 동시 사용자 상황을 만들었다.

확인한 지표는 다음과 같다.

평균 응답시간
p95 응답시간
에러율
Throughput

마지막으로 JMeter 결과만 본 것이 아니라 Prometheus, Grafana, cAdvisor, Node Exporter와 함께 보면서 컨테이너와 서버 리소스까지 같이 확인했다.

즉, 이 테스트의 핵심은 다음과 같다.

JMeter = 부하를 주는 도구
Prometheus = 메트릭을 수집하는 도구
Grafana = 상태를 시각화하는 도구
cAdvisor = 컨테이너 리소스를 보는 도구
Node Exporter = 서버 리소스를 보는 도구

결국 내가 한 것은 단순 API 테스트가 아니라, 운영 환경에서 API가 부하를 받을 때 어떤 상태가 되는지 관찰하는 실험이었다.
