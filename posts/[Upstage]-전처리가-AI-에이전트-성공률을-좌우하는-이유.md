AI 에이전트 자동화의 성패는 단순히 모델 성능만으로 결정되지 않는다.  
실제 운영 환경에서는 모델이 얼마나 똑똑한가보다, **모델이 읽는 데이터가 얼마나 안정적으로 정제되어 있는가**가 더 큰 영향을 준다.

특히 금융권 광고심의와 같은 업무에서는 광고안, 약관, 상품설명서, 규정집처럼 다양한 문서를 비교해야 한다. 이 문서들은 대부분 PDF, 이미지, 표, 다단 레이아웃, 중첩 구조를 포함하고 있다. 이런 문서를 제대로 읽어내지 못하면, 이후의 RAG 검색이나 LLM 판단도 신뢰하기 어렵다.

업스테이지의 광고심의 자동화 사례는 AI 에이전트를 운영 환경에 적용할 때 왜 **전처리 안정성**과 **파이프라인 설계**가 중요한지를 잘 보여준다.

---

# 광고심의 자동화가 어려운 이유

금융 상품 광고 하나가 외부에 공개되기까지는 여러 검토 단계를 거친다.

광고안, 상품 약관, 상품설명서, 관련 규정집을 비교하면서 광고 문구가 법적 기준과 내부 규정을 만족하는지 확인해야 한다. 담당자는 각 항목별로 통과 또는 불통과를 판단하고, 그 근거가 되는 원문 조항까지 찾아야 한다.

이 작업은 단순 질의응답이 아니다.

예를 들어 다음과 같은 문제가 생길 수 있다.

```text
광고안이 이미지라서 텍스트 추출이 되지 않음
약관 PDF 안의 표 구조가 깨져서 문맥이 잘못 인식됨
RAG가 의도와 다른 조항을 검색함
통과/불통과 판정은 하지만 근거 조항을 찾지 못함
````

겉으로 보면 LLM의 답변 품질 문제처럼 보일 수 있다.
하지만 실제 원인은 더 앞단에 있다.

바로 **문서를 제대로 읽어내지 못하는 전처리 단계**다.

---

# AI 광고심의 파이프라인 구조

광고심의 자동화 파이프라인은 다음과 같은 흐름으로 구성된다.

```text
AI-OCR / Document Parse
        ↓
규정 RAG 검색
        ↓
체크리스트 자동 생성
        ↓
광고안 정합성 검토
        ↓
Evidence 매핑
        ↓
Guardrail
```

각 단계는 독립적으로 보이지만 실제로는 강하게 연결되어 있다.
앞 단계의 품질이 무너지면 뒤 단계의 결과도 함께 흔들린다.

---

# 1. AI-OCR / Document Parse

첫 번째 단계는 광고안이나 약관 문서를 읽어내는 것이다.

일반적인 OCR은 이미지나 PDF에서 텍스트를 추출하는 데 초점이 있다.
하지만 광고심의와 같은 업무에서는 단순 텍스트 추출만으로 부족하다.

문서에는 다음과 같은 구조가 포함되어 있기 때문이다.

```text
제목
소제목
표
각주
인덱스
이미지 속 텍스트
다단 레이아웃
중첩 표
```

이 구조가 보존되지 않으면 문서의 의미가 깨진다.

예를 들어 PDF가 이중 컬럼으로 작성되어 있는데, OCR 결과가 왼쪽 문단과 오른쪽 문단을 섞어서 추출한다면 RAG는 잘못된 문맥을 검색하게 된다. 표 안의 셀 데이터가 누락되면 금리, 조건, 예외 조항 같은 핵심 정보가 사라질 수도 있다.

따라서 Document Parse는 단순 OCR보다 더 넓은 역할을 한다.

```text
단순 OCR
이미지/PDF에서 글자 추출

Document Parse
텍스트 + 문서 구조 + 표 + 제목 + 레이아웃을 함께 인식
```

즉, Document Parse는 AI 에이전트가 문서를 실제로 이해할 수 있도록 **읽을 수 있는 데이터 형태로 재구성하는 전처리 단계**다.

---

# 2. 규정 RAG 검색

전처리된 문서는 이후 RAG 검색에 사용된다.

RAG는 사용자의 질문이나 검토 항목에 맞는 관련 규정, 약관, 조항을 검색하고 LLM이 이를 바탕으로 답변하도록 만드는 구조다.

하지만 RAG의 품질은 결국 입력 문서의 품질에 의존한다.

```text
문서 구조가 깨짐
→ 잘못된 chunk 생성
→ 잘못된 embedding 생성
→ 관련 없는 문서 검색
→ LLM 판단 오류
```

즉, RAG 검색 오류는 단순히 벡터 DB나 임베딩 모델만의 문제가 아닐 수 있다.
그 이전 단계에서 문서가 잘못 파싱되었기 때문에 생기는 문제일 수 있다.

AI DevOps 관점에서는 RAG 검색 품질을 볼 때 다음 항목을 함께 확인해야 한다.

```text
OCR 성공률
문서 파싱 실패율
chunk 생성 품질
검색 hit rate
top-k relevance
검색 latency
검색 실패율
```

---

# 3. 체크리스트 자동 생성

광고심의 업무에서는 규정을 그대로 보여주는 것만으로 충분하지 않다.

AI 에이전트는 관련 규정을 바탕으로 광고안 검토용 체크리스트를 생성해야 한다.

예를 들어 다음과 같은 항목이 만들어질 수 있다.

```text
필수 고지 문구가 포함되어 있는가
금리 조건이 명확하게 표시되어 있는가
과장 표현이 포함되어 있지 않은가
소비자 오인 가능성이 있는 문구가 있는가
상품 설명서와 광고 문구가 일치하는가
```

이 체크리스트는 이후 광고안 정합성 검토의 기준이 된다.

따라서 체크리스트 생성 단계에서도 중요한 것은 단순한 자연어 생성 능력이 아니다.
관련 규정을 정확히 검색하고, 그 규정을 업무 기준으로 변환하는 능력이 중요하다.

---

# 4. 광고안 정합성 검토

체크리스트가 생성되면 AI 에이전트는 광고안과 규정, 약관, 상품설명서를 비교한다.

이 단계에서는 각 항목별로 통과 또는 불통과를 판단한다.

```text
검토 항목: 필수 고지 문구 포함 여부
광고안 내용: 일부 문구 누락
관련 규정: 금융상품 광고 시 필수 고지사항 명시 필요
판정: 불통과
```

하지만 여기서 중요한 것은 단순히 판정 결과를 내는 것이 아니다.

운영 환경에서는 다음 질문에 답할 수 있어야 한다.

```text
왜 불통과인가?
어떤 문서의 어떤 조항에 근거했는가?
광고안의 어느 부분이 문제인가?
담당자가 원문을 확인할 수 있는가?
```

그래서 다음 단계인 Evidence 매핑이 중요해진다.

---

# 5. Evidence 매핑

Evidence 매핑은 AI 판단의 근거를 원문과 연결하는 단계다.

광고심의 자동화의 최종 결과물은 단순한 통과/불통과 판정이 아니다.
실무 담당자가 검토하고 신뢰할 수 있는 **Evidence Pack**이다.

Evidence Pack에는 다음과 같은 정보가 포함된다.

```text
체크리스트 기반 심의 결과
항목별 통과/불통과 판단
판정 근거 원문 인용
출처 문서 하이라이트
개선 권고사항
```

이 구조가 중요한 이유는 금융권 업무의 특성 때문이다.

금융권, 보험권, 공공기관과 같은 도메인에서는 AI가 단순히 “맞다”, “틀리다”라고 답하는 것만으로는 부족하다.
결과에 대한 책임과 검토 가능성이 필요하다.

즉, AI의 답변은 다음 조건을 만족해야 한다.

```text
근거가 있어야 함
출처가 명확해야 함
원문 확인이 가능해야 함
판단 과정이 추적 가능해야 함
사람이 최종 검토할 수 있어야 함
```

운영 단계에서 중요한 것은 “자동화가 되었는가”가 아니라, **담당자가 그 결과를 실제 업무에 사용할 수 있는가**다.

---

# 6. Guardrail

마지막 단계는 Guardrail이다.

Guardrail은 AI 시스템이 잘못된 답변을 하거나, 민감한 정보를 노출하거나, 근거 없는 판단을 생성하지 않도록 제어하는 장치다.

광고심의 자동화에서 Guardrail은 다음과 같은 역할을 할 수 있다.

```text
민감정보 마스킹
할루시네이션 검증
근거 없는 답변 차단
정책 위반 응답 방지
출처 누락 결과 탐지
```

특히 금융권 업무에서는 잘못된 답변이 실제 리스크로 이어질 수 있다.
따라서 Guardrail은 단순한 부가 기능이 아니라 운영 안정성을 위한 필수 구성 요소다.

---

# 전처리가 무너지면 어떤 일이 발생하는가

전처리 품질이 낮으면 이후 모든 단계가 영향을 받는다.

```text
표 안의 표
→ 구조 붕괴, 셀 데이터 누락

이미지 속 텍스트
→ 핵심 문구 추출 실패

이중 컬럼 레이아웃
→ 문단 순서 혼재

중첩 PDF
→ RAG 검색 오류

잘못된 chunk
→ 관련 없는 규정 검색

근거 매핑 실패
→ 담당자가 결과를 신뢰하지 못함
```

결국 전처리 실패는 OCR 단계에서 끝나지 않는다.
RAG 품질, LLM 답변, Evidence 매핑, Guardrail 판단까지 모두 흔들리게 만든다.

그래서 AI 에이전트 운영에서는 모델 성능만 보는 것이 아니라, 입력 데이터가 어떤 상태로 모델에 전달되는지를 함께 봐야 한다.

---

# 운영 확장을 위해 필요한 3가지 조건

## 1. 문서를 끝까지 읽는 전처리 구조

AI 시스템은 비정형 문서를 안정적으로 처리할 수 있어야 한다.

특히 실무 문서는 깔끔한 텍스트 파일로 주어지지 않는다.

```text
스캔 PDF
이미지 광고안
복잡한 표
다단 문서
중첩 표
각주와 부가 설명
문서별 다른 양식
```

이런 문서들을 안정적으로 처리하려면 OCR뿐 아니라 문서 구조를 이해하는 Document AI 파이프라인이 필요하다.

---

## 2. 근거 추적이 가능한 파이프라인

AI가 생성한 답변은 반드시 근거와 연결되어야 한다.

특히 광고심의처럼 규정 기반 판단이 필요한 업무에서는 다음 구조가 중요하다.

```text
검토 항목
→ 관련 규정
→ 광고안 내용
→ 판정 결과
→ 근거 문서
→ 원문 위치
```

이 연결이 가능해야 담당자가 결과를 검토할 수 있다.

근거 추적이 되지 않는 AI 답변은 운영 환경에서 신뢰받기 어렵다.

---

## 3. 에이전트 간 맥락이 끊기지 않는 연결

AI 에이전트 자동화는 단일 챗봇으로 끝나지 않는다.

실제 업무에서는 여러 단계의 에이전트가 협업한다.

```text
문서 파싱 에이전트
규정 검색 에이전트
체크리스트 생성 에이전트
검토 에이전트
근거 매핑 에이전트
Guardrail 에이전트
```

이때 중간 단계에서 컨텍스트가 끊기면 전체 결과 품질이 떨어진다.

따라서 업무 서비스 영역, 데이터 저장 구조, 모델 계층, 검색 시스템, 검증 시스템이 유기적으로 연결되어야 한다.

---

# AI DevOps 관점에서 봐야 할 지표

AI DevOps는 단순히 모델을 배포하는 역할에 그치지 않는다.
AI 기능이 실제 고객 업무에서 안정적으로 작동하도록 전체 파이프라인을 운영해야 한다.

광고심의 자동화 시스템에서는 다음 지표들을 볼 수 있다.

## OCR / Document Parse 단계

```text
문서 파싱 성공률
OCR 실패율
레이아웃 인식 실패율
표 추출 실패율
처리 latency
문서 크기별 처리 시간
재시도 횟수
```

## RAG 단계

```text
검색 hit rate
top-k relevance
chunk 품질
embedding 생성 실패율
vector DB query latency
검색 timeout 비율
잘못된 조항 검색 비율
```

## LLM 단계

```text
응답 latency
error rate
timeout
retry count
token usage
응답 실패율
모델별 비용
```

## Evidence 매핑 단계

```text
근거 조항 매핑 성공률
출처 누락률
원문 하이라이트 실패율
판정-근거 불일치율
```

## Guardrail 단계

```text
민감정보 마스킹 성공률
할루시네이션 탐지율
근거 없는 답변 차단율
false positive
false negative
정책 위반 응답 비율
```

이런 지표를 수집하고 모니터링해야 AI 에이전트 시스템을 운영 가능한 수준으로 만들 수 있다.

# 내가 경험한 DevOps 프로젝트와의 연결

FastAPI 기반 백엔드 서비스를 운영 환경에 가깝게 구성하면서 Docker, Jenkins, Prometheus, Grafana, JMeter를 사용한 경험이 있다.

이 경험은 AI DevOps 업무와 다음과 같이 연결될 수 있다.

```text
FastAPI
→ AI API 서버, OCR API, RAG API, LLM Gateway 구성

Docker
→ 모델/서비스 실행 환경 표준화

Jenkins
→ CI/CD 파이프라인 자동화

Prometheus
→ latency, error rate, resource usage 수집

Grafana
→ 운영 대시보드 구성

JMeter
→ API 부하 테스트 및 성능 검증

Troubleshooting
→ 배포 실패, timeout, 장애 원인 분석
```

AI 서비스도 결국 운영 시스템이다.
OCR, RAG, LLM, Guardrail 각각은 하나의 장애 지점이 될 수 있다.

따라서 AI DevOps는 각 컴포넌트가 안정적으로 연결되고, 문제가 생겼을 때 원인을 추적할 수 있도록 관측성과 자동화 구조를 만드는 역할이라고 볼 수 있다.

---

# 면접에서 활용할 수 있는 정리

## 질문 1. 광고심의 자동화에서 왜 전처리가 중요한가요?

광고심의는 광고안, 상품 약관, 상품설명서, 규정집을 비교해야 하는 업무입니다. 그런데 실제 문서는 이미지, PDF, 표, 다단 레이아웃처럼 비정형 구조가 많습니다. 이 문서들이 제대로 파싱되지 않으면 RAG 검색 단계에서 잘못된 문맥이 들어가고, 이후 LLM의 판단도 신뢰하기 어려워집니다. 그래서 전처리는 단순한 사전 작업이 아니라 전체 AI 에이전트 품질을 결정하는 핵심 단계라고 생각합니다.

---

## 질문 2. 단순 OCR과 Document Parse의 차이는 무엇인가요?

단순 OCR은 이미지나 PDF에서 텍스트를 추출하는 데 초점이 있습니다. 반면 Document Parse는 텍스트뿐만 아니라 제목, 표, 인덱스, 다단 레이아웃 같은 문서 구조까지 인식해서 RAG나 AI 에이전트가 이해할 수 있는 형태로 재구성하는 단계라고 이해했습니다.

---

## 질문 3. AI DevOps 입장에서 이런 시스템에서 무엇을 모니터링해야 하나요?

단계별로 나누어 봐야 한다고 생각합니다. Document Parse 단계에서는 파싱 성공률, 표 추출 실패율, 처리 시간을 보고, RAG 단계에서는 검색 hit rate, top-k relevance, chunk 품질을 봐야 합니다. LLM 단계에서는 latency, timeout, token usage, error rate를 보고, Evidence 매핑 단계에서는 근거 조항 연결 실패율이나 출처 누락률을 확인해야 합니다. Guardrail 단계에서는 민감정보 마스킹 실패나 할루시네이션 차단 여부를 모니터링할 수 있습니다.

---

## 질문 4. 왜 Evidence가 중요한가요?

금융권 업무에서는 AI가 단순히 답을 생성하는 것보다, 그 답이 어떤 규정과 원문에 근거했는지가 더 중요하다고 생각합니다. 담당자가 결과를 검토하고 책임질 수 있어야 하기 때문에, 판정 결과와 함께 원문 근거, 출처 문서, 하이라이트가 제공되어야 실제 운영에 사용할 수 있습니다.

---

## 질문 5. 이 글을 보고 AI DevOps를 어떻게 이해했나요?

AI DevOps는 단순히 모델을 배포하는 일이 아니라, OCR, RAG, LLM, Guardrail이 실제 고객 업무에서 신뢰 가능한 결과를 내도록 전체 파이프라인을 안정적으로 운영하는 일이라고 이해했습니다. 특히 전처리 품질, 근거 추적, 모니터링, 장애 대응, 성능 검증까지 포함해 AI 기능을 운영 가능한 서비스로 만드는 역할이라고 생각합니다.

---

# 핵심 요약

AI 에이전트 자동화의 성패는 모델 하나로 결정되지 않는다.

좋은 모델은 필요조건이지만 충분조건은 아니다.
문서를 정확히 읽고, 구조를 유지하며, 관련 규정을 검색하고, 판단 근거를 원문과 연결하고, Guardrail로 결과를 검증하는 전체 파이프라인이 필요하다.

특히 금융권 광고심의처럼 규정 기반 판단이 필요한 업무에서는 다음 조건이 중요하다.

```text
정확하게 읽기
일관되게 처리하기
근거를 추적하기
운영 지표를 모니터링하기
장애를 빠르게 파악하기
```

결국 AI DevOps는 모델을 잘 띄우는 역할을 넘어, AI가 실제 업무에서 신뢰 가능한 결과를 내도록 만드는 운영 구조를 설계하고 관리하는 역할이다.

---

# 한 줄 정리

AI DevOps는 모델 배포를 넘어, OCR·RAG·LLM·Guardrail이 실제 고객 업무에서 신뢰 가능한 결과를 내도록 전처리 품질, 근거 추적, 모니터링, 장애 대응까지 운영 구조로 묶는 일이다.
