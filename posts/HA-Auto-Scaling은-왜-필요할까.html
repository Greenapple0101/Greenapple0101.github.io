[HA] Auto Scaling은 왜 필요할까?

2026.05.11 Velog 원문

서비스를 운영하다 보면 가장 무서운 순간 중 하나는:

```text id=”7j1v1i” 갑자기 트래픽이 폭증하는 상황


이다.

예를 들어:

* 쇼핑몰 할인 이벤트
* 콘서트 티켓팅
* 게임 업데이트
* 유명 커뮤니티 유입
* 바이럴 영상

같은 상황이 발생하면 순간적으로 엄청난 요청이 몰린다.

문제는 서버 자원이 한정되어 있다는 점이다.

예:

* CPU 100%
* 메모리 부족
* 응답 지연
* 서버 다운

같은 문제가 발생한다.

그래서 등장한 개념이:

```text id="4g7x3h"
Auto Scaling

이다.

Auto Scaling이란?

Auto Scaling은 말 그대로:

```text id=”3d9q0o” 트래픽 상황에 따라 서버 개수를 자동으로 늘리거나 줄이는 기술


이다.

즉:

* 사용자가 많아지면 서버 증가
* 사용자가 줄어들면 서버 감소

를 자동으로 수행한다.

핵심 목표는:

```text id="4m5r2y"
성능 유지 + 비용 최적화

다.

왜 서버를 자동으로 늘려야 할까?

예를 들어 서버가 2대 있다고 가정해보자.

평소에는 충분하다.

하지만 이벤트가 시작되면:

```text id=”9z2p1x” 사용자 10배 증가


같은 상황이 발생할 수 있다.

이때 서버를 수동으로 추가하면 늦는다.

이미 장애가 발생한 뒤일 수도 있다.

그래서 시스템이 스스로:

```text id="u1o3g7"
“지금 트래픽 위험하다”

고 판단하고 자동으로 서버를 늘린다.

Scale Up vs Scale Out

서버 확장에는 크게 두 가지 방식이 있다.

1. Scale Up

기존 서버 성능 증가.

예:

  • CPU 업그레이드
  • 메모리 추가

즉:

```text id=”d9x0e8” 더 강한 서버 사용


이다.

하지만:

* 비용 비쌈
* 물리적 한계 존재
* 다운타임 발생 가능

같은 문제가 있다.

---

## 2. Scale Out

서버 개수 증가.

예:

```text id="1y0l7g"
Web Server 2대 → 10대

처럼 확장한다.

현대 클라우드는 대부분:

```text id=”x4k8v9” Horizontal Scaling (Scale Out)


을 선호한다.

그리고 Auto Scaling도 대부분 이 방식이다.

---

# 기본 구조

보통 이런 구조를 사용한다.

```text id="x6z9n2"
              ┌──────────────┐
              │ Load Balancer│
              └──────┬───────┘
                     │
      ┌──────────────┼──────────────┐
      │              │              │
┌─────────┐   ┌─────────┐   ┌─────────┐
│ Web #1  │   │ Web #2  │   │ Web #3  │
└─────────┘   └─────────┘   └─────────┘

트래픽이 증가하면:

```text id=”q9v7d5” 서버 자동 추가


트래픽이 감소하면:

```text id="8j5r4f"
서버 자동 제거

된다.

언제 확장할까?

Auto Scaling은 특정 조건을 기준으로 동작한다.

예:

  • CPU 사용률 70% 이상
  • 메모리 사용량 증가
  • 요청 수 증가
  • 응답 시간 증가

같은 조건을 감지한다.

예시:

```text id=”6x3j9v” CPU > 70% 5분 지속 → 서버 2대 추가


같은 정책을 설정한다.

---

# AWS Auto Scaling

AWS에서는 대표적으로:

```text id="m1t6s8"
Auto Scaling Group (ASG)

을 사용한다.

ASG는:

  • 최소 서버 수
  • 최대 서버 수
  • 원하는 서버 수

를 기준으로 자동 관리한다.

예:

```text id=”2g4d0n” Min: 2 Desired: 3 Max: 10


이면:

* 최소 2대 유지
* 기본 3대 운영
* 최대 10대까지 자동 증가

가능하다.

---

# 하지만 DB는 어렵다

여기서 중요한 현실이 있다.

```text id="n0f7w3"
애플리케이션 서버는 쉽게 늘릴 수 있지만
DB는 쉽지 않다

는 점이다.

왜냐면 DB는:

  • 상태(State)를 가지고 있고
  • 데이터 동기화가 필요하며
  • Consistency 문제가 발생하기 때문이다.

즉:

```text id=”y5d8m1” Web Server는 Stateless에 가깝고 DB는 Stateful하다


는 차이가 존재한다.

그래서 대부분 시스템은:

* Web/App 서버 → Auto Scaling 적극 사용
* DB → 제한적으로 확장

구조를 가진다.

---

# Kubernetes와 Auto Scaling

Kubernetes도 Auto Scaling 기능을 제공한다.

대표적으로:

```text id="p6w1v4"
HPA (Horizontal Pod Autoscaler)

가 있다.

예:

  • CPU 증가
  • 메모리 증가

시 Pod 개수를 자동으로 늘린다.

즉:

```text id=”k4s2t7” Pod 자동 확장


을 수행한다.

클라우드 환경에서는 거의 필수 기능이다.

---

# Auto Scaling의 핵심 철학

결국 Auto Scaling의 핵심은:

```text id="f2m8z5"
트래픽 변화에 시스템이 자동 적응

하는 것이다.

즉 운영자가 매번 수동으로 대응하지 않아도:

  • 성능 유지
  • 장애 방지
  • 비용 절약

을 동시에 노린다.

마무리

현대 서비스의 트래픽은 예측하기 어렵다.

갑자기 수십 배의 요청이 몰릴 수도 있고, 반대로 새벽에는 거의 사용자가 없을 수도 있다.

그래서 클라우드 환경에서는:

text id="d5r0x2" 필요할 때 늘어나고 필요 없을 때 줄어드는 구조

가 매우 중요해졌다.

그리고 Auto Scaling은:

  • 클라우드
  • Kubernetes
  • DevOps
  • MSA

시대의 핵심 인프라 기술 중 하나가 되었다.
