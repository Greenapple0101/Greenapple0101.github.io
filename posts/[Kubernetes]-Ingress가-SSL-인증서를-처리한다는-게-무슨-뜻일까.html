[Kubernetes] Ingress가 SSL 인증서를 처리한다는 게 무슨 뜻일까?

Fri, 29 Ma Velog 원문

쿠버네티스 Ingress를 공부하다 보면 이런 설명이 나온다.

Ingress는 HTTP/HTTPS를 통해 클러스터 내부 서비스를 외부로 노출한다.

그리고 기능으로는 보통 이런 것들이 나온다.

Service에 외부 URL 제공
트래픽 로드밸런싱
SSL 인증서 처리
Virtual Hosting 지정

처음 보면 여기서 막힌다.

“Ingress가 SSL 인증서를 처리한다는 게 뭐지?” “SSL은 HTTPS 붙이는 거 아닌가?” “그럼 Service가 HTTPS를 처리하는 게 아니라 Ingress가 처리하는 건가?”

이걸 이해하려면 먼저 외부 사용자가 쿠버네티스 내부 서비스에 들어오는 흐름을 봐야 한다.

1. Ingress는 왜 필요할까?

쿠버네티스 안에는 보통 여러 Service가 있다.

frontend-service
backend-service
admin-service
api-service

이 Service들은 기본적으로 클러스터 내부에서 사용된다.

외부 사용자가 접근하려면 Service를 NodePort나 LoadBalancer로 열 수도 있다.

하지만 서비스가 많아질수록 문제가 생긴다.

서비스마다 포트를 따로 열어야 함
서비스마다 LoadBalancer를 만들면 비용 증가
도메인별 라우팅이 어려움
HTTPS 인증서 관리가 복잡함

예를 들어 이런 식으로 운영하고 싶다고 하자.

https://example.com          → frontend-service
https://example.com/api      → backend-service
https://admin.example.com    → admin-service

이걸 Service만으로 깔끔하게 처리하기는 어렵다.

그래서 Ingress를 쓴다.

Ingress는 외부 HTTP/HTTPS 요청을 받아서, 조건에 따라 내부 Service로 보내주는 입구 역할을 한다.

외부 사용자
   ↓
Ingress
   ↓
Service
   ↓
Pod

2. Ingress와 Ingress Controller는 다르다

여기서 먼저 중요한 구분이 있다.

Ingress
= 라우팅 규칙을 적어둔 Kubernetes 리소스

Ingress Controller
= 그 규칙을 실제로 동작시키는 컨트롤러

Ingress YAML은 이런 규칙을 적는다.

example.com으로 들어오면 frontend-service로 보내라.
example.com/api로 들어오면 backend-service로 보내라.
HTTPS 인증서는 이 Secret을 써라.

하지만 Ingress 리소스 자체가 트래픽을 직접 처리하지는 않는다.

실제로 외부 요청을 받고, Nginx나 HAProxy 같은 프록시 설정을 만들고, 트래픽을 Service로 보내는 건 Ingress Controller 다.

대표적인 Ingress Controller는 이런 것들이 있다.

NGINX Ingress Controller
Traefik
HAProxy Ingress
AWS Load Balancer Controller
GCE Ingress Controller

즉 정리하면 이렇다.

Ingress
= 규칙

Ingress Controller
= 규칙을 보고 실제 트래픽을 처리하는 실행자

3. SSL 인증서 처리란?

SSL 인증서 처리는 쉽게 말하면 HTTPS 요청을 받아서 암호화 통신을 처리하는 것 이다.

우리가 브라우저에서 이런 주소로 접속한다고 하자.

https://example.com

그러면 브라우저와 서버 사이에는 TLS 통신이 맺어진다.

이때 서버는 인증서를 보여준다.

브라우저: 너 example.com 맞아?
서버: 응, 이 인증서 봐.
브라우저: 인증서 유효하네. 그럼 암호화 통신 시작.

이 과정을 통해 사용자는 HTTPS로 안전하게 통신할 수 있다.

쿠버네티스에서는 이 인증서를 보통 Secret 에 저장하고, Ingress가 그 Secret을 참조하게 만든다.

4. Ingress가 SSL을 처리한다는 말의 의미

정확히 말하면 Ingress 리소스가 직접 SSL을 처리하는 것은 아니다.

실제로 SSL/TLS 처리를 하는 것은 Ingress Controller다.

하지만 사용자는 Ingress YAML에 이런 식으로 설정한다.

tls:
  - hosts:
      - example.com
    secretName: example-tls

이 설정의 의미는 이렇다.

example.com으로 HTTPS 요청이 들어오면
example-tls Secret 안에 있는 인증서와 개인키를 사용해서
TLS 연결을 처리해라.

즉 “Ingress가 SSL 인증서를 처리한다”는 말은 실무적으로는 이런 뜻이다.

Ingress에 TLS 설정을 적는다.
Ingress Controller가 해당 Secret의 인증서를 읽는다.
외부 HTTPS 요청을 Ingress Controller가 받아서 TLS를 종료한다.
그 뒤 내부 Service로 트래픽을 전달한다.

5. TLS Termination이란?

Ingress에서 SSL 인증서 처리를 이해할 때 가장 중요한 말이 TLS Termination 이다.

Termination은 “끝낸다”는 뜻이다.

TLS Termination은 쉽게 말하면 이거다.

HTTPS 암호화 연결을 Ingress Controller에서 끝낸다.

흐름을 보면 이렇다.

사용자 브라우저
   ↓ HTTPS
Ingress Controller
   ↓ HTTP
Service
   ↓
Pod

외부 사용자는 HTTPS로 접속한다.

하지만 Ingress Controller가 인증서를 이용해서 HTTPS 처리를 끝내고, 내부 Service로는 HTTP로 넘길 수 있다.

외부 구간: HTTPS
내부 구간: HTTP

이런 구조를 TLS Termination이라고 한다.

6. 왜 Ingress에서 TLS를 끝낼까?

인증서를 각 애플리케이션 Pod마다 넣어서 처리할 수도 있다.

하지만 그렇게 하면 운영이 복잡해진다.

예를 들어 서비스가 5개 있다고 하자.

frontend
backend
admin
payment
monitoring

각 Pod마다 인증서를 넣고, HTTPS 설정을 하고, 갱신까지 직접 처리하면 관리가 힘들다.

반면 Ingress Controller에서 TLS를 처리하면 구조가 단순해진다.

외부 HTTPS 처리는 Ingress Controller가 담당
내부 애플리케이션은 HTTP로 단순하게 동작
인증서 관리는 Ingress 쪽에서 집중 관리

즉 Ingress에서 SSL을 처리하면 이런 장점이 있다.

인증서 관리 위치가 한 곳으로 모임
애플리케이션은 HTTPS 설정을 몰라도 됨
도메인별 인증서를 Ingress에서 관리 가능
서비스가 많아져도 구조가 깔끔함

7. TLS Secret은 어떻게 생겼을까?

쿠버네티스에서 TLS 인증서는 보통 Secret으로 만든다.

kubectl create secret tls example-tls \
  --cert=example.crt \
  --key=example.key

이 명령은 TLS 타입 Secret을 만든다.

example.crt
= 인증서 파일

example.key
= 개인키 파일

생성된 Secret은 이런 식으로 확인할 수 있다.

kubectl get secret example-tls

출력 예시는 이렇다.

NAME          TYPE                DATA   AGE
example-tls   kubernetes.io/tls   2      1m

여기서 타입이 중요하다.

kubernetes.io/tls

이 타입의 Secret은 TLS 인증서와 개인키를 담기 위한 Secret이다.

8. Ingress YAML 예시

이제 Ingress에서 TLS를 설정하는 예시를 보자.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-ingress
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - example.com
      secretName: example-tls
  rules:
    - host: example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 80

핵심은 이 부분이다.

tls:
  - hosts:
      - example.com
    secretName: example-tls

이 뜻은 다음과 같다.

example.com 도메인에 대한 HTTPS 요청은
example-tls Secret에 들어 있는 인증서를 사용해서 처리한다.

그리고 아래 부분은 라우팅 규칙이다.

rules:
  - host: example.com
    http:
      paths:
        - path: /
          pathType: Prefix
          backend:
            service:
              name: frontend-service
              port:
                number: 80

이 뜻은 이렇다.

example.com으로 들어온 요청 중
/ 경로로 들어온 요청은
frontend-service의 80번 포트로 보낸다.

즉 이 Ingress는 두 가지 일을 정의한다.

TLS 설정
= 어떤 인증서를 쓸지

Routing 설정
= 어느 Service로 보낼지

9. HTTPS 요청이 들어왔을 때 흐름

사용자가 브라우저에서 접속한다.

https://example.com

전체 흐름은 이렇게 된다.

[사용자 브라우저]
    |
    | HTTPS 요청
    v
[Ingress Controller]
    |
    | example-tls Secret의 인증서로 TLS 처리
    v
[HTTP 요청으로 변환]
    |
    v
[frontend-service]
    |
    v
[frontend Pod]

조금 더 자세히 보면 이렇다.

1. 사용자가 https://example.com 접속
2. DNS가 example.com을 Ingress Controller의 외부 IP로 해석
3. 요청이 Ingress Controller로 들어옴
4. Ingress Controller가 example-tls Secret의 인증서를 사용
5. TLS Handshake 수행
6. HTTPS 암호화 구간 종료
7. 내부 Service로 HTTP 요청 전달
8. Service가 Pod로 트래픽 전달

이게 Ingress의 SSL 인증서 처리 흐름이다.

10. 여러 도메인 인증서도 처리할 수 있을까?

가능하다.

Ingress는 여러 host에 대해 서로 다른 TLS Secret을 지정할 수 있다.

spec:
  tls:
    - hosts:
        - app.example.com
      secretName: app-tls
    - hosts:
        - admin.example.com
      secretName: admin-tls

이렇게 하면 각각 다른 인증서를 사용할 수 있다.

app.example.com
→ app-tls 인증서 사용

admin.example.com
→ admin-tls 인증서 사용

또는 와일드카드 인증서를 사용할 수도 있다.

*.example.com

이 경우 여러 서브도메인을 하나의 인증서로 처리할 수 있다.

11. Virtual Hosting과 TLS

Ingress는 Virtual Hosting도 지원한다.

Virtual Hosting은 하나의 Ingress Controller가 여러 도메인을 처리하는 방식이다.

예를 들어 이런 구조다.

app.example.com      → app-service
admin.example.com    → admin-service
api.example.com      → api-service

Ingress에서는 host별로 라우팅 규칙을 나눈다.

rules:
  - host: app.example.com
    http:
      paths:
        - path: /
          pathType: Prefix
          backend:
            service:
              name: app-service
              port:
                number: 80

  - host: admin.example.com
    http:
      paths:
        - path: /
          pathType: Prefix
          backend:
            service:
              name: admin-service
              port:
                number: 80

여기에 TLS Secret까지 붙이면 도메인별 HTTPS 처리가 가능하다.

tls:
  - hosts:
      - app.example.com
    secretName: app-tls
  - hosts:
      - admin.example.com
    secretName: admin-tls
  - hosts:
      - api.example.com
    secretName: api-tls

즉 Ingress는 단순히 외부 URL을 제공하는 것만이 아니라, 도메인별 라우팅과 HTTPS 인증서 처리까지 같이 담당한다.

12. cert-manager를 쓰면 인증서 자동 발급도 가능하다

실무에서는 인증서를 직접 만들고 Secret에 넣는 방식보다 cert-manager를 많이 사용한다.

cert-manager는 Kubernetes 안에서 인증서를 자동으로 발급하고 갱신해주는 도구다.

예를 들어 Let’s Encrypt와 연동하면 이런 흐름이 가능하다.

Ingress에 인증서 필요
   ↓
cert-manager가 인증서 발급 요청
   ↓
Let’s Encrypt가 도메인 검증
   ↓
TLS Secret 자동 생성
   ↓
Ingress Controller가 해당 Secret 사용

즉 사람이 직접 인증서를 다운받고 Secret을 만들지 않아도 된다.

인증서 만료 전에 자동 갱신도 가능하다.

그래서 실무에서는 보통 이런 조합을 많이 쓴다.

Ingress Controller
+ cert-manager
+ Let’s Encrypt

13. 자주 헷갈리는 포인트

Ingress만 만들면 HTTPS가 바로 되나?

아니다.

Ingress에 TLS 설정이 있어야 하고, 해당 Secret이 존재해야 한다.

또 실제로 트래픽을 처리할 Ingress Controller도 설치되어 있어야 한다.

Ingress 리소스만 있음
→ 규칙만 있는 상태

Ingress Controller 없음
→ 실제 요청 처리 불가

TLS Secret 없음
→ 인증서 사용 불가

Secret만 만들면 HTTPS가 되나?

아니다.

Secret은 인증서를 저장하는 곳일 뿐이다.

Ingress에서 그 Secret을 참조해야 한다.

tls:
  - hosts:
      - example.com
    secretName: example-tls

그리고 Ingress Controller가 이 설정을 읽어야 한다.

Service에도 HTTPS 설정을 해야 하나?

일반적인 TLS Termination 구조에서는 Service와 Pod는 HTTP로 동작해도 된다.

외부 사용자 → HTTPS → Ingress Controller → HTTP → Service → Pod

물론 내부 구간까지 HTTPS로 하고 싶다면 Pod 애플리케이션도 HTTPS를 처리하도록 구성할 수 있다.

하지만 처음 공부할 때는 보통 Ingress에서 HTTPS를 끝내고, 내부는 HTTP로 보내는 구조부터 이해하면 된다.

인증서는 어디에 저장되나?

쿠버네티스에서는 보통 TLS Secret에 저장한다.

인증서 파일 → tls.crt
개인키 파일 → tls.key

Secret 타입은 보통 다음과 같다.

kubernetes.io/tls

15. 그림으로 정리

Ingress에서 SSL 인증서를 처리하는 흐름은 이렇게 볼 수 있다.

[사용자 브라우저]
    |
    | HTTPS 요청
    v
[Ingress Controller]
    |
    | TLS Secret의 인증서로 HTTPS 처리
    v
[HTTP 요청]
    |
    v
[Service]
    |
    v
[Pod]

조금 더 구성 요소 중심으로 보면 이렇다.

[Ingress Resource]
    |
    | host, path, tls.secretName 정의
    v
[Ingress Controller]
    |
    | 설정 반영
    v
[TLS Secret]
    |
    | tls.crt / tls.key 제공
    v
[HTTPS 요청 처리]
    |
    v
[Service로 라우팅]

16. 한 번에 정리

Ingress는 HTTP/HTTPS 요청을 기준으로 클러스터 내부 Service를 외부에 노출하는 Kubernetes 리소스다.

Ingress가 SSL 인증서를 처리한다는 말은 정확히는 이 뜻이다.

Ingress에 TLS 설정을 적고,
Ingress Controller가 TLS Secret의 인증서를 읽어서,
외부 HTTPS 요청을 처리한다.

핵심 요소는 다음과 같다.

Ingress
= host/path 라우팅 규칙과 TLS 설정을 정의

Ingress Controller
= 실제 외부 요청을 받고 HTTPS/TLS를 처리

TLS Secret
= 인증서와 개인키를 저장

TLS Termination
= Ingress Controller에서 HTTPS 암호화 연결을 끝내고 내부로 전달

결국 흐름은 이렇게 정리할 수 있다.

사용자
   ↓ HTTPS
Ingress Controller
   ↓ HTTP
Service
   ↓
Pod

처음에는 이것만 기억하면 된다.

Ingress는 외부 URL을 Service에 연결한다.
TLS Secret은 인증서를 저장한다.
Ingress Controller는 그 인증서로 HTTPS를 처리한다.

그래서 Ingress는 단순한 라우팅 규칙이 아니라, 외부 트래픽의 입구에서 HTTPS 처리, 도메인 기반 라우팅, 경로 기반 라우팅까지 담당하는 중요한 네트워크 리소스라고 보면 된다.
