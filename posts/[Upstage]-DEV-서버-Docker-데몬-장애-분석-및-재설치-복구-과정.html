[Upstage] DEV 서버 Docker 데몬 장애 분석 및 재설치 복구 과정

2026.05.30

Jenkins 파이프라인에서 DEV 서버 배포 단계가 실패하면서 시작된 Docker 데몬 장애 분석 및 복구 과정을 정리한 문서다. 처음에는 Jenkins 파이프라인 자체 문제처럼 보였지만, 실제 원인은 Jenkins가 SSH로 접속한 DEV 서버 내부의 Docker daemon 장애였다. 최종적으로 Docker와 containerd 관련 데이터 및 메타데이터를 초기화한 뒤 Docker를 재설치하여 복구했다.

1. 문제 상황 개요

1-1. 발생한 증상

Jenkins 파이프라인의 DEV 배포 단계에서 Docker 명령이 실패했다.

실패한 명령은 다음과 같았다.

docker rm dev-conv || true
docker run -d -p 8000:8000 ...
````

에러 메시지는 다음과 같았다.

```text
Cannot connect to the Docker daemon at unix:///var/run/docker.sock

이 메시지는 Docker CLI가 Docker daemon과 통신하지 못한다는 뜻이다.

즉, docker 명령어 자체를 못 찾는 문제가 아니라, Docker 명령어는 존재하지만 백그라운드에서 떠 있어야 할 Docker daemon이 정상 동작하지 않는 상태였다.

2. Jenkins 파이프라인 실패 원인 확인

2-1. DEV 배포 단계 실패

Jenkins 파이프라인은 DEV 서버에 SSH로 접속한 뒤 Docker 명령을 실행하는 구조였다.

배포 단계에서는 기존 컨테이너를 삭제하고 새 컨테이너를 실행해야 했다.

docker rm dev-conv || true
docker run -d -p 8000:8000 ...

하지만 Docker daemon 연결 실패로 인해 컨테이너 삭제와 실행이 모두 실패했다.

2-2. 이후 단계 전부 skip

DEV 배포 단계가 실패하면서 이후 단계는 모두 실행되지 않았다.

DEV Deploy 실패
  ↓
Load Test skipped
  ↓
PR 생성 skipped
  ↓
Auto Merge skipped
  ↓
PROD Deploy skipped
  ↓
Cleanup skipped

Jenkins 파이프라인에서는 앞 단계가 실패하면 뒤 단계가 자동으로 skip될 수 있다.

따라서 Load Test나 PROD 배포가 실패한 것이 아니라, DEV 배포가 실패했기 때문에 이후 단계가 실행되지 않은 것이다.

2-3. 원인 위치 판단

처음에는 Jenkins 서버 자체 문제일 가능성도 검토했다.

하지만 Jenkins가 직접 Docker를 실행하는 것이 아니라, SSH로 DEV 서버에 접속한 뒤 DEV 서버 안에서 Docker 명령을 실행하는 구조였다.

따라서 문제 위치는 다음처럼 정리할 수 있다.

Jenkins 서버 문제 아님
Jenkinsfile 문법 문제 아님
Docker 이미지 문제 아님
DEV 서버 내부 Docker daemon 문제

최종 판단은 다음과 같았다.

Jenkins 서버가 아니라 DEV 서버의 Docker daemon 장애

3. DEV 서버 Docker 상태 점검

3-1. 수동 Docker 실행 테스트

DEV 서버에 직접 접속해서 Docker가 정상 동작하는지 확인했다.

docker run hello-world

하지만 Jenkins에서 발생한 것과 동일하게 Docker daemon 연결 실패가 발생했다.

Cannot connect to the Docker daemon at unix:///var/run/docker.sock

이 테스트를 통해 Jenkins 문제가 아니라 DEV 서버 Docker 자체 문제임을 확인할 수 있었다.

3-2. Docker 서비스 상태 확인

Docker daemon은 systemd 서비스로 관리된다.

상태 확인 명령어는 다음과 같다.

sudo systemctl status docker

확인 결과 Docker 서비스는 실패 상태였다.

Active: failed (Result: exit-code)
docker.service: Main process exited, status=1/FAILURE

3-3. 장애 상태 해석

이 상태는 Docker daemon이 시작되다가 바로 종료된다는 의미다.

즉, 다음과 같은 단순 문제가 아니었다.

Docker 명령어 미설치
사용자 권한 부족
일시적인 연결 문제

실제 상태는 다음에 가까웠다.

Docker 엔진이 기동 시도
  ↓
내부 오류 발생
  ↓
Docker daemon 종료
  ↓
systemd가 재시도
  ↓
반복 실패 후 failed 상태

4. 장애 원인 추론

4-1. containerd 메타데이터 손상 가능성

Docker는 내부적으로 containerd를 사용한다.

containerd는 컨테이너 런타임을 관리하는 구성 요소이며, 이미지 레이어, 스냅샷, 컨테이너 메타데이터 등을 관리한다.

과거에 다음 경로의 파일을 삭제한 이력이 있었다.

/var/lib/containerd/.../snapshots/*

이 이력 때문에 containerd의 snapshotter 또는 metadata가 손상되었을 가능성을 검토했다.

4-2. Docker와 containerd 관계

Docker를 실행하면 내부적으로 containerd도 함께 사용된다.

구조는 다음과 같이 볼 수 있다.

docker CLI
  ↓
Docker daemon
  ↓
containerd
  ↓
runc
  ↓
container

따라서 containerd의 데이터나 메타데이터가 손상되면 Docker daemon이 정상적으로 시작하지 못할 수 있다.

4-3. 복구 방향 결정

처음에는 단순 재시작을 시도할 수도 있었다.

sudo systemctl restart docker

하지만 Docker daemon 자체가 계속 실패 상태였기 때문에 단순 재시작으로는 해결되기 어렵다고 판단했다.

복구 방향은 다음과 같이 정했다.

Docker/containerd 서비스 중지
Docker/containerd 관련 데이터 삭제
손상된 메타데이터 초기화
Docker 재설치
hello-world로 검증

5. Docker 재설치 1차 시도

5-1. Docker 관련 패키지 제거

Docker와 containerd, runc 관련 패키지를 제거하려고 했다.

sudo apt remove docker docker-engine docker.io containerd runc
sudo apt purge docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

패키지 이름은 설치 방식에 따라 다를 수 있으므로, 실제 환경에서는 설치된 패키지를 확인해야 한다.

dpkg -l | grep docker
dpkg -l | grep containerd

5-2. Docker/containerd 데이터 삭제

Docker와 containerd 관련 데이터 디렉토리도 삭제했다.

sudo rm -rf /var/lib/docker
sudo rm -rf /var/lib/containerd
sudo rm -rf /etc/docker
sudo rm -rf /run/containerd

각 경로의 의미는 다음과 같다.

/var/lib/docker
Docker 이미지, 컨테이너, 볼륨, 네트워크 데이터 저장 위치

/var/lib/containerd
containerd 스냅샷, 이미지 레이어, 메타데이터 저장 위치

/etc/docker
Docker daemon 설정 파일 위치

/run/containerd
containerd 런타임 상태 파일 위치

5-3. get-docker.sh 사용

Docker 공식 설치 스크립트를 내려받아 실행했다.

curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

이 스크립트는 Docker 공식 저장소를 설정하고, Docker Engine 관련 패키지를 설치한다.

5-4. 설치 중 오해 발생

설치 중 apt 출력이 적어 멈춘 것처럼 보이는 구간이 있었다.

이때 중간에 Ctrl+C로 설치를 중단한 적이 있어 반쯤 설치된 상태가 되었을 가능성이 있었다.

설치 중단은 다음과 같은 문제를 만들 수 있다.

일부 패키지만 설치됨
서비스 파일은 생겼지만 의존성이 불완전함
containerd와 docker 버전이 애매하게 꼬임
systemd 서비스가 정상 기동하지 않음

5-5. apt 프로세스 확인

설치가 계속 진행 중인지 확인하기 위해 apt 프로세스를 확인했다.

ps aux | grep apt

apt 프로세스가 없어서 설치가 중단되었거나 실패한 것으로 판단했다.

6. 재설치 후 추가 장애 분석

6-1. Docker 명령어 자체는 복구

재설치 후 Docker 명령어와 containerd 명령어는 확인되었다.

docker --version
containerd --version

둘 다 정상적으로 버전을 출력했다.

이것은 Docker CLI와 관련 바이너리는 설치되었다는 의미다.

6-2. 하지만 docker.service는 정상 기동하지 않음

Docker 서비스 상태를 다시 확인했다.

sudo systemctl status docker

상태는 다음과 비슷했다.

Active: activating (start)

정상 상태는 다음이어야 한다.

Active: active (running)

즉, Docker가 설치되기는 했지만 daemon이 완전히 올라오지 못하고 있었다.

6-3. 로그 확인

Docker 서비스 로그를 확인했다.

journalctl -u docker -n 50

로그를 통해 Docker daemon이 시작 과정에서 실패하고 재시도하는 흐름을 확인했다.

이 시점에서도 containerd snapshotter 또는 metadata 손상 가능성이 계속 남아 있었다.

7. containerd 런타임 디렉토리 완전 초기화

7-1. Docker/containerd 중지

먼저 Docker와 containerd를 완전히 중지했다.

sudo systemctl stop docker
sudo systemctl stop containerd || true

containerd가 이미 죽어 있을 수 있으므로 || true를 붙여도 된다.

7-2. 런타임/데이터 디렉토리 제거

Docker와 containerd 관련 디렉토리를 다시 확실하게 삭제했다.

sudo rm -rf /var/lib/containerd
sudo rm -rf /run/containerd
sudo rm -rf /var/lib/docker
sudo rm -rf /etc/docker

이번 복구에서 특히 중요했던 것은 containerd 관련 디렉토리를 완전히 지우는 것이었다.

sudo rm -rf /var/lib/containerd
sudo rm -rf /run/containerd

7-3. 삭제 확인

삭제 후 실제로 디렉토리가 없는지 확인했다.

ls -l /var/lib/containerd
ls -l /var/lib/docker

결과는 다음과 같았다.

No such file or directory

이 상태는 Docker/containerd 데이터가 초기화되었다는 의미다.

8. Docker 재설치 성공

8-1. Docker 재설치 재진행

Docker 공식 설치 스크립트를 다시 실행했다.

sudo sh get-docker.sh

이번에는 설치를 중단하지 않고 끝까지 완료했다.

8-2. Docker 서비스 시작

설치 후 Docker 서비스를 시작했다.

sudo systemctl start docker

부팅 시 자동 시작되도록 enable도 설정했다.

sudo systemctl enable docker

8-3. Docker daemon 정상 상태 확인

서비스 상태를 확인했다.

sudo systemctl status docker

정상 상태는 다음과 같다.

Active: active (running)

이 상태가 되어야 Jenkins에서 Docker 명령을 사용할 수 있다.

8-4. hello-world 검증

최종적으로 Docker 공식 테스트 이미지를 실행했다.

docker run hello-world

성공 메시지는 다음과 같았다.

Hello from Docker!
This message shows that your installation appears to be working correctly.

이 메시지가 나오면 Docker daemon, containerd, 이미지 pull, 컨테이너 실행까지 정상이라는 뜻이다.

9. 최종 복구 결과

9-1. 복구된 항목

이번 복구로 다음 항목들이 정상화되었다.

docker CLI
Docker daemon
containerd
Docker image pull
Docker container run
hello-world 실행

9-2. Jenkins 파이프라인 관점의 의미

Docker가 복구되었으므로 Jenkins 파이프라인의 DEV 배포 단계도 다시 실행 가능한 상태가 되었다.

기존 실패 지점은 다음이었다.

docker rm dev-conv || true
docker run -d -p 8000:8000 ...

Docker daemon이 정상화되었으므로 해당 명령들이 다시 동작할 수 있는 상태가 되었다.

10. 중간에 검토한 대안

10-1. DEV 서버 전체 초기화

중간에 DEV 서버 전체를 초기화하거나 새 인스턴스로 재생성하는 방법도 검토했다.

이 방법은 가장 확실하지만 비용이 크다.

장점:
서버 상태를 완전히 깨끗하게 만들 수 있음
복잡한 Docker 내부 문제를 추적하지 않아도 됨

단점:
기존 설정을 다시 해야 함
보안 그룹, 키, 환경 변수, Jenkins SSH 설정 등을 다시 맞춰야 함
시간이 더 걸릴 수 있음

10-2. Docker/containerd만 복구

최종적으로는 서버 전체 초기화 대신 Docker와 containerd만 복구하는 방향을 선택했다.

Docker/containerd 정지
관련 데이터 삭제
Docker 재설치
hello-world 검증

이 방법으로 성공했기 때문에 서버 전체 초기화는 하지 않아도 되었다.

11. 최종 원인 정리

11-1. 직접 원인

직접 원인은 DEV 서버의 Docker daemon 장애였다.

Cannot connect to the Docker daemon at unix:///var/run/docker.sock

이 오류는 Docker daemon이 정상 실행 중이 아니라는 뜻이다.

11-2. 추정 근본 원인

근본 원인은 containerd snapshot 또는 metadata 손상 가능성이 높았다.

특히 과거에 다음 경로를 수동으로 삭제한 이력이 영향을 줬을 가능성이 있다.

/var/lib/containerd/.../snapshots/*

또한 Docker 재설치 중 Ctrl+C로 설치를 중단한 적이 있어, 중간 설치 상태가 문제를 더 복잡하게 만들었을 가능성도 있다.

11-3. 최종 해결 방법

최종 해결 방법은 다음과 같다.

Docker/containerd 서비스 중지
Docker/containerd 관련 데이터 디렉토리 삭제
Docker 공식 스크립트로 재설치
Docker 서비스 start/enable
docker run hello-world로 검증

12. 재현 가능한 복구 명령어 정리

아래는 이번 상황과 비슷하게 Docker daemon이 손상되었을 때 사용할 수 있는 복구 명령어 흐름이다.

주의할 점은 /var/lib/docker를 삭제하면 기존 이미지, 컨테이너, 볼륨 데이터가 삭제된다는 것이다.

운영 서버에서는 반드시 백업 여부를 먼저 확인해야 한다.

# 1. Docker/containerd 중지
sudo systemctl stop docker || true
sudo systemctl stop containerd || true

# 2. 기존 Docker 관련 패키지 제거
sudo apt remove -y docker docker-engine docker.io containerd runc || true
sudo apt purge -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin || true

# 3. 손상 가능성이 있는 데이터 및 설정 제거
sudo rm -rf /var/lib/docker
sudo rm -rf /var/lib/containerd
sudo rm -rf /etc/docker
sudo rm -rf /run/containerd

# 4. 삭제 확인
ls -l /var/lib/docker
ls -l /var/lib/containerd

# 5. Docker 공식 설치 스크립트 다운로드
curl -fsSL https://get.docker.com -o get-docker.sh

# 6. Docker 재설치
sudo sh get-docker.sh

# 7. Docker 시작 및 자동 시작 등록
sudo systemctl start docker
sudo systemctl enable docker

# 8. 상태 확인
sudo systemctl status docker

# 9. 정상 동작 검증
docker --version
containerd --version
docker run hello-world

13. Jenkins 파이프라인 재검증 순서

Docker 복구 후에는 Jenkins 파이프라인을 다시 실행해야 한다.

재검증 순서는 다음과 같다.

1. Jenkins에서 파이프라인 재실행
2. DEV 배포 단계 성공 여부 확인
3. DEV 서버에서 컨테이너 실행 여부 확인
4. DEV 서버 애플리케이션 접속 확인
5. Load Test 단계 실행 여부 확인
6. PR 생성 단계 실행 여부 확인
7. Auto Merge 단계 실행 여부 확인
8. PROD 배포 단계 실행 여부 확인
9. Cleanup 단계 실행 여부 확인

DEV 서버에서 컨테이너 상태는 다음 명령어로 확인할 수 있다.

docker ps
docker logs dev-conv

서비스 접속은 다음처럼 확인할 수 있다.

curl -i http://localhost:8000

또는 외부에서 접속 가능한 주소가 있다면 해당 주소로 확인한다.

14. 이번 장애에서 얻은 교훈

14-1. Jenkins 실패가 항상 Jenkins 문제는 아니다

Jenkins 파이프라인에서 실패가 발생하면 Jenkinsfile 문제처럼 보일 수 있다.

하지만 이번 경우 실제 문제는 Jenkins 서버가 아니라 SSH 대상인 DEV 서버 내부 Docker daemon 문제였다.

따라서 배포 파이프라인 문제는 다음처럼 나눠서 봐야 한다.

Jenkinsfile 문제
Jenkins credential 문제
SSH 접속 문제
대상 서버 문제
Docker daemon 문제
애플리케이션 이미지 문제
컨테이너 실행 문제

14-2. Docker CLI와 Docker daemon은 다르다

docker --version이 정상 출력된다고 해서 Docker가 정상 동작하는 것은 아니다.

Docker CLI는 명령어 클라이언트이고, 실제 컨테이너 실행은 Docker daemon이 담당한다.

따라서 반드시 다음까지 확인해야 한다.

sudo systemctl status docker
docker run hello-world

14-3. containerd 데이터 손상은 Docker 기동 실패로 이어질 수 있다

Docker는 내부적으로 containerd를 사용한다.

따라서 containerd의 snapshot, metadata, runtime directory가 꼬이면 Docker daemon도 정상 기동하지 못할 수 있다.

14-4. 설치 중 Ctrl+C는 위험하다

apt 설치나 Docker 설치 스크립트 실행 중 중단하면 패키지가 반쯤 설치된 상태가 될 수 있다.

이 상태에서는 명령어는 존재하지만 서비스는 정상 실행되지 않는 애매한 문제가 생길 수 있다.

14-5. 최종 검증은 hello-world로 한다

Docker 복구 후 가장 확실한 검증은 다음 명령이다.

docker run hello-world

이 명령이 성공하면 최소한 다음이 정상이라는 뜻이다.

Docker daemon 실행
containerd 실행
이미지 pull 가능
컨테이너 생성 가능
컨테이너 실행 가능
표준 출력 확인 가능

15. 최종 요약

이번 장애는 Jenkins 파이프라인의 DEV 배포 단계에서 Docker 명령이 실패하면서 발견되었다.

처음에는 Jenkins 문제처럼 보였지만, 실제 원인은 DEV 서버 내부의 Docker daemon 장애였다.

핵심 에러는 다음이었다.

Cannot connect to the Docker daemon at unix:///var/run/docker.sock

상태 확인 결과 Docker 서비스는 다음과 같이 실패 상태였다.

Active: failed (Result: exit-code)

원인은 containerd snapshot 또는 metadata 손상 가능성이 높았고, 중간 설치 중단 이력도 문제를 악화시켰을 가능성이 있었다.

최종 복구는 다음 순서로 진행했다.

Docker/containerd 중지
Docker/containerd 데이터 디렉토리 삭제
Docker 공식 스크립트로 재설치
Docker 서비스 start/enable
docker run hello-world로 검증

복구 후 hello-world 실행에 성공하면서 DEV 서버 Docker 환경은 정상화되었다.

이제 Jenkins 파이프라인은 다음 순서로 다시 검증하면 된다.

DEV 배포
Load Test
PR 생성
Auto Merge
PROD 배포
Cleanup

이번 경험의 핵심은 Jenkins 파이프라인 실패를 단순히 Jenkinsfile 문제로 보지 않고, 실제 명령이 실행되는 대상 서버의 상태까지 추적해야 한다는 점이다.
