[Upstage] Solar LLM은 대체 뭐고 어디에 쓰는 걸까

Solar LLM은 대체 뭐고, 어디에 쓰는 걸까?

Upstage를 이해하려면 Solar LLM을 알아야 한다.

Upstage는 OCR, Document Parse, Embedding, Information Extraction 같은 문서 AI API도 제공하지만, 그중에서 답변 생성과 추론을 담당하는 핵심 언어 모델 라인이 Solar다.

쉽게 말하면 Solar LLM은 이런 역할이다.

사용자 질문을 이해하고
문서를 읽고
요약하고
추론하고
답변을 생성하는 언어 모델

조금 더 서비스 관점으로 말하면:

내 서비스
  ↓
Solar LLM API 호출
  ↓
Solar 모델이 답변 생성
  ↓
JSON / streaming 응답 반환

즉, Solar LLM은 “내가 직접 모델 서버를 구축하지 않아도 API로 호출해서 쓸 수 있는 Upstage의 언어 모델”이라고 보면 된다.

LLM이 뭔데?

LLM은 Large Language Model의 줄임말이다.

한국어로 하면 대형 언어 모델이다.

사람이 쓰는 자연어를 입력받고, 그 문맥에 맞는 텍스트를 생성하는 모델이다.

예를 들면 이런 일을 한다.

질문 답변
문서 요약
번역
분류
코드 생성
보고서 작성
문서 기반 질의응답

예를 들어 사용자가 이렇게 물어본다.

OCR API가 뭐야?

그러면 LLM은 단순히 DB에서 정답을 꺼내는 게 아니라, 학습된 언어 패턴과 입력 문맥을 바탕으로 답변을 생성한다.

OCR API는 이미지나 PDF 안의 글자를 인식해서 텍스트로 변환하는 API입니다.

즉, LLM은 “텍스트를 이해하고 생성하는 모델”이다.

Solar는 그 LLM을 Upstage가 만든 모델 라인이라고 보면 된다.

Solar LLM은 Upstage에서 어떤 위치일까?

Upstage는 단순히 LLM 하나만 제공하는 회사라기보다는, 문서 기반 AI 업무 자동화에 필요한 여러 API를 제공하는 회사에 가깝다.

예를 들면 이런 식이다.

Document OCR
→ 이미지나 스캔 문서에서 글자 추출

Document Parse
→ PDF/문서를 LLM이 읽기 좋은 구조로 변환

Embedding
→ 문장을 벡터로 변환

Solar LLM
→ 질문에 대한 답변 생성

Groundedness Checker
→ 답변이 근거 문서에 기반했는지 확인

Upstage 공식 문서도 Chat API를 “지능형 어시스턴트 구축”에 쓰는 기능으로 설명하고, grounding, memory, function calling, 한국어 정밀도 같은 기능을 언급한다. (Upstage Console)

즉, Solar LLM은 Upstage API 묶음 안에서 생성/추론 담당이라고 보면 된다.

Solar LLM은 어디에 쓰일까?

Solar LLM은 보통 이런 곳에 붙는다.

챗봇
문서 요약
사내 지식 검색
고객 상담 자동화
OCR 결과 후처리
계약서/약관 질의응답
RAG 기반 문서 Q&A
업무 자동화 에이전트

예를 들어 회사 내부 문서 검색 챗봇을 만든다고 해보자.

사용자는 이렇게 묻는다.

우리 회사 휴가 규정에서 병가 기준이 뭐야?

그런데 LLM 혼자 답변하게 하면 틀릴 수 있다.

그래서 보통은 RAG 구조로 만든다.

사내 문서 수집
  ↓
Document Parse로 문서 구조화
  ↓
청킹
  ↓
Embedding
  ↓
Vector DB 저장
  ↓
사용자 질문 입력
  ↓
관련 문서 검색
  ↓
검색된 문서 + 질문을 Solar LLM에 전달
  ↓
Solar가 근거 기반 답변 생성

Upstage의 RAG 관련 공식 블로그도 기업 내부 RAG 애플리케이션에서 Google Workspace, Notion, GitHub, Linear 같은 여러 소스의 정보를 검색하고, 그 데이터를 사용해 챗봇 답변을 생성하는 구조를 설명한다. (Upstage AI)

Solar LLM API는 어떻게 생겼을까?

Solar LLM은 API로 호출할 수 있다.

기본적으로는 “질문을 보내고 답변을 받는 구조”다.

Python에서는 OpenAI SDK와 비슷한 방식으로 사용할 수 있다.

from openai import OpenAI

client = OpenAI(
    api_key="UPSTAGE_API_KEY",
    base_url="https://api.upstage.ai/v1"
)

response = client.chat.completions.create(
    model="solar-pro3",
    messages=[
        {
            "role": "user",
            "content": "Solar LLM이 뭐야?"
        }
    ],
    stream=False
)

print(response.choices[0].message.content)

여기서 중요한 값은 네 개다.

api_key
→ Upstage API 인증키

base_url
→ Upstage API 서버 주소

model
→ 사용할 Solar 모델명

messages
→ 모델에게 보낼 대화 내용

Upstage Console 쪽 문서에서는 Chat API로 지능형 어시스턴트를 만들 수 있고, Upstage 모델 카탈로그에서는 텍스트 생성, 문서 처리, 임베딩, 정보 추출 등 다양한 모델/API 범주를 제공한다고 설명한다. (Upstage Console)

messages는 뭐야?

LLM API에서 제일 중요한 건 messages다.

messages는 모델에게 전달하는 대화 내용이다.

[
  {
    "role": "system",
    "content": "너는 친절한 AI DevOps 면접 코치야."
  },
  {
    "role": "user",
    "content": "Solar LLM을 면접에서 어떻게 설명하면 좋을까?"
  }
]

여기서 role은 누가 말하는지를 뜻한다.

system
→ 모델의 역할, 규칙, 말투를 지정

user
→ 사용자의 질문

assistant
→ 이전 모델 답변

예를 들어 system에 이렇게 넣으면:

너는 AI DevOps 면접관이야. 답변을 현실적으로 평가해줘.

모델은 단순 설명이 아니라 면접 답변처럼 정리하려고 한다.

그래서 LLM API에서는 단순히 질문만 던지는 게 아니라, 어떤 역할로 답해야 하는지 프롬프트를 설계하는 것도 중요하다.

Solar 모델 라인은 어떻게 발전했을까?

Solar는 한 번 나온 모델 하나가 아니라, 여러 버전과 라인으로 발전해왔다.

Upstage는 2023년에 Solar 10.7B 모델이 Hugging Face Open LLM Leaderboard에서 높은 성과를 냈다고 발표했고, 이후 Solar Pro, Solar Pro 2 같은 모델을 계속 공개했다. (Upstage AI)

2025년 7월 발표된 Solar Pro 2는 31B 파라미터 규모의 모델로 소개되었고, Upstage는 이 모델이 다국어 처리, 고도화된 추론, 실제 업무에서의 도구 사용에 강점을 가진다고 설명했다. 특히 한국어 처리 성능과 실무 적용성을 강조했다. (Upstage AI)

2026년 자료에서는 Solar Pro 3도 언급된다. Upstage는 Solar Pro 3가 Upstage Console을 통해 API로 사용할 수 있고, 기존 Solar Pro 2 사용자가 큰 설정 변경 없이 사용할 수 있다고 설명한다. (Upstage AI)

정리하면 Solar는 이런 방향으로 발전해왔다고 보면 된다.

Solar 10.7B
→ 작지만 강한 오픈 모델 이미지

Solar Pro
→ 기업용 고성능 LLM

Solar Pro 2
→ 한국어, 추론, 도구 사용, 다국어 성능 강조

Solar Pro 3
→ API 기반 실서비스 테스트와 agentic 성능 강조

단, 모델명과 제공 옵션은 시간이 지나면서 바뀔 수 있으니 실제 개발할 때는 항상 Upstage Console의 모델 카탈로그를 기준으로 확인해야 한다.

Solar LLM과 일반 API는 뭐가 다를까?

게시판 API를 생각해보자.

GET /boards/1

이 API는 보통 DB에서 1번 게시글을 가져온다.

{
  "id": 1,
  "title": "첫 번째 글",
  "content": "안녕하세요"
}

즉, 일반 API는 이미 저장된 데이터를 조회하거나 수정하는 경우가 많다.

그런데 Solar LLM API는 다르다.

POST /chat/completions

요청에 질문과 문맥을 넣으면, 모델이 답변을 새로 생성한다.

{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "Solar LLM은 Upstage의 대형 언어 모델입니다."
      }
    }
  ]
}

즉, 차이는 이거다.

일반 API
→ 정해진 데이터를 조회/생성/수정/삭제

Solar LLM API
→ 입력 문맥을 바탕으로 답변을 생성

그래서 LLM API는 항상 운영상 불확실성이 있다.

응답 시간이 길 수 있음
답변 길이가 매번 다를 수 있음
토큰 비용이 발생함
가끔 근거 없는 답변을 만들 수 있음
프롬프트 설계에 따라 품질이 달라짐

Solar LLM과 RAG는 왜 같이 나오나?

Solar LLM을 설명할 때 RAG를 빼면 아쉽다.

RAG는 Retrieval-Augmented Generation의 줄임말이다.

한국어로 풀면 “검색 증강 생성”이다.

쉽게 말하면:

LLM 혼자 기억에 의존해서 답하게 하지 말고
먼저 관련 문서를 검색한 뒤
그 문서를 근거로 답변하게 만드는 구조

왜 필요할까?

LLM은 모든 최신 정보나 회사 내부 문서를 알고 있지 않다.

예를 들어 회사 내부 규정, 계약서, 고객사 문서, 기술 문서는 모델이 기본적으로 모를 수 있다.

그래서 RAG에서는 먼저 문서를 검색한다.

사용자 질문:
업스테이지 OCR API는 어떤 형식이야?

검색 결과:
OCR API는 이미지와 스캔 문서에서 텍스트를 추출한다...
응답에는 텍스트, 좌표, confidence 등이 포함될 수 있다...

Solar LLM 입력:
위 문서를 참고해서 답변해줘.

그러면 Solar LLM은 검색된 문서를 근거로 답변을 만든다.

OCR API는 이미지나 PDF에서 텍스트를 추출하고, 결과를 JSON 형태로 반환하는 API입니다...

즉, Solar LLM은 RAG에서 마지막 답변 생성기 역할을 한다.

RAG 안에서 Solar LLM의 위치

RAG 전체 흐름을 다시 보면 이렇다.

1. 문서 수집
2. Document Parse / OCR
3. 청킹
4. Embedding
5. Vector DB 저장
6. 사용자 질문 입력
7. 질문 Embedding
8. 유사 문서 검색
9. 검색 결과를 prompt에 넣기
10. Solar LLM으로 답변 생성
11. 필요하면 Groundedness 확인

Solar LLM은 여기서 10번이다.

검색된 근거 문서 + 사용자 질문
        ↓
Solar LLM
        ↓
최종 답변

그래서 Solar LLM은 단독 챗봇에도 쓸 수 있지만, Upstage의 Document Parse, Embedding API와 조합하면 문서 기반 업무 자동화에 더 잘 붙는다.

Upstage는 Document Parse를 복잡한 문서를 LLM이 처리하기 쉬운 형식으로 변환하는 제품으로 설명하고, RAG 구성에서 Solar LLM과 문서 처리/임베딩을 함께 활용하는 사례를 소개한다. (Upstage AI)

Solar LLM API 응답에서 뭘 봐야 할까?

LLM API 응답은 대충 이런 식으로 생긴다.

{
  "id": "chatcmpl_xxx",
  "object": "chat.completion",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Solar LLM은 Upstage의 대형 언어 모델입니다."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 120,
    "completion_tokens": 80,
    "total_tokens": 200
  }
}

여기서 중요한 건 두 가지다.

choices[0].message.content
→ 모델이 생성한 실제 답변

usage
→ 입력/출력 토큰 사용량

개발할 때는 보통 content만 출력하면 된다.

하지만 운영할 때는 usage도 봐야 한다.

왜냐하면 LLM API는 토큰 사용량이 latency와 비용에 영향을 줄 수 있기 때문이다.

prompt_tokens 증가
→ 입력이 길어짐
→ 처리 시간 증가 가능
→ 비용 증가 가능

completion_tokens 증가
→ 답변이 길어짐
→ 응답 시간 증가 가능
→ 비용 증가 가능

RAG에서는 검색된 문서 chunk를 많이 넣을수록 prompt token이 커진다.

그래서 무작정 context를 많이 넣는 게 좋은 게 아니다.

stream 옵션은 왜 중요할까?

LLM 답변은 생성 시간이 걸린다.

그래서 답변이 완성될 때까지 기다렸다가 한 번에 받으면 사용자가 답답할 수 있다.

이때 streaming을 쓴다.

stream=False
→ 답변이 완성된 뒤 한 번에 받음

stream=True
→ 답변이 생성되는 대로 조금씩 받음

챗봇에서 글자가 조금씩 나오는 것처럼 보이는 게 streaming 방식이다.

개발자 입장에서는 streaming이 조금 더 귀찮을 수 있지만, 사용자 경험은 좋아진다.

사용자 체감 속도 개선
긴 답변 대기 부담 감소
챗봇 UI에 적합

DevOps 관점에서 Solar LLM은 어떻게 봐야 할까?

면접에서는 이 부분이 중요하다.

Solar LLM을 “좋은 모델입니다” 정도로만 말하면 약하다.

AI DevOps라면 이렇게 봐야 한다.

이 API를 안정적으로 호출할 수 있는가?
느려지면 어떻게 감지할 것인가?
실패하면 재시도할 것인가?
토큰 비용은 어떻게 관리할 것인가?
API Key는 어디에 보관할 것인가?
사용자 요청이 몰리면 어떻게 제한할 것인가?
로그에 민감정보가 남지 않는가?

즉, Solar LLM을 DevOps 관점에서 보면 외부 AI API 의존성이다.

내 서비스가 Solar LLM API에 의존한다면, Solar 호출이 느려지거나 실패할 때 내 서비스도 영향을 받는다.

그래서 이런 지표를 봐야 한다.

latency
→ Solar API 응답 시간

p95 latency
→ 대부분은 괜찮아도 느린 요청이 얼마나 있는지

error rate
→ Solar API 호출 실패율

timeout
→ 정해진 시간 안에 응답이 안 온 요청

retry count
→ 재시도 횟수

token usage
→ 입력/출력 토큰 사용량

cost
→ API 비용

rate limit
→ 요청 제한 초과 여부

이건 그냥 이론이 아니라 실제 운영에서 바로 필요한 지표다.

Solar LLM 장애 시나리오

예를 들어 RAG 챗봇을 운영 중이라고 하자.

갑자기 사용자가 말한다.

답변이 너무 느려요.
가끔 답변이 안 와요.

이때 확인할 수 있는 흐름은 이렇다.

1. 전체 latency 확인
2. p95 latency 확인
3. Solar API error rate 확인
4. timeout 발생 수 확인
5. retry가 과도하게 발생하는지 확인
6. token usage가 갑자기 늘었는지 확인
7. 최근 prompt 변경이 있었는지 확인
8. 외부 API 상태나 rate limit 여부 확인

만약 token usage가 갑자기 늘었다면?

검색 chunk를 너무 많이 넣었을 수 있음
대화 히스토리를 너무 길게 넣었을 수 있음
prompt template이 길어졌을 수 있음
max output 설정이 너무 클 수 있음

만약 429 에러가 늘었다면?

rate limit 초과 가능성

Solar LLM을 Kubernetes에서 쓴다면?

직접 Solar 모델을 쿠버네티스에 올리는 게 아니라, Upstage API를 호출하는 애플리케이션을 쿠버네티스에 올린다고 생각하면 된다.

구조는 이렇게 될 수 있다.

사용자
  ↓
Ingress
  ↓
Spring/FastAPI Backend Pod
  ↓
Solar LLM API 호출
  ↓
응답 수신
  ↓
사용자에게 반환

Kubernetes에서는 이런 것들이 중요하다.

API Key를 Secret으로 관리
Pod 환경변수로 주입
timeout 설정
retry/backoff 설정
HPA로 백엔드 확장
Prometheus로 latency/error rate 수집
Grafana로 대시보드 구성
로그에서 prompt/개인정보 마스킹

예를 들어 API Key는 코드에 직접 넣으면 안 된다.

나쁜 예:

api_key = "up_xxxxxxxxxxxxx"

좋은 예:

import os

api_key = os.getenv("UPSTAGE_API_KEY")

Kubernetes에서는 이런 식으로 Secret을 만들어서 주입한다.

apiVersion: v1
kind: Secret
metadata:
  name: upstage-api-secret
type: Opaque
stringData:
  UPSTAGE_API_KEY: "your-api-key"

그리고 Deployment에서 환경변수로 넣는다.

env:
  - name: UPSTAGE_API_KEY
    valueFrom:
      secretKeyRef:
        name: upstage-api-secret
        key: UPSTAGE_API_KEY

면접에서 이 정도로 말하면 DevOps 관점이 살아난다.

Solar LLM을 RAG 프로젝트에 붙이면 이런 그림

너가 했던 RAG/MLOps 프로젝트 식으로 보면 이런 구조가 자연스럽다.

FastAPI RAG Server
  ↓
/ingest
  - 문서 수집
  - Document Parse/OCR
  - 청킹
  - Embedding
  - Vector DB 저장

/query
  - 사용자 질문 수신
  - 질문 Embedding
  - Vector Search
  - Prompt 구성
  - Solar LLM 호출
  - 답변 반환

/evaluate
  - recall@k
  - nDCG
  - answer similarity
  - latency 측정

/metrics
  - Prometheus metrics 노출

여기서 Solar LLM은 /query 안의 생성 단계에 들어간다.

retrieved context + user question
        ↓
Solar LLM API
        ↓
answer

그리고 MLOps 관점에서는 답변 품질과 운영 지표를 같이 본다.

품질 지표
- answer similarity
- groundedness
- hallucination rate
- user feedback

검색 지표
- recall@k
- nDCG
- hit rate

운영 지표
- latency
- p95 latency
- error rate
- timeout
- retry count
- token usage

이렇게 말하면 “Solar LLM을 그냥 써봤다”가 아니라, “운영형 RAG 시스템 안에서 Solar LLM 호출 지점을 관리한다”는 느낌이 난다.

Solar LLM의 장점은 어떻게 말하면 좋을까?

면접에서는 과장하지 말고 이렇게 말하면 좋다.

Solar LLM은 Upstage가 제공하는 언어 모델 API로,
한국어와 문서 기반 업무에 강점을 가진 모델 라인으로 이해했습니다.
특히 Upstage의 Document Parse, Embedding API와 같이 사용하면
기업 내부 문서 검색이나 RAG 서비스에 자연스럽게 붙일 수 있다고 봤습니다.

공식 자료 기준으로 Solar Pro 2는 한국어 처리, 추론, 다국어 처리, 도구 사용을 강조하고 있고, Solar Pro 3는 API로 실서비스 환경에서 테스트할 수 있는 모델로 소개된다. (Upstage AI)

그래서 Upstage 면접에서는 이렇게 연결하면 좋다.

저는 Solar LLM을 단순 챗봇 모델이라기보다,
문서 AI 파이프라인의 생성 엔진으로 이해했습니다.

예를 들어 Document Parse로 문서를 구조화하고,
Embedding으로 검색 가능한 형태로 만든 뒤,
검색된 context를 Solar LLM에 넣어 답변을 생성하는 식입니다.

DevOps 관점에서는 이 Solar 호출 구간의 latency, error rate,
timeout, retry, token usage를 모니터링하고,
API Key와 민감정보를 안전하게 관리하는 것이 중요하다고 생각합니다.

이 답변이 좋은 이유는 모델 설명에서 끝나지 않고, Upstage 서비스와 네 직무 관심사인 AI DevOps까지 연결되기 때문이다.

Solar LLM을 한 문장으로 정리하면

Solar LLM은 Upstage가 제공하는 대형 언어 모델 API로, 사용자의 질문과 문맥을 입력받아 답변을 생성하고, Document Parse·Embedding 같은 API와 함께 RAG/문서 자동화 서비스의 생성 엔진으로 사용될 수 있는 모델이다.

더 짧게 말하면:

Document Parse가 문서를 읽고
Embedding이 문서를 검색 가능하게 만들고
Solar LLM이 검색된 문서를 바탕으로 답변을 생성한다

그리고 DevOps 관점에서는:

Solar LLM은 외부 AI API 호출 지점이므로
latency, error rate, timeout, retry, token usage, p95 latency,
API Key 보안, 비용, rate limit을 함께 관리해야 한다

결국 Solar LLM을 이해한다는 건 모델 이름을 외우는 게 아니다.

문서가 들어와서, 검색되고, 근거가 붙고, 최종 답변이 생성되는 전체 AI 서비스 흐름에서 Solar가 어디에 들어가는지 이해하는 것이다.
