[GITHUB] GitHub Webhook은 도대체 뭘까?

2026.05.17 Velog 원문

프로젝트를 배포하다 보면 이런 흐름을 자주 보게 된다.

개발자가 GitHub에 코드를 push한다
→ Jenkins가 자동으로 빌드를 시작한다
→ 테스트를 돌린다
→ Docker 이미지를 만든다
→ 서버에 배포한다

처음에는 이게 되게 신기하다.

“내가 Jenkins에 들어가서 빌드 버튼을 누른 것도 아닌데, 왜 자동으로 실행되지?”

이때 GitHub와 Jenkins 사이에서 신호를 보내주는 역할을 하는 것이 바로 GitHub Webhook이다.

1. Webhook이란?

Webhook은 쉽게 말하면 어떤 이벤트가 발생했을 때 다른 서버에게 알려주는 자동 알림 장치다.

예를 들어 GitHub에서 이런 일이 발생할 수 있다.

코드 push
Pull Request 생성
Issue 생성
브랜치 삭제
태그 생성

이런 이벤트가 발생했을 때 GitHub가 외부 서버에게 HTTP 요청을 보낼 수 있다.

이 요청을 받는 쪽은 Jenkins일 수도 있고, Slack일 수도 있고, 직접 만든 백엔드 서버일 수도 있다.

즉 Webhook은 이런 구조다.

GitHub에서 이벤트 발생
→ GitHub가 지정된 URL로 HTTP 요청 전송
→ 외부 서비스가 요청을 받고 자동 작업 실행

2. GitHub Webhook은 왜 필요할까?

Webhook이 없으면 배포 과정이 수동이 된다.

예를 들어 Jenkins를 사용한다고 해보자.

Webhook이 없다면 개발자는 매번 이렇게 해야 한다.

1. GitHub에 코드 push
2. Jenkins 접속
3. Job 선택
4. Build Now 클릭
5. 빌드 결과 확인

작은 프로젝트에서는 괜찮아 보일 수 있다.

하지만 실무에서는 코드가 자주 바뀌고, 여러 명이 동시에 작업하고, 테스트와 배포가 반복된다.

이걸 매번 사람이 직접 누르면 실수가 생긴다.

그래서 보통은 이렇게 만든다.

개발자가 GitHub에 push
→ GitHub Webhook이 Jenkins에 알림
→ Jenkins가 자동으로 빌드 시작

이렇게 하면 사람이 직접 빌드 버튼을 누르지 않아도 된다.

3. GitHub Webhook의 기본 구조

GitHub Webhook은 크게 두 가지로 구성된다.

1. 어떤 이벤트를 감지할 것인가
2. 이벤트가 발생하면 어디로 요청을 보낼 것인가

예를 들어 다음과 같이 설정할 수 있다.

이벤트: push
요청 보낼 주소: http://my-jenkins-server/github-webhook/

그러면 GitHub는 해당 저장소에 push가 발생할 때마다 Jenkins 서버의 Webhook URL로 HTTP 요청을 보낸다.

구조는 이렇게 볼 수 있다.

Developer
   |
   | git push
   v
GitHub Repository
   |
   | Webhook HTTP POST
   v
Jenkins Server
   |
   | Build / Test / Deploy
   v
Application Server

4. Webhook은 결국 HTTP 요청이다

Webhook을 특별한 기술처럼 느낄 수 있지만, 본질은 단순하다.

GitHub가 외부 서버로 HTTP POST 요청을 보내는 것이다.

예를 들어 GitHub가 Jenkins에게 이런 식으로 알려주는 것이다.

"방금 이 저장소에 push가 발생했어."
"브랜치는 main이야."
"커밋한 사람은 누구야."
"커밋 메시지는 뭐야."
"변경된 파일은 뭐야."

GitHub는 이런 정보를 요청 본문에 담아서 보낸다.

Jenkins는 그 요청을 받고 다음 행동을 결정한다.

main 브랜치에 push가 왔네?
→ 빌드 시작

develop 브랜치에 push가 왔네?
→ 테스트만 실행

PR이 생성됐네?
→ 검증 파이프라인 실행

5. Jenkins와 GitHub Webhook 흐름

Jenkins를 기준으로 보면 흐름은 다음과 같다.

1. 개발자가 GitHub에 코드 push
2. GitHub Webhook이 Jenkins URL로 요청 전송
3. Jenkins가 해당 요청을 수신
4. Jenkins Job 또는 Pipeline이 실행됨
5. GitHub에서 코드를 pull/clone
6. 빌드, 테스트, 배포 수행

예를 들어 Spring Boot 프로젝트라면 Jenkins Pipeline에서 이런 작업을 할 수 있다.

GitHub에서 코드 가져오기
→ Gradle 또는 Maven 빌드
→ 테스트 실행
→ Docker 이미지 빌드
→ Docker Hub 또는 ECR에 push
→ EC2 서버에서 컨테이너 재시작

즉 GitHub Webhook은 직접 배포를 하는 도구는 아니다.

정확히 말하면 배포를 시작하게 만드는 트리거 역할을 한다.

6. Webhook과 CI/CD의 관계

CI/CD에서 GitHub Webhook은 보통 CI/CD 파이프라인의 출발점이다.

CI/CD 전체 흐름

코드 변경
→ GitHub push
→ Webhook 발생
→ Jenkins/GitHub Actions 실행
→ Build
→ Test
→ Docker Image 생성
→ Deploy

여기서 Webhook은 이 부분이다.

GitHub push
→ Webhook 발생
→ Jenkins 실행

즉 Webhook은 CI/CD 자체가 아니라, CI/CD를 자동으로 시작하게 해주는 장치다.

7. Webhook이 없을 때와 있을 때의 차이

Webhook이 없을 때

개발자가 직접 Jenkins에 들어감
직접 Build Now 클릭
빌드 실행 여부를 사람이 관리
실수 가능성 있음
반복 작업이 많음

Webhook이 있을 때

GitHub push만 하면 자동으로 Jenkins 실행
사람이 직접 버튼을 누를 필요 없음
커밋 단위로 자동 검증 가능
배포 흐름을 표준화할 수 있음

정리하면 Webhook은 수동 작업을 자동화하는 입구다.

8. GitHub Webhook 설정에서 자주 보는 항목

GitHub 저장소에서 Webhook을 설정하면 보통 이런 항목들이 나온다.

Payload URL

GitHub가 요청을 보낼 대상 URL이다.

예를 들어 Jenkins라면 보통 이런 형태를 사용한다.

http://jenkins-server/github-webhook/

또는 HTTPS를 사용하면 다음과 같다.

https://jenkins.example.com/github-webhook/

이 URL이 잘못되면 GitHub는 요청을 보내도 Jenkins가 받지 못한다.

Content type

GitHub가 데이터를 어떤 형식으로 보낼지 정하는 항목이다.

보통은 다음을 많이 사용한다.

application/json

JSON 형식으로 이벤트 정보를 보내겠다는 뜻이다.

Secret

Webhook 요청이 진짜 GitHub에서 온 것인지 검증하기 위한 값이다.

Webhook URL이 외부에 노출되면 누군가 임의로 요청을 보낼 수도 있다.

그래서 Secret을 설정해두면, 요청을 받는 서버가 다음을 확인할 수 있다.

이 요청이 정말 GitHub에서 온 요청인가?
중간에 위조된 요청은 아닌가?

운영 환경에서는 Secret을 설정하는 것이 좋다.

Events

어떤 이벤트가 발생했을 때 Webhook을 보낼지 정하는 부분이다.

대표적으로 다음 이벤트들이 있다.

push
pull request
issues
release
workflow run

CI/CD에서는 보통 push 또는 pull request 이벤트를 많이 사용한다.

9. GitHub Webhook과 GitHub Actions의 차이

처음에는 이 둘이 헷갈릴 수 있다.

GitHub Actions도 push하면 자동으로 실행된다.

Jenkins Webhook도 push하면 자동으로 실행된다.

둘 다 자동화와 관련이 있기 때문이다.

차이는 이렇다.

GitHub Webhook

GitHub에서 외부 서비스로 이벤트 알림을 보내는 기능
예: GitHub → Jenkins
예: GitHub → Slack
예: GitHub → 직접 만든 서버

GitHub Actions

GitHub 안에서 실행되는 자동화 플랫폼
예: GitHub 내부에서 build/test/deploy 실행

즉 GitHub Actions는 GitHub 자체의 CI/CD 도구이고, Webhook은 GitHub 밖의 서비스에게 이벤트를 전달하는 방식이다.

정리하면 다음과 같다.

GitHub Actions
= GitHub 안에서 자동화 실행

GitHub Webhook
= GitHub 밖의 시스템에게 이벤트 전달

10. GitHub Webhook과 Polling의 차이

Webhook을 이해할 때 Polling과 비교하면 더 쉽다.

Polling

Polling은 Jenkins가 GitHub에게 계속 물어보는 방식이다.

Jenkins: 새 코드 올라왔어?
GitHub: 아니

Jenkins: 새 코드 올라왔어?
GitHub: 아니

Jenkins: 새 코드 올라왔어?
GitHub: 응

즉 Jenkins가 주기적으로 GitHub를 확인한다.

Webhook

Webhook은 GitHub가 이벤트가 생겼을 때 바로 알려주는 방식이다.

GitHub: 새 코드 올라왔어. Jenkins야 실행해.

Polling은 계속 확인해야 하므로 불필요한 요청이 많다.

Webhook은 이벤트가 발생했을 때만 요청을 보내므로 더 효율적이다.

11. Webhook이 실패하는 흔한 이유

GitHub Webhook을 설정했는데 Jenkins가 실행되지 않는 경우가 있다.

이때는 보통 다음을 확인해야 한다.

1. Payload URL이 잘못됨

Jenkins Webhook URL이 틀렸을 수 있다.

http://jenkins-server/github-webhook/

끝에 /github-webhook/ 경로가 맞는지 확인해야 한다.

2. Jenkins 서버가 외부에서 접근 불가능함

GitHub는 외부 인터넷에서 Jenkins 서버로 요청을 보내야 한다.

그런데 Jenkins가 로컬 PC에서만 실행 중이라면 GitHub가 접근할 수 없다.

예를 들어 이런 주소는 GitHub가 접근할 수 없다.

http://localhost:8080/github-webhook/

localhost는 내 컴퓨터 안에서만 의미가 있기 때문이다.

외부에서 접근 가능한 서버 주소가 필요하다.

3. 방화벽 또는 보안 그룹 문제

EC2에 Jenkins를 띄운 경우 보안 그룹에서 포트가 열려 있어야 한다.

예를 들어 Jenkins가 8080 포트로 떠 있다면 다음이 필요하다.

인바운드 규칙에서 8080 포트 허용

단, 운영 환경에서는 Jenkins를 그대로 8080으로 외부 노출하기보다는 Nginx 리버스 프록시와 HTTPS를 붙이는 것이 더 안전하다.

4. Jenkins Job 설정 문제

GitHub에서 Webhook 요청이 정상적으로 갔더라도 Jenkins Job이 이를 받도록 설정되어 있지 않으면 빌드가 실행되지 않을 수 있다.

Jenkins에서는 보통 다음과 같은 설정이 필요하다.

GitHub hook trigger for GITScm polling

또는 Pipeline 설정에서 GitHub 이벤트를 받을 수 있도록 구성해야 한다.

5. 브랜치 조건 문제

Webhook은 들어왔지만 Jenkins Pipeline 내부 조건 때문에 실행이 안 될 수도 있다.

예를 들어 Pipeline이 main 브랜치만 배포하도록 되어 있으면 develop 브랜치 push에는 배포가 안 될 수 있다.

main push → 배포
develop push → 테스트만 실행
feature push → 실행 안 함

이런 조건을 Jenkinsfile에서 확인해야 한다.

12. GitHub Webhook을 실제 운영 관점에서 보면

운영 관점에서 Webhook은 단순히 “자동 빌드 시작”이 아니다.

더 크게 보면 사람의 수동 개입을 줄이고, 배포 흐름을 표준화하는 장치다.

예를 들어 Webhook을 통해 이런 흐름을 만들 수 있다.

main 브랜치에 push
→ Jenkins 자동 실행
→ 테스트 수행
→ 품질 검사
→ Docker 이미지 빌드
→ 서버 배포
→ 실패 시 배포 중단

이렇게 만들면 배포가 사람의 기억이나 손작업에 의존하지 않는다.

누가 push하더라도 같은 파이프라인을 거치게 된다.

이게 CI/CD 자동화의 핵심이다.

13. 면접에서 GitHub Webhook을 설명한다면

면접에서 “GitHub Webhook을 사용해봤나요?”라고 물어보면 이렇게 설명할 수 있다.

GitHub Webhook은 GitHub 저장소에서 push나 pull request 같은 이벤트가 발생했을 때 외부 서비스로 HTTP 요청을 보내는 기능입니다.

저는 GitHub에 코드가 push되면 Jenkins가 자동으로 빌드와 테스트를 실행하도록 Webhook을 연결했습니다.

Webhook 자체가 빌드나 배포를 수행하는 것은 아니고, Jenkins Pipeline을 실행시키는 트리거 역할을 했습니다.

이를 통해 개발자가 Jenkins에 직접 들어가 Build Now를 누르지 않아도 커밋 단위로 자동 검증과 배포 흐름을 시작할 수 있게 했습니다.

조금 더 운영 관점으로 말하면 이렇게도 설명할 수 있다.

Webhook을 통해 수동 배포 버튼 클릭을 줄이고, 코드 변경 이벤트를 기준으로 빌드·테스트·배포 파이프라인이 일관되게 실행되도록 구성했습니다.

특히 main 브랜치 push 시에만 배포가 일어나도록 브랜치 조건을 두고, 테스트 실패 시 배포가 진행되지 않도록 Jenkins Pipeline에서 gate 역할을 하게 했습니다.

14. 한 줄 정리

GitHub Webhook은 GitHub에서 이벤트가 발생했을 때 외부 시스템에게 알려주는 HTTP 기반 알림 기능이다.

CI/CD에서는 보통 GitHub의 push 이벤트를 Jenkins 같은 외부 빌드 서버에 전달해서 자동 빌드와 배포를 시작하는 트리거로 사용한다.

GitHub Webhook
= GitHub 이벤트를 외부 서비스에 알려주는 자동 신호

CI/CD에서의 역할
= Jenkins Pipeline을 자동으로 실행시키는 출발점
