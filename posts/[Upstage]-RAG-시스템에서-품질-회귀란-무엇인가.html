[Upstage] RAG 시스템에서 품질 회귀란 무엇인가

2026.05.30

RAG 시스템은 LLM이 외부 문서를 검색한 뒤, 검색된 문서를 근거로 답변을 생성하는 구조이다. 일반적인 LLM 서비스와 달리 RAG는 모델 하나만 잘 동작한다고 품질이 보장되지 않는다. 문서 수집, 전처리, 청킹, 임베딩, 벡터 DB, 검색, reranking, 프롬프트, LLM 응답 생성이 모두 연결되어 있기 때문이다.

이 구조에서는 작은 변경 하나가 전체 답변 품질에 영향을 줄 수 있다. 예를 들어 문서를 새로 추가했을 뿐인데 검색 결과 순위가 바뀔 수 있고, 청크 크기를 조금 조정했을 뿐인데 정답 근거가 잘려 나갈 수 있다. 임베딩 모델을 바꾸면 벡터 공간 자체가 달라지고, 프롬프트를 수정하면 같은 문서를 넣어도 LLM의 답변 방식이 달라질 수 있다.

이처럼 이전에는 잘 동작하던 RAG 시스템이 특정 변경 이후 더 나쁜 답변을 생성하는 현상을 품질 회귀라고 볼 수 있다.

품질 회귀란 무엇인가

품질 회귀는 시스템이 업데이트된 이후 기존보다 성능이나 품질이 떨어지는 현상이다. 일반적인 소프트웨어에서는 기능 회귀를 많이 이야기한다. 예를 들어 기존에는 로그인 기능이 잘 동작했는데, 배포 이후 로그인이 실패한다면 기능 회귀이다.

RAG에서는 이 개념이 조금 더 복잡하다. RAG 시스템은 API가 정상적으로 응답하더라도 답변 내용이 틀릴 수 있다. 서버는 200 OK를 반환하고, latency도 정상이고, timeout도 발생하지 않았지만, 사용자가 받은 답변은 부정확할 수 있다.

RAG 품질 회귀는 다음과 같은 형태로 나타날 수 있다.

  • 질문과 관련 없는 문서를 검색함
  • 정답 근거 문서를 검색하지 못함
  • 검색된 문서는 맞지만 답변이 부정확함
  • 문서에 없는 내용을 지어냄
  • 이전보다 답변이 장황하거나 모호해짐
  • 특정 유형의 질문에서만 품질이 떨어짐
  • 특정 고객사 또는 특정 도메인 문서에서만 답변 품질이 낮아짐

즉, RAG 품질 회귀는 단순한 장애가 아니다. 시스템은 정상적으로 동작하는 것처럼 보이지만, 실제 사용자가 느끼는 답변 품질은 떨어지는 상태이다.

RAG에서 품질 회귀가 발생하는 이유

RAG 시스템에서 품질 회귀가 자주 발생하는 이유는 RAG가 여러 단계로 구성된 파이프라인이기 때문이다. 각 단계는 독립적으로 보이지만, 실제로는 최종 답변 품질에 함께 영향을 준다.

1. 문서 데이터 변경

RAG는 외부 문서를 기반으로 답변한다. 따라서 문서가 추가, 삭제, 수정되면 검색 결과가 달라질 수 있다.

예를 들어 사내 매뉴얼 기반 챗봇에서 새로운 문서가 추가되었다고 하자. 새 문서가 기존 문서보다 표현이 더 일반적이거나 키워드가 많이 포함되어 있다면, 원래 정답 문서보다 새 문서가 검색 상위에 노출될 수 있다. 이 경우 사용자는 기존보다 부정확한 답변을 받을 수 있다.

문서 변경은 단순한 데이터 업데이트처럼 보이지만, RAG에서는 검색 품질과 답변 품질을 직접 흔드는 원인이 된다.

2. 청킹 전략 변경

청킹은 긴 문서를 검색 가능한 작은 단위로 나누는 과정이다. RAG에서는 문서를 그대로 벡터 DB에 넣기보다 일정한 크기의 청크로 나누어 저장하는 경우가 많다.

청크 크기가 너무 작으면 필요한 문맥이 잘려 나갈 수 있다. 반대로 청크 크기가 너무 크면 검색 정확도가 떨어질 수 있다. overlap 설정도 중요하다. 문맥이 이어지는 부분에서 overlap이 부족하면 조건과 예외 조항이 서로 다른 청크로 분리될 수 있다.

예를 들어 기존에는 하나의 정책 설명이 하나의 청크 안에 들어갔지만, 청킹 방식을 바꾸면서 조건과 예외 조항이 서로 분리되었다고 하자. 이 경우 검색은 성공해도 LLM이 충분한 문맥을 받지 못해 잘못된 답변을 생성할 수 있다.

3. 임베딩 모델 변경

임베딩 모델은 텍스트를 벡터로 변환하는 역할을 한다. RAG에서는 사용자의 질문과 문서 청크를 벡터로 변환한 뒤, 두 벡터의 유사도를 기준으로 관련 문서를 검색한다.

임베딩 모델을 교체하면 텍스트 간 유사도를 계산하는 기준이 달라진다. 같은 질문과 같은 문서를 사용하더라도 검색 순위가 바뀔 수 있다.

따라서 임베딩 모델 변경은 단순한 모델 교체가 아니다. 전체 검색 시스템의 의미 공간을 바꾸는 작업에 가깝다. 임베딩 모델을 바꾼 뒤에는 기존 평가셋을 이용해 검색 품질이 유지되는지 반드시 확인해야 한다.

4. 벡터 DB 재색인 문제

문서나 임베딩 모델이 바뀌면 벡터 DB를 다시 색인해야 한다. 하지만 재색인이 일부만 수행되거나, 구버전 임베딩과 신버전 임베딩이 섞이면 검색 품질이 떨어질 수 있다.

예를 들어 일부 문서는 이전 임베딩 모델로 색인되어 있고, 일부 문서는 새로운 임베딩 모델로 색인되어 있다고 하자. 이 경우 서로 다른 기준으로 만들어진 벡터가 같은 공간에서 비교된다. 그러면 유사도 점수가 일관되지 않고, 검색 결과가 불안정해질 수 있다.

이 문제는 단순 기능 테스트로는 잘 드러나지 않는다. API는 정상적으로 응답하지만, 검색 품질은 조용히 나빠질 수 있다.

5. 프롬프트 변경

RAG에서는 검색된 문서를 LLM에게 어떻게 전달하는지도 중요하다. 프롬프트는 LLM이 문서를 해석하고 답변을 생성하는 방식에 영향을 준다.

예를 들어 기존 프롬프트에는 “제공된 문서에 근거해서만 답변하라”는 지시가 있었는데, 새 프롬프트에서 이 문장이 약해졌다고 하자. 이 경우 LLM은 문서에 없는 내용을 더 자유롭게 추론할 수 있고, 환각이 늘어날 수 있다.

반대로 프롬프트를 너무 엄격하게 만들면 답변이 지나치게 짧아지거나, 문서에 충분한 근거가 있어도 답변을 회피할 수 있다. 따라서 프롬프트 변경도 품질 회귀의 원인이 될 수 있다.

6. LLM 모델 변경

LLM 자체를 교체해도 품질 회귀가 발생할 수 있다. 새 모델이 평균적으로 더 좋은 성능을 보이더라도, 특정 도메인이나 특정 질문 유형에서는 기존 모델보다 나쁜 답변을 할 수 있다.

예를 들어 새 모델이 요약 능력은 좋아졌지만, 긴 정책 문서에서 세부 조건을 엄격히 따지는 능력은 떨어질 수 있다. 또는 답변이 더 자연스러워졌지만, 근거 문서에 없는 내용을 섞어 말할 수도 있다.

따라서 LLM 모델 변경은 최신 모델을 적용하는 문제가 아니라, 기존 서비스 품질 기준을 유지하는지 검증하는 작업이다.

RAG 품질 회귀는 왜 탐지하기 어려운가

일반적인 백엔드 시스템에서는 error rate, latency, timeout, CPU 사용량, 메모리 사용량 등을 보면 장애를 어느 정도 파악할 수 있다. 하지만 RAG 시스템에서는 이런 지표만으로 품질 문제를 발견하기 어렵다.

RAG 시스템은 응답이 정상적으로 반환되어도 답변이 틀릴 수 있다. 즉, 시스템 지표는 정상인데 서비스 품질은 낮아지는 상황이 발생한다.

예를 들어 다음과 같은 상황이 가능하다.

  • API 응답 코드는 200 OK
  • latency는 정상 범위
  • timeout 없음
  • 서버 리소스 사용량 정상
  • 하지만 답변은 부정확함

이런 문제를 잡기 위해서는 일반적인 모니터링뿐 아니라 RAG 특화 품질 평가가 필요하다.

RAG에서는 다음 질문을 계속 확인해야 한다.

  • 검색 결과가 질문과 관련 있는가
  • 정답 근거 문서가 검색되었는가
  • 답변이 검색된 문서에 근거하고 있는가
  • 답변이 실제 정답과 의미적으로 일치하는가
  • 문서에 없는 내용을 지어내지 않았는가
  • 이전 버전보다 품질이 떨어지지 않았는가

결국 RAG 품질 회귀 탐지는 시스템이 살아 있는지를 보는 것이 아니라, 시스템이 여전히 좋은 답변을 하고 있는지를 보는 문제이다.

RAG 품질 회귀를 탐지하는 방법

RAG 품질 회귀를 탐지하려면 기준 평가셋이 필요하다. 평가셋은 RAG 시스템이 반드시 잘 답변해야 하는 질문과 기대 답변, 정답 근거 문서로 구성된다.

예를 들어 평가 데이터는 다음과 같이 만들 수 있다.

question: 사용자의 질문
reference_answer: 기대 답변
gold_documents: 정답 근거 문서
metadata: 도메인, 난이도, 질문 유형

이 평가셋을 이용해 기존 버전과 새 버전의 RAG 시스템을 비교한다. 새 버전에서 검색 품질이나 생성 품질이 기준 이하로 떨어지면 품질 회귀가 발생했다고 판단할 수 있다.

검색 품질 평가

RAG에서 검색은 매우 중요하다. 검색 단계에서 올바른 문서를 찾지 못하면 LLM이 좋은 답변을 만들기 어렵다.

검색 품질을 평가할 때는 다음과 같은 지표를 사용할 수 있다.

Recall@K

Recall@K는 정답 근거 문서가 상위 K개의 검색 결과 안에 포함되었는지를 보는 지표이다.

예를 들어 Recall@5는 정답 문서가 검색 결과 상위 5개 안에 들어왔는지를 확인한다. RAG에서는 보통 LLM에게 상위 몇 개의 문서만 전달하므로, Recall@K는 매우 중요한 지표이다.

정답 문서가 검색되지 않으면 LLM은 애초에 올바른 근거를 받지 못한다. 이 경우 생성 모델이 아무리 좋아도 정답을 만들기 어렵다.

Precision@K

Precision@K는 상위 K개의 검색 결과 중 실제로 질문과 관련 있는 문서의 비율을 보는 지표이다.

관련 없는 문서가 많이 섞이면 LLM이 혼란스러운 문맥을 받게 된다. 이 경우 답변이 장황해지거나, 서로 다른 문서 내용을 섞어 잘못된 답변을 만들 수 있다.

MRR

MRR은 정답 문서가 검색 결과에서 얼마나 높은 순위에 등장하는지를 평가하는 지표이다. 정답 문서가 1등으로 검색되는 것과 5등으로 검색되는 것은 실제 답변 품질에서 차이를 만들 수 있다.

LLM에게 전달되는 문서 수가 제한되어 있기 때문에, 정답 문서가 가능한 한 높은 순위에 위치하는 것이 중요하다.

생성 품질 평가

검색 결과가 좋아도 LLM이 답변을 잘못 생성할 수 있다. 따라서 생성 단계의 품질도 별도로 평가해야 한다.

생성 품질에서는 다음 항목을 볼 수 있다.

정답성

정답성은 생성된 답변이 기대 답변과 의미적으로 일치하는지를 보는 기준이다. 단순히 문자열이 같은지를 보는 것이 아니라, 답변의 의미가 맞는지를 평가해야 한다.

예를 들어 표현은 다르지만 핵심 내용이 같다면 정답으로 볼 수 있다. 반대로 문장은 자연스럽지만 중요한 조건이 빠져 있다면 오답에 가깝다.

근거성

근거성은 답변이 검색된 문서에 기반하고 있는지를 보는 기준이다. RAG에서 중요한 것은 답변이 그럴듯한가가 아니라, 제공된 문서에 근거하고 있는가이다.

답변이 맞아 보이더라도 문서에 없는 내용을 포함하고 있다면 운영 환경에서는 위험할 수 있다.

환각 여부

환각은 LLM이 문서에 없는 내용을 만들어내는 현상이다. RAG 시스템에서는 사용자가 답변이 문서에 기반한다고 믿기 때문에, 환각이 발생하면 신뢰 문제가 커질 수 있다.

특히 금융, 법률, 의료, 기업 내부 문서 검색처럼 정확성이 중요한 도메인에서는 환각을 강하게 관리해야 한다.

형식 준수

운영형 RAG에서는 답변 형식도 중요하다. 고객 지원 챗봇이라면 답변이 너무 장황하면 안 되고, 내부 문서 검색 시스템이라면 근거 문서나 조항을 명확히 제시해야 한다.

따라서 품질 평가는 단순히 맞고 틀림뿐 아니라 서비스 목적에 맞는 답변 형식까지 포함해야 한다.

품질 회귀 탐지와 MLOps 파이프라인

RAG 품질 회귀 탐지는 단순히 평가 리포트를 보는 데서 끝나면 안 된다. 운영형 시스템에서는 평가 결과가 배포 정책과 연결되어야 한다.

예를 들어 다음과 같은 흐름을 만들 수 있다.

배포 승인 또는 배포 차단

이때 중요한 개념이 quality gate이다. quality gate는 배포 전에 품질 기준을 통과했는지 확인하는 장치이다.

예를 들어 다음과 같은 기준을 둘 수 있다.

Recall@5가 기존 대비 3% 이상 하락하면 배포 차단
정답성 점수가 기준 이하이면 배포 차단
환각률이 증가하면 배포 차단
중요 카테고리 질문에서 실패하면 배포 차단

이렇게 하면 RAG 시스템의 변경 사항이 운영 환경에 반영되기 전에 품질 저하를 자동으로 감지할 수 있다.

재색인 전략과 품질 회귀

RAG 시스템에서는 문서가 계속 바뀐다. 새 문서가 추가되고, 기존 문서가 수정되며, 오래된 문서가 삭제된다. 이때 벡터 DB를 어떻게 재색인할지가 중요하다.

모든 문서를 매번 재색인하면 품질 일관성은 높아질 수 있지만 비용이 크다. 반대로 재색인을 너무 적게 하면 최신 문서가 검색되지 않거나, 오래된 벡터가 남아 검색 품질이 떨어질 수 있다.

대표적인 재색인 방식은 다음과 같다.

전체 재색인

전체 재색인은 모든 문서를 다시 임베딩하고 벡터 DB에 저장하는 방식이다. 가장 일관성은 높지만 비용이 크다.

임베딩 모델을 교체했거나, 청킹 전략을 크게 변경했거나, 기존 색인 상태를 신뢰하기 어려울 때는 전체 재색인이 필요할 수 있다.

증분 재색인

증분 재색인은 변경된 문서만 다시 색인하는 방식이다. 운영 비용을 줄일 수 있지만, 변경 감지 로직이 정확해야 한다.

문서가 수정되었는데 변경 감지가 실패하면 오래된 벡터가 계속 사용될 수 있다. 반대로 변경되지 않은 문서를 불필요하게 자주 재색인하면 비용이 증가한다.

조건 기반 재색인

조건 기반 재색인은 문서 변경량, 검색 품질, 실패율, 비용 등을 기준으로 재색인 여부를 결정하는 방식이다.

예를 들어 문서 변경량이 적고 품질 지표가 안정적이라면 재색인을 미룰 수 있다. 반대로 특정 도메인에서 Recall@K가 하락하거나 검색 실패율이 증가하면 해당 영역만 재색인할 수 있다.

운영형 RAG에서는 조건 기반 재색인이 비용과 품질의 균형을 맞추는 데 유리하다.

운영형 RAG에서 봐야 할 지표

RAG 시스템을 운영할 때는 일반적인 시스템 지표와 RAG 품질 지표를 함께 봐야 한다.

시스템 지표는 다음과 같다.

latency
p95 latency
error rate
timeout
retry count
token usage
cost

이 지표들은 시스템이 안정적으로 동작하는지 확인하는 데 필요하다.

검색 지표는 다음과 같다.

Recall@K
Precision@K
MRR
검색 결과 없음 비율
평균 검색 점수
reranking 전후 순위 변화

검색 단계가 정상적으로 작동하는지 확인하는 데 사용된다.

생성 지표는 다음과 같다.

정답성
근거성
환각률
답변 완성도
형식 준수율

LLM이 검색 결과를 바탕으로 제대로 답변하는지 확인하는 데 사용된다.

운영 지표는 다음과 같다.

문서 변경량
재색인 소요 시간
재색인 비용
배포 차단 횟수
품질 게이트 실패율
사용자 피드백 비율

운영형 RAG에서는 이 지표들이 중요하다. RAG는 한 번 만들고 끝나는 시스템이 아니라, 문서와 모델과 사용자 질문이 계속 바뀌는 시스템이기 때문이다.

품질 회귀가 발생했을 때 확인할 것

품질 회귀가 감지되면 먼저 문제가 검색 단계인지 생성 단계인지 분리해서 봐야 한다.

정답 문서가 검색 결과에 포함되지 않았다면 검색 단계 문제일 가능성이 높다.

이 경우 다음을 확인해야 한다.

청킹 방식이 바뀌었는가
임베딩 모델이 바뀌었는가
벡터 DB 재색인이 정상적으로 되었는가
문서 전처리 과정에서 누락이 있었는가
검색 파라미터가 변경되었는가

반대로 정답 문서가 검색되었는데도 답변이 틀렸다면 생성 단계 문제일 가능성이 높다.

이 경우 다음을 확인해야 한다.

프롬프트가 변경되었는가
LLM 모델이 변경되었는가
context 길이가 부족하지 않은가
문서 순서가 답변에 영향을 주지 않았는가
답변 형식 제약이 과도하지 않은가

특정 문서군이나 특정 도메인에서만 품질이 떨어진다면 데이터 문제일 수도 있다.

예를 들어 문서가 오래되었거나, 중복 문서가 많거나, 서로 충돌하는 내용이 있을 수 있다. RAG에서는 문서 품질 관리도 모델 품질 관리만큼 중요하다.

정리

RAG 품질 회귀는 단순한 버그가 아니다. API는 정상적으로 응답하고 서버도 안정적으로 동작하지만, 사용자가 받는 답변 품질은 나빠질 수 있다.

따라서 운영형 RAG에서는 다음과 같은 체계가 필요하다.

기준 평가셋 구축
검색 품질 평가
생성 품질 평가
기존 버전과 비교
quality gate 적용
배포 자동 차단
조건 기반 재색인
운영 지표 모니터링

RAG 시스템은 문서, 청킹, 임베딩, 검색, 프롬프트, LLM, 배포 환경이 함께 만드는 시스템이다. 그래서 RAG 품질을 안정적으로 유지하려면 모델을 잘 쓰는 것만으로는 부족하다.

변경이 발생할 때마다 품질이 유지되는지 검증하고, 문제가 생기면 배포를 막고, 필요한 경우 재색인까지 연결하는 운영 체계가 필요하다.

결국 RAG 품질 회귀 관리는 RAG를 실험용 데모에서 실제 서비스로 옮기기 위한 핵심 MLOps 문제이다.
