# 실무에서는 런타임과 메모리 중 뭐가 더 중요할까?

알고리즘 문제를 풀다 보면 자주 보게 되는 것:

```text
Runtime
Memory
```

리트코드에서는 퍼센트까지 보여준다.

```text
Runtime: 77ms
Memory: 28MB
```

그러다 보면 자연스럽게 이런 생각이 든다.

> “실무에서는 런타임이 더 중요할까?”
>
> “아니면 메모리가 더 중요할까?”

결론부터 말하면:

```text
실무에서는 병목이 되는 쪽이 더 중요하다.
```

그리고 대부분은 상황에 따라 달라진다.

---

# 1. 런타임이 중요한 경우

웹 서비스나 API 서버에서는 보통 응답 속도가 굉장히 중요하다.

예를 들어:

```text
로그인 API → 5초
상품 조회 → 7초
검색 API → 10초
```

사용자는 바로 느낀다.

서비스가 느리면:

* 사용자 이탈
* UX 악화
* 트래픽 감소
* 서버 대기 증가

등이 발생한다.

그래서 실무에서는 흔히:

```text
Latency(지연시간)
Response Time(응답속도)
TPS/RPS(처리량)
```

를 매우 중요하게 본다.

---

# 예시

## O(n²) 때문에 느린 경우

```python
for i in arr:
    for j in arr:
        ...
```

데이터가 커질수록 급격히 느려진다.

실무에서는:

* 해시 사용
* 캐싱
* 인덱스 최적화
* 병렬 처리

등으로 속도를 줄인다.

---

# 2. 메모리가 중요한 경우

반대로 메모리가 더 중요한 상황도 많다.

특히:

* Docker
* Kubernetes
* 대용량 데이터 처리
* AI/ML
* 배치 시스템

같은 환경에서는 메모리가 진짜 중요하다.

예를 들어:

```text
컨테이너 메모리 제한 512MB
```

인데 프로그램이 2GB를 먹는다?

그러면:

```text
OOM (Out Of Memory)
```

가 발생한다.

쿠버네티스에서는:

```text
OOMKilled
```

상태가 뜨면서 Pod가 죽는다.

---

# 실무에서 자주 보는 상황

## 1. 느리지만 안 죽는 서비스

```text
응답시간 3~5초
```

이 경우:

```text
런타임 최적화
```

를 먼저 한다.

예시:

* 쿼리 최적화
* 캐싱
* 알고리즘 개선
* 비동기 처리

---

## 2. 빠르지만 메모리를 너무 먹는 서비스

```text
서버 메모리 95%
```

이 경우:

```text
메모리 최적화
```

가 더 중요하다.

예시:

* 불필요한 객체 제거
* 스트리밍 처리
* chunk 단위 처리
* GC 튜닝
* 캐시 크기 조절

---

# 결국 중요한 것

실무에서는 단순히:

```text
몇 ms 빠르다
```

보다 더 중요한 게 있다.

바로:

```text
실제 장애를 만드는가?
```

이다.

예를 들어:

## 이건 위험하다

```text
메모리 부족으로 서버 죽음
```

## 이것도 위험하다

```text
API 응답 15초
```

---

# 코테와 실무의 차이

코테에서는:

```text
시간복잡도
공간복잡도
```

를 주로 본다.

예시:

```text
O(n²) → 위험
O(n log n) → 가능
O(n) → 좋음
```

하지만 실무에서는:

```text
CPU
메모리
네트워크
디스크 I/O
DB
캐시
동시성
```

전부 함께 본다.

즉:

```text
알고리즘만 빠르다고 끝이 아니다.
```

# 실무에서 많이 하는 판단

보통 이런 순서로 본다.

```text
1. 서비스가 죽는가?
2. 사용자가 느린 걸 체감하는가?
3. 서버 비용이 많이 드는가?
4. 유지보수가 어려운가?
```

그리고 마지막에는 결국:

```text
읽기 쉽고 유지보수 가능한 코드
```

가 굉장히 중요해진다.

# 결론

실무에서는:

```text
“더 중요한 하나”가 있는 게 아니라
현재 병목이 무엇인지 찾는 능력이 중요하다.
```

그리고 대부분의 좋은 엔지니어는:

```text
시간복잡도 + 메모리 + 운영 환경
```

을 같이 본다.


투포인터는 한마디로 말하면 **메모리에 강한 편이고, 런타임도 같이 좋아지는 경우가 많다.**

정리하면 이렇게 묶으면 돼.

---

# 투포인터는 런타임과 메모리 중 어디에 강할까?

투포인터는 보통 **배열이나 문자열을 한 번 훑으면서** 문제를 푸는 방식이다.

```text
left  → 왼쪽 포인터
right → 오른쪽 포인터
```

두 개의 인덱스를 움직이면서 필요한 구간만 확인한다.

그래서 투포인터의 장점은 보통 두 가지다.

```text
1. 불필요한 반복을 줄여서 런타임 개선
2. 새 배열이나 딕셔너리를 덜 만들어서 메모리 절약
```

---

# 1. 런타임 측면

투포인터는 보통 `O(n²)`으로 풀 수 있는 문제를 `O(n)`이나 `O(n log n)` 정도로 줄이는 데 자주 쓰인다.

예를 들어 모든 쌍을 비교하면:

```python
for i in range(len(arr)):
    for j in range(i + 1, len(arr)):
        ...
```

시간복잡도는:

```text
O(n²)
```

하지만 투포인터를 쓰면 양쪽에서 포인터를 움직이면서 한 번만 훑을 수 있다.

```python
left = 0
right = len(arr) - 1

while left < right:
    ...
```

이 경우 보통:

```text
O(n)
```

이 된다.

즉, 투포인터는 **런타임 개선에도 강하다.**

---

# 2. 메모리 측면

투포인터의 더 큰 장점은 **추가 자료구조를 거의 안 쓴다는 것**이다.

보통 필요한 건 인덱스 변수 두 개뿐이다.

```python
left = 0
right = 1
```

그래서 공간복잡도는 보통:

```text
O(1)
```

이다.

새로운 리스트를 만들거나, 딕셔너리에 값을 저장하거나, 배열을 복사하지 않아도 되는 경우가 많다.

즉, 투포인터는 **메모리에도 강하다.**

---

# 그럼 둘 중 어디에 더 강한가?

굳이 하나를 고르면:

```text
투포인터는 메모리 절약에 특히 강하고,
동시에 런타임도 좋아지는 경우가 많다.
```

왜냐하면 투포인터의 기본 철학이 이거라서다.

```text
새 자료구조를 만들지 말고,
원래 배열 위에서 인덱스만 움직이자.
```

그래서 메모리 사용량은 거의 늘어나지 않는다.

---

# 예시: Best Time to Buy and Sell Stock

```python
class Solution:
    def maxProfit(self, prices: List[int]) -> int:
        left = 0
        right = 1
        ans = 0

        while right < len(prices):
            if prices[left] < prices[right]:
                ans = max(ans, prices[right] - prices[left])
            else:
                left = right

            right += 1

        return ans
```

이 코드는 배열을 한 번만 훑는다.

```text
시간복잡도: O(n)
공간복잡도: O(1)
```

즉:

```text
런타임도 좋고
메모리도 거의 안 쓴다
```

---

# 정리

투포인터는 이렇게 이해하면 된다.

```text
런타임 측면:
불필요한 이중 반복을 줄여서 빠르게 만든다.

메모리 측면:
새 자료구조를 거의 만들지 않고 인덱스만 움직인다.

결론:
투포인터는 메모리에 특히 강하고,
많은 경우 런타임까지 같이 좋아진다.
```
