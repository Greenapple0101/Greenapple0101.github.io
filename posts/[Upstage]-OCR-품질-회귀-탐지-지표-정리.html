[Upstage] OCR 품질 회귀 탐지 지표 정리

2026.05.30

OCR 시스템에서 품질 회귀를 탐지하려면 단순히 API가 정상 응답하는지만 보면 안 된다. OCR은 서버가 정상적으로 동작해도 인식 결과가 틀릴 수 있기 때문이다.

예를 들어 API 응답 코드는 200 OK이고 latency도 정상인데, 실제 OCR 결과가 다음처럼 틀릴 수 있다.

원본: 125,000원
OCR 결과: 12,500원

이 경우 시스템 장애는 아니지만 서비스 품질은 명확히 떨어진 것이다. 따라서 OCR 품질 회귀 탐지에서는 일반적인 시스템 지표와 OCR 결과 품질 지표를 함께 봐야 한다.

OCR 품질 회귀 탐지 지표는 크게 다음과 같이 나눌 수 있다.

1. 문자·단어 인식 정확도 지표
2. 필드 추출 정확도 지표
3. 문서 구조 인식 지표
4. 모델 신뢰도 지표
5. 운영 안정성 지표
6. 데이터 분포 변화 지표

1. CER

CER은 Character Error Rate의 약자로, 문자 단위 오류율을 의미한다.

정답 텍스트와 OCR 결과를 문자 단위로 비교해서 삽입, 삭제, 치환 오류가 얼마나 발생했는지 계산한다.

CER = (삽입 오류 수 + 삭제 오류 수 + 치환 오류 수) / 정답 문자 수

예를 들어 정답과 OCR 결과가 다음과 같다고 하자.

정답: 백서연
OCR 결과: 백서역

마지막 한 글자가 잘못 인식되었으므로 문자 단위 치환 오류가 발생한 것이다.

CER은 OCR 품질 평가에서 가장 기본적인 지표 중 하나다. 특히 한국어, 숫자, 영문, 특수문자가 섞인 문서에서 문자 단위 오류를 확인하는 데 유용하다.

다만 CER만으로는 업무상 중요한 오류를 구분하기 어렵다. 예를 들어 일반 문장의 조사 하나가 틀린 것과 금액의 숫자 하나가 틀린 것을 같은 문자 오류로 계산하기 때문이다.

일반 문장 오류: 신청서를 제출합니디
금액 오류: 125,000원 → 12,500원

둘 다 문자 오류이지만 실제 업무 영향도는 완전히 다르다. 따라서 CER은 기본 지표로 사용하되, 필드 단위 정확도와 함께 봐야 한다.

2. WER

WER은 Word Error Rate의 약자로, 단어 단위 오류율을 의미한다.

정답 텍스트와 OCR 결과를 단어 단위로 비교해서 오류율을 계산한다.

WER = (삽입 오류 수 + 삭제 오류 수 + 치환 오류 수) / 정답 단어 수

예를 들어 다음과 같은 경우를 볼 수 있다.

정답: 계약 금액은 125,000원입니다
OCR 결과: 계약 금액은 12,500원입니다

이 경우 단어 단위로 보면 125,000원입니다가 12,500원입니다로 잘못 인식된 것이다.

WER은 영어처럼 단어 경계가 명확한 언어에서는 유용하다. 하지만 한국어 문서에서는 띄어쓰기 차이에 민감할 수 있다.

예를 들어 다음 두 문장은 의미상 거의 같지만 WER에서는 다르게 평가될 수 있다.

정답: 신청일은 2026년 5월 30일입니다
OCR 결과: 신청 일은 2026년 5월 30일 입니다

따라서 한국어 OCR에서는 WER만 단독으로 사용하기보다 CER과 함께 보는 것이 좋다.

3. Exact Match Accuracy

Exact Match Accuracy는 OCR 결과가 정답과 완전히 일치하는 비율을 의미한다.

예를 들어 특정 필드의 정답과 OCR 결과가 완전히 같으면 성공, 하나라도 다르면 실패로 계산한다.

정답: 2026-05-30
OCR 결과: 2026-05-30
결과: 성공

정답: 2026-05-30
OCR 결과: 2026-05-3O
결과: 실패

Exact Match는 주민등록번호, 사업자등록번호, 계좌번호, 날짜, 금액처럼 정확히 일치해야 하는 필드에서 중요하다.

하지만 긴 문장 전체에 Exact Match를 적용하면 너무 엄격할 수 있다. 띄어쓰기나 문장부호 하나만 달라도 실패로 처리되기 때문이다.

따라서 Exact Match는 전체 문서보다는 중요한 필드 평가에 적합하다.

4. Field Accuracy

Field Accuracy는 문서에서 추출해야 하는 주요 필드가 정확히 인식되었는지를 보는 지표이다.

문서 AI나 OCR 기반 자동 입력 서비스에서는 전체 텍스트보다 특정 필드가 더 중요하다.

예를 들어 세금계산서 OCR에서는 다음 필드가 중요할 수 있다.

공급자 사업자등록번호
공급받는자 사업자등록번호
작성일자
공급가액
부가세
합계금액

각 필드가 정답과 일치하는지 확인하고, 전체 필드 중 정확히 추출된 비율을 계산한다.

Field Accuracy = 정확히 추출된 필드 수 / 전체 필드 수

예를 들어 10개 필드 중 9개를 정확히 추출했다면 Field Accuracy는 90%이다.

Field Accuracy는 OCR 품질 회귀 탐지에서 매우 중요하다. 고객 입장에서는 OCR이 문서 전체를 얼마나 잘 읽었는지보다, 실제 업무에 필요한 필드를 정확히 뽑았는지가 더 중요할 수 있기 때문이다.

5. Critical Field Accuracy

Critical Field Accuracy는 업무상 중요한 필드만 따로 평가하는 지표이다.

모든 필드가 같은 중요도를 가지는 것은 아니다. 예를 들어 계약서에서 일반 설명 문장의 일부 오류보다 계약 금액, 계약 기간, 이름, 계좌번호 오류가 훨씬 치명적이다.

중요 필드는 다음과 같이 설정할 수 있다.

금액
날짜
이름
계좌번호
주민등록번호
사업자등록번호
계약 기간
청구 금액
납부 기한

Critical Field Accuracy는 이런 핵심 필드의 정확도만 별도로 계산한다.

Critical Field Accuracy = 정확히 추출된 중요 필드 수 / 전체 중요 필드 수

품질 게이트를 만들 때는 전체 정확도보다 Critical Field Accuracy를 더 엄격하게 보는 것이 좋다.

예를 들어 전체 CER은 조금 나빠져도 허용할 수 있지만, 금액 필드 정확도가 떨어지면 배포를 차단해야 한다.

전체 CER: 1.2% → 1.4%
금액 필드 정확도: 99.1% → 93.0%

이 경우 전체 문자 오류율은 크게 변하지 않았지만, 실제 업무 품질은 크게 떨어진 것이다.

6. Field Missing Rate

Field Missing Rate는 반드시 추출해야 하는 필드가 누락된 비율을 의미한다.

OCR 시스템에서는 필드값을 잘못 읽는 것뿐 아니라, 아예 필드를 찾지 못하는 문제도 중요하다.

예를 들어 계약서에서 다음 필드가 반드시 필요하다고 하자.

계약자명
계약일
계약 금액
계약 기간
서명 여부

OCR 결과에서 계약 금액이 빠졌다면 이는 필드 누락이다.

Field Missing Rate = 누락된 필드 수 / 전체 필수 필드 수

Field Missing Rate가 증가하면 레이아웃 분석, 필드 탐지, 후처리 로직에 문제가 생겼을 가능성이 있다.

특히 문서 양식이 바뀌었거나, 전처리 과정에서 특정 영역이 잘렸을 때 필드 누락이 증가할 수 있다.

7. Field Mismatch Rate

Field Mismatch Rate는 필드명과 필드값이 잘못 연결된 비율을 의미한다.

예를 들어 문서에는 다음과 같이 적혀 있다고 하자.

이름: 백서연
날짜: 2026-05-30
금액: 125,000원

OCR 결과가 다음과 같이 나왔다면 문자 자체는 읽었지만 필드 연결이 틀린 것이다.

{
  "이름": "2026-05-30",
  "날짜": "125,000원",
  "금액": "백서연"
}

이런 문제는 단순 CER이나 WER로는 잡기 어렵다. 텍스트는 맞게 읽혔지만 구조화 결과가 틀렸기 때문이다.

Field Mismatch Rate는 신청서, 계약서, 세금계산서처럼 key-value 구조가 중요한 문서에서 반드시 봐야 하는 지표이다.

8. Key-Value Matching Accuracy

Key-Value Matching Accuracy는 문서의 항목명과 값이 올바르게 연결되었는지 평가하는 지표이다.

예를 들어 다음과 같은 문서가 있다.

성명: 백서연
신청일: 2026-05-30
금액: 125,000원

OCR 또는 Document AI 시스템은 단순히 텍스트를 읽는 것에서 끝나지 않고 다음처럼 구조화해야 한다.

{
  "성명": "백서연",
  "신청일": "2026-05-30",
  "금액": "125,000원"
}

Key-Value Matching Accuracy는 이 연결이 맞는지를 본다.

문서 자동화 업무에서는 이 지표가 매우 중요하다. 실제 서비스에서는 OCR 결과를 사람이 읽는 것이 아니라, 후속 시스템에 자동 입력하는 경우가 많기 때문이다.

9. Table Cell Accuracy

Table Cell Accuracy는 표 안의 각 셀 값이 정확히 인식되었는지를 보는 지표이다.

예를 들어 세금계산서, 견적서, 거래명세서, 청구서에는 표가 자주 등장한다.

품목      수량    단가       금액
노트북    1      1,200,000  1,200,000
마우스    2      25,000     50,000

이때 OCR은 단순히 글자를 읽는 것뿐 아니라 각 값이 어느 셀에 속하는지도 알아야 한다.

Table Cell Accuracy는 전체 셀 중 정확히 인식된 셀의 비율로 계산할 수 있다.

Table Cell Accuracy = 정확히 인식된 셀 수 / 전체 셀 수

표 기반 문서에서는 이 지표가 중요하다. 숫자 하나가 옆 열로 밀리거나, 행이 합쳐지면 후속 정산 시스템에서 사용할 수 없는 데이터가 되기 때문이다.

10. Table Structure Accuracy

Table Structure Accuracy는 표의 행과 열 구조를 올바르게 인식했는지 평가하는 지표이다.

문자 인식이 맞아도 표 구조가 깨지면 문제가 된다.

예를 들어 원래는 다음과 같은 표였다고 하자.

품목      수량    금액
노트북    1      1,200,000
마우스    2      50,000

그런데 OCR 결과가 다음처럼 나오면 구조가 깨진 것이다.

품목: 노트북, 마우스
수량: 1, 2
금액: 1,200,000, 50,000

글자는 읽었지만 각 행의 관계가 사라졌다.

Table Structure Accuracy는 행, 열, 셀 병합, 헤더, 본문 구조가 유지되었는지 평가하는 데 사용한다.

11. Layout Detection Accuracy

Layout Detection Accuracy는 문서의 영역을 올바르게 탐지했는지 보는 지표이다.

문서에는 여러 영역이 있다.

제목
본문
표
이미지
서명란
도장
체크박스
머리말
꼬리말
각주

OCR 시스템이 각 영역을 올바르게 구분하지 못하면 텍스트 순서가 꼬이거나, 표가 일반 문장처럼 처리되거나, 서명란과 본문이 섞일 수 있다.

Layout Detection Accuracy는 특히 Document AI에서 중요하다. 단순 텍스트 추출보다 문서 구조 이해가 필요한 서비스에서는 레이아웃 인식 품질이 최종 결과에 큰 영향을 주기 때문이다.

12. Reading Order Accuracy

Reading Order Accuracy는 OCR 결과의 읽기 순서가 원본 문서와 맞는지를 평가하는 지표이다.

문서에 여러 단, 표, 주석, 머리말, 꼬리말이 있는 경우 OCR 결과의 순서가 뒤섞일 수 있다.

예를 들어 원본 문서는 다음 순서로 읽어야 한다.

제목
본문 1
본문 2
표
각주

그런데 OCR 결과가 다음 순서로 나올 수 있다.

제목
각주
표
본문 1
본문 2

이 경우 텍스트 자체는 대부분 맞더라도 LLM이나 후속 시스템이 문서를 잘못 이해할 수 있다.

OCR 결과가 RAG의 입력으로 사용된다면 Reading Order Accuracy가 특히 중요하다. 읽기 순서가 잘못되면 청킹 결과가 이상해지고, 검색과 답변 품질까지 떨어질 수 있다.

13. Confidence Score

Confidence Score는 OCR 모델이 자신의 인식 결과를 얼마나 확신하는지 나타내는 값이다.

예를 들어 OCR 결과가 다음과 같이 나올 수 있다.

{
  "text": "125,000원",
  "confidence": 0.98
}

confidence가 높으면 모델이 해당 인식 결과를 비교적 확신한다는 의미이고, 낮으면 불확실하다는 의미이다.

운영 중 평균 confidence score가 갑자기 낮아지면 품질 회귀나 입력 데이터 변화 가능성을 의심할 수 있다.

예를 들어 다음과 같은 변화가 있을 수 있다.

평균 confidence: 0.94 → 0.81

API 에러율은 그대로라도 confidence가 낮아졌다면 저해상도 이미지가 늘었거나, 새 문서 양식에서 모델이 어려움을 겪고 있을 수 있다.

다만 confidence score는 실제 정확도와 항상 완벽히 일치하지 않는다. 모델이 틀린 답을 높은 confidence로 내는 경우도 있기 때문에, 정답 기반 평가 지표와 함께 사용해야 한다.

14. Confidence Calibration Error

Confidence Calibration Error는 모델의 confidence score가 실제 정답률과 얼마나 잘 맞는지 보는 지표이다.

예를 들어 모델이 confidence 0.9라고 예측한 결과들이 실제로도 약 90% 정도 맞는다면 calibration이 잘 되어 있다고 볼 수 있다.

하지만 confidence 0.9라고 했는데 실제 정답률이 70%라면 문제가 있다. 모델이 자기 결과를 과신하고 있는 것이다.

OCR 운영에서는 confidence score를 기준으로 수동 검수 여부를 결정할 수 있다.

confidence 0.95 이상: 자동 처리
confidence 0.80 ~ 0.95: 추가 검증
confidence 0.80 미만: 수동 검수

이때 confidence가 실제 정확도와 잘 맞지 않으면 자동 처리 기준이 위험해진다.

따라서 OCR 품질 회귀 탐지에서는 confidence 자체의 평균뿐 아니라 confidence와 실제 정확도의 관계도 확인하는 것이 좋다.

15. Correction Rate

Correction Rate는 OCR 결과가 후처리 단계에서 얼마나 많이 수정되었는지를 보는 지표이다.

OCR 시스템은 인식 결과를 그대로 반환하지 않고, 후처리 로직으로 보정하는 경우가 많다.

예를 들어 다음과 같은 보정이 있을 수 있다.

O → 0
I → 1
공백 제거
날짜 형식 변환
금액 쉼표 정규화
전화번호 형식 보정

Correction Rate가 갑자기 증가하면 OCR 원시 결과 품질이 나빠졌거나, 후처리 로직이 과도하게 개입하고 있을 수 있다.

Correction Rate: 4.2% → 15.8%

이 경우 모델 변경, 입력 데이터 변화, 후처리 규칙 변경을 확인해야 한다.

후처리는 품질을 개선할 수도 있지만 잘못 적용되면 오히려 정답을 망칠 수 있다. 따라서 후처리 전 결과와 후처리 후 결과를 모두 저장하고 비교하는 것이 좋다.

16. Rejection Rate

Rejection Rate는 OCR 시스템이 자동 처리를 포기하고 실패 또는 수동 검수로 넘긴 비율이다.

예를 들어 confidence가 너무 낮거나 필수 필드가 누락되면 자동 처리를 하지 않고 검수 대상으로 보낼 수 있다.

Rejection Rate = 수동 검수 또는 실패 처리 건수 / 전체 요청 수

Rejection Rate가 증가하면 OCR 시스템이 어려운 입력을 더 많이 만나고 있거나, 모델 품질이 낮아졌거나, 기준이 너무 엄격해졌을 수 있다.

운영 관점에서는 Rejection Rate가 너무 낮아도 문제일 수 있다. 품질이 낮은 결과를 무리하게 자동 처리하고 있을 가능성이 있기 때문이다.

따라서 Rejection Rate는 Field Accuracy, Confidence Score와 함께 봐야 한다.

17. Human Correction Rate

Human Correction Rate는 사람이 OCR 결과를 수정한 비율이다.

실제 운영에서는 OCR 결과를 사람이 검수하거나 수정하는 워크플로우가 있을 수 있다. 이때 사용자 또는 검수자가 얼마나 자주 OCR 결과를 고치는지 기록하면 중요한 품질 지표가 된다.

Human Correction Rate = 사람이 수정한 필드 수 / 전체 OCR 필드 수

Human Correction Rate가 증가하면 실제 사용자가 느끼는 OCR 품질이 낮아졌다는 신호일 수 있다.

이 지표는 정답 데이터가 없는 운영 환경에서도 품질 변화를 감지하는 데 유용하다.

18. Document Type Accuracy

Document Type Accuracy는 입력 문서의 유형을 올바르게 분류했는지 보는 지표이다.

OCR 서비스는 문서 유형에 따라 다른 처리 파이프라인을 사용할 수 있다.

예를 들어 다음과 같이 분기할 수 있다.

영수증 → 영수증 OCR 파이프라인
세금계산서 → 세금계산서 OCR 파이프라인
신분증 → 신분증 OCR 파이프라인
계약서 → 계약서 OCR 파이프라인

문서 유형 분류가 틀리면 뒤쪽 OCR 및 필드 추출도 연쇄적으로 틀릴 수 있다.

예를 들어 세금계산서를 영수증으로 분류하면 공급가액, 부가세, 사업자등록번호 같은 필드를 제대로 추출하지 못할 수 있다.

따라서 문서 유형 분류가 있는 OCR 시스템에서는 Document Type Accuracy도 품질 회귀 지표에 포함해야 한다.

19. Latency

Latency는 OCR 요청이 들어온 뒤 응답이 반환되기까지 걸리는 시간이다.

OCR은 이미지 처리, 모델 추론, 후처리, 구조화 과정이 포함되므로 일반 API보다 시간이 오래 걸릴 수 있다.

품질 회귀 탐지에서 latency도 중요하다. 모델을 바꾸거나 전처리를 추가하면서 정확도는 좋아졌지만 응답 시간이 크게 느려질 수 있기 때문이다.

예를 들어 다음과 같은 변화가 있을 수 있다.

Field Accuracy: 96.0% → 96.5%
p95 latency: 1.2초 → 4.8초

정확도는 조금 좋아졌지만 고객 서비스에서 허용하기 어려운 수준으로 느려졌다면 좋은 배포라고 보기 어렵다.

20. p95 Latency

p95 latency는 전체 요청 중 95%가 이 시간 안에 처리되었다는 의미이다.

평균 latency만 보면 일부 느린 요청을 놓칠 수 있다.

예를 들어 평균 latency는 안정적이어도 특정 문서 유형에서만 매우 느려질 수 있다.

평균 latency: 1.1초
p95 latency: 5.8초

이 경우 대부분은 빠르지만 일부 사용자는 매우 느린 응답을 경험한다.

OCR에서는 고해상도 이미지, 페이지 수가 많은 PDF, 표가 많은 문서에서 latency가 증가할 수 있다. 따라서 평균보다 p95, p99 latency를 함께 보는 것이 좋다.

21. Error Rate

Error Rate는 OCR 요청 중 실패한 요청의 비율이다.

Error Rate = 실패 요청 수 / 전체 요청 수

여기서 실패는 서버 오류, 파일 파싱 실패, 모델 추론 실패, timeout 등을 포함할 수 있다.

Error Rate는 OCR 품질 자체보다는 시스템 안정성 지표에 가깝다. 하지만 운영에서는 매우 중요하다.

예를 들어 품질 지표가 좋아도 error rate가 증가하면 배포를 막아야 한다.

CER: 개선
Field Accuracy: 개선
Error Rate: 0.2% → 3.5%

이 경우 모델 품질은 좋아졌더라도 서비스 안정성은 나빠진 것이다.

22. Timeout Rate

Timeout Rate는 제한 시간 안에 OCR 처리가 끝나지 않은 비율이다.

OCR은 문서 크기와 이미지 품질에 따라 처리 시간이 크게 달라질 수 있다.

Timeout Rate가 증가하면 다음 원인을 의심할 수 있다.

모델 추론 시간이 길어짐
전처리 로직이 무거워짐
고해상도 이미지 증가
대용량 PDF 증가
GPU 또는 CPU 리소스 부족
큐 적체 발생

Timeout은 사용자 입장에서 명확한 실패이므로, 정확도 지표와 별개로 관리해야 한다.

23. Throughput

Throughput은 단위 시간당 처리할 수 있는 OCR 요청 수를 의미한다.

예시:
초당 10건 처리
분당 300페이지 처리
시간당 10,000장 처리

AI DevOps 관점에서는 throughput이 중요하다. OCR 모델이 무거워지거나 전처리가 복잡해지면 동일한 인프라에서 처리량이 줄어들 수 있다.

품질은 개선되었지만 처리량이 크게 떨어지면 비용이 증가하거나 고객 요청을 감당하지 못할 수 있다.

24. Cost per Document

Cost per Document는 문서 한 건을 처리하는 데 드는 비용이다.

OCR 시스템에서는 GPU 사용량, CPU 사용량, 저장소 비용, 외부 API 비용 등이 비용에 영향을 준다.

Cost per Document = 전체 처리 비용 / 처리 문서 수

모델을 변경했을 때 정확도는 조금 좋아졌지만 비용이 크게 증가할 수 있다.

Field Accuracy: 96.0% → 96.4%
Cost per Document: 10원 → 35원

이 경우 운영 관점에서는 배포 여부를 신중하게 판단해야 한다.

AI DevOps에서는 품질, latency, 비용을 함께 봐야 한다.

25. Image Quality Distribution

Image Quality Distribution은 입력 이미지 품질의 분포를 의미한다.

OCR 품질은 입력 이미지에 크게 영향을 받는다. 운영 중 입력 데이터가 바뀌면 모델이나 코드가 그대로여도 품질이 떨어질 수 있다.

확인할 수 있는 항목은 다음과 같다.

해상도
밝기
대비
기울기
노이즈
흐림 정도
그림자
파일 크기
압축률
스캔본/촬영본 비율

예를 들어 최근 저해상도 휴대폰 촬영 이미지가 증가했다면 CER이나 Field Accuracy가 나빠질 수 있다.

이 경우는 모델 배포로 인한 회귀라기보다는 입력 데이터 분포 변화에 따른 품질 저하이다.

26. Document Type Distribution

Document Type Distribution은 운영 중 들어오는 문서 유형의 비율을 의미한다.

예를 들어 기존에는 영수증이 대부분이었는데, 최근 세금계산서나 계약서가 늘어날 수 있다.

영수증: 70% → 40%
세금계산서: 10% → 35%
계약서: 5% → 15%
기타: 15% → 10%

전체 OCR 정확도가 떨어졌을 때, 모델이 나빠진 것이 아니라 어려운 문서 유형이 늘어난 것일 수 있다.

따라서 OCR 품질 회귀를 분석할 때는 품질 지표와 함께 문서 유형 분포를 봐야 한다.

27. Customer-wise Accuracy

Customer-wise Accuracy는 고객사별 OCR 품질을 보는 지표이다.

B2B OCR 서비스에서는 고객사마다 문서 양식, 스캔 방식, 촬영 환경, 언어, 필드 구조가 다를 수 있다.

전체 평균 품질은 안정적이어도 특정 고객사에서만 품질이 떨어질 수 있다.

전체 Field Accuracy: 96.2% → 96.0%
A 고객사: 97.1% → 97.0%
B 고객사: 95.8% → 88.4%
C 고객사: 96.5% → 96.2%

이 경우 전체 평균만 보면 문제를 놓칠 수 있다. 하지만 B 고객사 입장에서는 심각한 품질 회귀이다.

따라서 운영형 OCR에서는 고객사별 품질 지표를 따로 보는 것이 중요하다.

28. Version-wise Quality Difference

Version-wise Quality Difference는 OCR 모델 또는 파이프라인 버전 간 품질 차이를 비교하는 지표이다.

품질 회귀 탐지의 핵심은 새 버전이 기존 버전보다 나빠졌는지 확인하는 것이다.

비교 대상은 다음과 같을 수 있다.

모델 버전
전처리 버전
후처리 버전
레이아웃 분석 버전
필드 추출 로직 버전
배포 환경 버전

예를 들어 다음과 같이 비교할 수 있다.

기존 버전 v1.2
새 버전 v1.3

CER: 1.8% → 2.4%
Field Accuracy: 96.5% → 94.1%
p95 latency: 1.3초 → 1.5초
Cost per Document: 12원 → 13원

이 경우 새 버전은 latency와 비용은 거의 비슷하지만 OCR 품질이 하락했으므로 품질 회귀가 발생했다고 볼 수 있다.

29. Quality Gate Pass Rate

Quality Gate Pass Rate는 OCR 평가 파이프라인에서 품질 기준을 통과한 비율이다.

배포 전에 기준 평가셋을 돌리고, 정해진 기준을 만족하면 통과, 아니면 실패로 처리한다.

예를 들어 quality gate 기준은 다음과 같이 둘 수 있다.

전체 CER이 기존 대비 2% 이상 악화되면 실패
Critical Field Accuracy가 98% 미만이면 실패
금액 필드 정확도가 기존 대비 1% 이상 하락하면 실패
p95 latency가 3초를 넘으면 실패
Timeout Rate가 1%를 넘으면 실패

Quality Gate Pass Rate가 낮아지면 최근 변경들이 운영 품질 기준을 자주 만족하지 못한다는 뜻이다.

AI DevOps 관점에서는 이 지표를 통해 배포 안정성을 관리할 수 있다.

30. Rollback Trigger Count

Rollback Trigger Count는 품질 문제로 인해 롤백이 발생한 횟수이다.

OCR 시스템에서는 배포 이후 특정 문서 유형이나 고객사에서 품질 저하가 발견되면 이전 버전으로 롤백해야 할 수 있다.

롤백 트리거는 다음과 같을 수 있다.

Critical Field Accuracy 급락
고객사별 오류율 증가
Timeout Rate 증가
p95 latency 초과
Human Correction Rate 증가
Confidence Score 급락

Rollback Trigger Count는 운영 안정성을 보는 데 유용하다. 롤백이 자주 발생한다면 배포 전 평가셋이나 quality gate가 충분하지 않을 수 있다.

지표를 어떻게 조합해서 봐야 하나

OCR 품질 회귀 탐지에서는 하나의 지표만 보면 안 된다. 각 지표는 서로 다른 문제를 보여준다.

예를 들어 다음과 같이 해석할 수 있다.

CER 증가
→ 문자 인식 품질 저하 가능성

Field Accuracy 하락
→ 실제 업무 필드 추출 품질 저하 가능성

Critical Field Accuracy 하락
→ 금액, 날짜, 번호 등 핵심 필드 품질 문제

Field Missing Rate 증가
→ 필드 탐지 또는 레이아웃 분석 문제

Field Mismatch Rate 증가
→ key-value 매칭 또는 구조화 문제

Table Structure Accuracy 하락
→ 표 분석 문제

Confidence Score 하락
→ 모델 불확실성 증가 또는 입력 데이터 품질 저하

Human Correction Rate 증가
→ 실제 사용자 체감 품질 저하

p95 latency 증가
→ 무거운 모델, 전처리 증가, 리소스 부족 가능성

Cost per Document 증가
→ 운영 비용 증가

Customer-wise Accuracy 하락
→ 특정 고객사 문서 양식 문제

즉, OCR 품질 회귀 탐지는 단순히 “정확도가 몇 퍼센트인가”를 보는 것이 아니라, 어느 단계에서 어떤 품질이 떨어졌는지 분리해서 보는 과정이다.

AI DevOps에서 중요한 OCR 지표 우선순위

AI DevOps 업무 관점에서는 모든 지표를 한 번에 다 볼 수는 없다. 우선순위를 정하면 다음과 같다.

1순위: Critical Field Accuracy
2순위: Field Accuracy
3순위: CER
4순위: Field Missing Rate
5순위: Table Structure Accuracy
6순위: Confidence Score
7순위: Human Correction Rate
8순위: p95 Latency
9순위: Error Rate / Timeout Rate
10순위: Cost per Document

왜냐하면 OCR 서비스는 결국 고객 업무에 필요한 값을 정확히 추출해야 하기 때문이다.

전체 텍스트가 조금 틀리는 것보다, 금액이나 사업자등록번호 같은 중요 필드가 틀리는 것이 더 치명적이다.

따라서 품질 게이트도 다음처럼 설정하는 것이 좋다.

Critical Field Accuracy 하락 시 배포 차단
Field Missing Rate 증가 시 배포 차단
중요 문서 유형 정확도 하락 시 배포 차단
p95 latency 기준 초과 시 배포 차단
Timeout Rate 기준 초과 시 배포 차단

특히 B2B 문서 AI 서비스에서는 전체 평균 정확도보다 문서 유형별, 고객사별, 필드별 품질 저하를 빠르게 탐지하는 것이 중요하다.

결국 OCR 품질 회귀 탐지 지표는 OCR 시스템이 고객 문서를 여전히 정확하고 안정적으로 처리하고 있는지를 확인하기 위한 운영 기준이다.
