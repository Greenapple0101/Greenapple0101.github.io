[WEB] 브라우저는 도대체 뭘까?

2026.05.12 Velog 원문

웹 개발을 처음 공부하면 이런 말을 자주 듣는다.

브라우저가 서버에 요청을 보낸다
브라우저가 HTML을 해석한다
브라우저에서 JavaScript가 실행된다
브라우저가 CORS를 막는다
브라우저가 쿠키를 저장한다

처음 들으면 헷갈린다.

“브라우저가 그냥 인터넷 창 아닌가?” 싶은데, 사실 브라우저는 단순한 창이 아니다.

브라우저는 웹 애플리케이션을 실행해주는 프로그램이다.

대표적인 브라우저는 다음과 같다.

Chrome
Safari
Edge
Firefox
Whale

우리가 웹사이트에 접속할 때 켜는 프로그램이 브라우저다.

1. 브라우저를 한 문장으로 정의하면

브라우저는 인터넷에 있는 웹 문서를 가져와서, 사람이 볼 수 있는 화면으로 바꿔주는 프로그램이다.

조금 더 개발자스럽게 말하면 다음과 같다.

브라우저 = HTTP 요청을 보내고, HTML/CSS/JavaScript를 해석해서 화면을 그리는 실행 환경

예를 들어 사용자가 Chrome 주소창에 다음 주소를 입력한다고 하자.

https://www.example.com

그러면 브라우저는 이 주소에 해당하는 서버를 찾아가서 웹 페이지에 필요한 파일들을 받아온다.

HTML
CSS
JavaScript
이미지
폰트
동영상

그리고 이 파일들을 조합해서 화면에 보여준다.

2. 브라우저는 그냥 화면이 아니다

많은 사람이 처음에는 브라우저를 “인터넷 보는 창” 정도로 생각한다.

틀린 말은 아니지만, 개발 관점에서는 조금 부족하다.

브라우저는 다음 일을 한다.

URL 해석
DNS 조회
HTTP 요청 전송
HTTP 응답 수신
HTML 파싱
CSS 파싱
JavaScript 실행
화면 렌더링
쿠키 저장
캐시 관리
보안 정책 적용
개발자 도구 제공

생각보다 하는 일이 많다.

즉 브라우저는 단순한 뷰어가 아니라, 웹 애플리케이션 실행 플랫폼에 가깝다.

3. 사용자가 URL을 입력하면 무슨 일이 일어날까?

사용자가 주소창에 URL을 입력한다.

https://example.com

그러면 브라우저는 대략 이런 순서로 움직인다.

1. URL을 해석한다
2. 도메인 이름을 IP 주소로 바꾼다
3. 서버와 연결한다
4. HTTP 요청을 보낸다
5. 서버로부터 HTTP 응답을 받는다
6. HTML을 읽는다
7. CSS를 적용한다
8. JavaScript를 실행한다
9. 화면을 그린다

이 흐름을 하나씩 보면 브라우저가 얼마나 중요한지 알 수 있다.

4. URL 해석

브라우저는 먼저 사용자가 입력한 주소를 해석한다.

https://example.com/boards?page=1

이 주소는 여러 부분으로 나눌 수 있다.

https
→ 프로토콜

example.com
→ 도메인

/boards
→ 경로

?page=1
→ 쿼리 파라미터

브라우저는 이 정보를 보고 어떤 방식으로, 어느 서버에, 어떤 요청을 보낼지 결정한다.

5. DNS 조회

브라우저는 example.com이라는 이름만으로는 서버에 접속할 수 없다.

컴퓨터 네트워크는 실제로 IP 주소를 사용하기 때문이다.

그래서 브라우저는 도메인 이름을 IP 주소로 바꾸는 과정을 거친다.

이걸 DNS 조회라고 한다.

example.com
→ 93.184.216.34

비유하면 DNS는 전화번호부 같은 역할을 한다.

사람 이름 → 전화번호
도메인 이름 → IP 주소

브라우저는 DNS를 통해 서버의 IP 주소를 알아낸 뒤, 그 서버로 요청을 보낸다.

6. HTTP 요청 보내기

서버의 위치를 알았다면, 브라우저는 HTTP 요청을 보낸다.

예를 들어 게시글 목록 페이지를 요청한다고 하자.

GET /boards HTTP/1.1
Host: example.com

이 요청의 의미는 다음과 같다.

example.com 서버야
/boards 경로에 있는 데이터를 줘
GET 방식으로 요청할게

이 요청을 받은 서버는 HTML이나 JSON 같은 응답을 돌려준다.

7. HTTP 응답 받기

서버는 브라우저에게 응답을 보낸다.

예를 들어 HTML 문서를 돌려줄 수 있다.

HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <head>
    <title>Hello</title>
  </head>
  <body>
    <h1>Hello World</h1>
  </body>
</html>

여기서 중요한 것은 브라우저가 이 응답을 해석한다는 점이다.

브라우저는 HTML을 그냥 문자열로 보여주는 것이 아니라, 구조를 분석해서 화면으로 바꾼다.

8. HTML 해석

HTML은 웹 페이지의 구조를 담당한다.

<h1>게시판</h1>
<p>첫 번째 글입니다.</p>
<button>등록</button>

브라우저는 HTML을 읽고 이런 구조를 만든다.

제목이 있다
문단이 있다
버튼이 있다

즉 HTML은 화면의 뼈대다.

하지만 HTML만 있으면 화면은 밋밋하다.

그래서 CSS가 필요하다.

9. CSS 적용

CSS는 웹 페이지의 디자인을 담당한다.

h1 {
  color: blue;
  font-size: 32px;
}

브라우저는 CSS를 읽고 HTML 요소에 스타일을 입힌다.

글자 색
크기
여백
정렬
배경색
레이아웃

즉 CSS는 화면의 옷과 같다.

HTML이 뼈대라면, CSS는 디자인이다.

10. JavaScript 실행

JavaScript는 웹 페이지의 동작을 담당한다.

예를 들어 버튼을 눌렀을 때 알림을 띄우는 코드가 있다고 하자.

document.querySelector("button").addEventListener("click", () => {
  alert("버튼 클릭");
});

브라우저는 이 JavaScript 코드를 실행한다.

그래서 사용자가 버튼을 클릭하면 실제로 동작이 일어난다.

버튼 클릭
입력값 검사
API 요청
화면 변경
모달 열기
페이지 이동

이런 동작을 JavaScript가 담당한다.

11. 브라우저와 프론트엔드의 관계

여기서 중요한 구분이 나온다.

브라우저 = 웹을 실행하는 프로그램
프론트엔드 = 브라우저 안에서 실행되는 코드와 화면

예를 들어 React로 만든 화면이 있다고 하자.

React 앱
HTML
CSS
JavaScript

이 코드는 혼자 실행되는 게 아니다.

브라우저가 이 코드를 받아서 실행한다.

Chrome
└── React 프론트엔드 앱 실행

즉 브라우저는 실행 환경이고, 프론트엔드는 그 안에서 돌아가는 웹 애플리케이션이다.

12. 브라우저와 백엔드의 관계

브라우저는 백엔드 서버와 직접 통신하기도 한다.

예를 들어 프론트엔드 코드가 게시글 목록을 가져오고 싶다고 하자.

fetch("http://localhost:8080/boards")

이 코드는 브라우저에서 실행된다.

브라우저는 이 요청을 실제 HTTP 요청으로 만들어 백엔드 서버에 보낸다.

브라우저
→ HTTP 요청
→ 백엔드 서버

백엔드는 보통 JSON 데이터를 응답한다.

[
  {
    "id": 1,
    "title": "첫 번째 글"
  }
]

그러면 브라우저 안에서 실행 중인 프론트엔드 코드가 이 데이터를 받아서 화면에 뿌린다.

전체 흐름은 이렇게 볼 수 있다.

사용자
↓
브라우저
↓
프론트엔드 코드 실행
↓
백엔드 API 요청
↓
DB 조회
↓
JSON 응답
↓
프론트엔드 화면 갱신

13. 브라우저가 보안 정책도 적용한다

브라우저는 화면만 그리는 게 아니라 보안 정책도 적용한다.

대표적인 것이 CORS다.

예를 들어 프론트엔드는 다음 주소에서 실행 중이다.

http://localhost:5173

백엔드는 다음 주소에서 실행 중이다.

http://localhost:8080

포트가 다르기 때문에 브라우저 입장에서는 서로 다른 출처다.

이때 백엔드가 허용하지 않은 요청이면 브라우저가 막을 수 있다.

CORS policy error

중요한 점은 CORS는 브라우저가 적용하는 정책이라는 것이다.

서버끼리 통신할 때는 보통 CORS 문제가 나지 않는다.

예를 들어 Postman이나 curl에서는 되는데 브라우저에서는 안 되는 경우가 있다.

그럴 때 CORS를 의심할 수 있다.

14. 브라우저는 쿠키도 저장한다

로그인 기능을 만들 때 쿠키라는 말을 자주 듣는다.

쿠키는 브라우저에 저장되는 작은 데이터다.

서버가 응답하면서 이런 헤더를 보낼 수 있다.

Set-Cookie: sessionId=abc123

그러면 브라우저는 이 쿠키를 저장한다.

이후 같은 서버로 요청을 보낼 때 브라우저가 쿠키를 자동으로 같이 보낸다.

Cookie: sessionId=abc123

그래서 서버는 이 사용자가 누구인지 알 수 있다.

정리하면 다음과 같다.

서버가 Set-Cookie로 쿠키 저장 요청
브라우저가 쿠키 저장
브라우저가 다음 요청부터 Cookie 헤더에 쿠키 포함
서버가 쿠키를 보고 사용자 식별

15. 브라우저는 캐시도 관리한다

브라우저는 매번 모든 파일을 새로 다운로드하지 않는다.

이미 받은 이미지, CSS, JavaScript 파일을 저장해두었다가 다시 사용할 수 있다.

이걸 캐시라고 한다.

캐시를 사용하면 페이지 로딩 속도가 빨라진다.

하지만 개발 중에는 캐시 때문에 헷갈릴 때도 있다.

예를 들어 코드를 수정했는데 화면이 그대로라면 브라우저 캐시가 원인일 수 있다.

이럴 때는 강력 새로고침을 하거나 캐시를 비워본다.

Windows: Ctrl + F5
Mac: Cmd + Shift + R

16. 개발자 도구

브라우저에는 개발자 도구가 있다.

Chrome 기준으로 보통 F12 또는 우클릭 후 검사를 누르면 열린다.

개발자 도구에서는 다음을 확인할 수 있다.

Elements
→ HTML/CSS 구조 확인

Console
→ JavaScript 로그와 에러 확인

Network
→ HTTP 요청/응답 확인

Application
→ 쿠키, 로컬스토리지 확인

Sources
→ JavaScript 파일 디버깅

웹 개발에서 개발자 도구는 거의 필수다.

특히 Network 탭은 API 트러블슈팅할 때 매우 중요하다.

요청 URL이 맞는지
HTTP Method가 맞는지
Status Code가 무엇인지
Request Header가 무엇인지
Response Body가 무엇인지
CORS 에러가 있는지

이런 것들을 확인할 수 있다.

17. 브라우저, 프론트엔드, 백엔드를 같이 보면

게시판 서비스를 예로 들어보자.

Chrome
→ 브라우저

React 게시판 화면
→ 프론트엔드

Spring Boot 게시판 API
→ 백엔드

PostgreSQL
→ DB

사용자가 게시글 목록 페이지에 들어가면 흐름은 다음과 같다.

1. 사용자가 Chrome에서 게시판 주소 접속
2. 브라우저가 프론트엔드 파일 다운로드
3. 브라우저가 HTML/CSS/JS 실행
4. React가 화면을 그림
5. React 코드가 Spring Boot API 호출
6. Spring Boot가 DB에서 게시글 조회
7. Spring Boot가 JSON 응답
8. 브라우저 안의 React가 JSON을 받아 화면에 표시

여기서 브라우저는 전체 흐름의 실행 무대다.

프론트엔드는 그 무대 위에서 돌아가는 코드다.

백엔드는 데이터를 처리하는 서버다.

18. 헷갈리기 쉬운 정리

브라우저는 서버인가?

아니다.

브라우저는 사용자의 컴퓨터나 휴대폰에서 실행되는 클라이언트 프로그램이다.

Chrome
Safari
Edge

이런 것들이 브라우저다.

브라우저는 프론트엔드인가?

아니다.

브라우저는 프론트엔드를 실행하는 프로그램이다.

브라우저 = 실행 환경
프론트엔드 = 실행되는 코드

HTML/CSS/JS는 누가 실행하나?

브라우저가 실행한다.

HTML → 브라우저가 구조로 해석
CSS → 브라우저가 스타일로 적용
JavaScript → 브라우저의 JS 엔진이 실행

React는 브라우저에서 실행되나?

맞다.

React 코드는 최종적으로 JavaScript로 변환되어 브라우저에서 실행된다.

API 요청은 누가 보내나?

브라우저 안에서 실행 중인 프론트엔드 코드가 요청을 만들고, 브라우저가 실제 네트워크 요청을 보낸다.

프론트엔드 코드가 fetch 호출
브라우저가 HTTP 요청 전송
백엔드가 응답
브라우저가 응답을 프론트엔드 코드에 전달

19. 브라우저를 이해하면 좋은 이유

브라우저를 이해하면 웹 개발에서 생기는 많은 문제가 덜 헷갈린다.

예를 들어 이런 문제를 구분할 수 있다.

서버가 안 떠 있는 문제
API 경로가 틀린 문제
CORS에 막힌 문제
쿠키가 안 붙는 문제
캐시 때문에 화면이 안 바뀌는 문제
JavaScript 에러로 화면이 멈춘 문제
인증 토큰이 헤더에 안 들어간 문제

이 문제들은 전부 “웹이 안 된다”로 보이지만 원인은 다르다.

브라우저를 이해하면 어디를 봐야 할지 감이 생긴다.

20. 결론

브라우저는 단순히 인터넷을 보는 창이 아니다.

브라우저는 웹 애플리케이션을 실행하는 프로그램이다.

브라우저는 서버에 요청을 보내고, 응답으로 받은 HTML/CSS/JavaScript를 해석해서 화면을 만든다.

또한 쿠키, 캐시, 보안 정책, CORS, 개발자 도구, JavaScript 실행까지 담당한다.

정리하면 다음과 같다.

브라우저
= 웹 페이지를 가져오고 실행하는 프로그램

프론트엔드
= 브라우저 안에서 실행되는 화면과 코드
