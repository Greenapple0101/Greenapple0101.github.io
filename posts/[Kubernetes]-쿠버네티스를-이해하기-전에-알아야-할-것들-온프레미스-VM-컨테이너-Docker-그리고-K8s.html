2026.05.17

쿠버네티스를 처음 공부할 때 바로 Pod, Service, Deployment부터 들어가면 헷갈리기 쉽다. 쿠버네티스는 갑자기 튀어나온 기술이 아니라, 서버를 운영하는 방식이 변화하면서 자연스럽게 등장한 기술이기 때문이다.

흐름을 크게 보면 다음과 같다.

온프레미스 물리 서버
→ 가상머신 VM
→ 컨테이너
→ Docker
→ 컨테이너 오케스트레이션
→ Kubernetes

각 기술은 이전 기술을 완전히 없애기 위해 나온 것이 아니다. 지금도 온프레미스, VM, 컨테이너는 모두 사용된다. 다만 서비스 규모, 운영 방식, 배포 속도, 자원 활용률에 따라 선택지가 달라졌을 뿐이다.

1. 온프레미스란?

온프레미스는 기업이나 기관이 직접 물리 서버를 구매해서 운영하는 방식이다.

쉽게 말하면 회사 전산실이나 데이터센터에 실제 서버 장비를 두고, 그 위에 운영체제와 애플리케이션을 직접 설치해서 운영하는 구조다.

물리 서버
└── OS
    └── Java / WAS / DB / Application

온프레미스의 장점은 성능을 최대한 직접 활용할 수 있다는 점이다. 물리 서버 위에 바로 애플리케이션을 설치하면 중간 계층이 적기 때문에 CPU, 메모리, 디스크 자원을 비교적 직접적으로 사용할 수 있다.

하지만 단점도 크다.

서버 한 대를 구매하는 데 시간이 오래 걸린다. 장비 발주, 입고, 설치, 네트워크 연결, 보안 설정까지 하면 몇 달이 걸릴 수도 있다. 또한 실제 운영에서는 장애나 트래픽 급증에 대비해야 하므로 CPU를 항상 100%까지 쓰지 않는다.

예를 들어 평소 CPU 사용률을 40~50% 정도로 유지하고, 나머지는 피크 타임을 대비해 남겨둔다. 그러면 새벽처럼 트래픽이 거의 없는 시간에는 많은 자원이 놀게 된다.

즉, 온프레미스는 성능은 좋지만 자원 활용률과 확장성 측면에서는 한계가 있다.

2. VM은 왜 등장했을까?

VM은 Virtual Machine, 즉 가상머신이다.

온프레미스 물리 서버 하나를 여러 개의 가상 서버처럼 나눠 쓰기 위해 등장했다.

물리 서버
└── Hypervisor
    ├── VM1 - Linux
    ├── VM2 - Windows
    └── VM3 - Linux

VM의 핵심은 하이퍼바이저다.

하이퍼바이저는 물리 서버의 CPU, 메모리, 디스크 같은 자원을 여러 VM에 나누어주는 역할을 한다. 각각의 VM은 독립된 운영체제를 가진다. 어떤 VM에는 Linux를 설치하고, 다른 VM에는 Windows를 설치할 수도 있다.

VM을 쓰면 하나의 물리 서버를 더 잘게 나누어 사용할 수 있다. 애플리케이션끼리 격리도 된다. 한 VM에서 문제가 생겨도 다른 VM에 직접적인 영향을 덜 준다.

하지만 VM은 무겁다. VM마다 운영체제가 올라가야 하기 때문이다.

VM = OS까지 포함한 가상 서버

운영체제까지 매번 부팅해야 하므로 생성과 실행에 시간이 걸린다. 자원도 많이 사용한다.

3. 컨테이너는 VM과 무엇이 다를까?

컨테이너는 VM보다 가볍다.

VM은 각 가상머신마다 OS를 따로 가진다. 반면 컨테이너는 호스트 OS의 커널을 공유한다.

VM 구조

물리 서버
└── Hypervisor
    ├── Guest OS
    │   └── App
    └── Guest OS
        └── App

Container 구조

물리 서버 또는 VM
└── Host OS
    └── Container Runtime
        ├── Container A
        ├── Container B
        └── Container C

컨테이너는 OS 전체를 새로 띄우는 것이 아니라, 애플리케이션 실행에 필요한 파일, 라이브러리, 바이너리, 설정만 묶어서 실행한다.

그래서 VM보다 훨씬 빠르게 생성되고 삭제된다.

VM 생성 = OS 부팅부터 시작
컨테이너 생성 = 이미지를 실행

개발이나 테스트 환경에서 컨테이너가 특히 강력한 이유가 여기에 있다. 로컬 OS에 직접 이것저것 설치하지 않고, 컨테이너 안에 필요한 환경을 만들어서 실행하면 된다. 지우고 다시 띄우기도 쉽고, 환경 충돌도 줄어든다.

4. AWS에서 컨테이너를 띄운다는 것은?

AWS 같은 클라우드에서는 보통 VM 위에 컨테이너를 띄운다.

예를 들어 EC2 인스턴스는 가상머신이다. 그 EC2 위에 Docker를 설치하고 컨테이너를 실행하는 구조다.

AWS 물리 인프라
└── EC2 VM
    └── Docker / containerd
        └── Container

즉, 현실에서는 이런 조합이 흔하다.

VM + Container

VM을 통해 클라우드 자원을 유연하게 확보하고, 그 위에서 컨테이너를 실행해 애플리케이션을 빠르게 배포한다.

물론 성능이 극도로 중요하면 베어메탈 서버에 직접 설치하는 방식이 더 나을 수 있다. 중간 가상화 계층이 줄어들기 때문이다. 하지만 대부분의 서비스는 클라우드 VM 위에서 컨테이너를 실행하는 방식으로도 충분하다.

그래서 애플리케이션을 작게 쪼개는 MSA 구조도 함께 중요해진다. 작게 나눈 서비스들은 컨테이너로 배포하기 좋고, 필요할 때 빠르게 늘리고 줄일 수 있기 때문이다.

5. Docker 이미지와 컨테이너

Docker를 이해하려면 이미지와 컨테이너의 차이를 알아야 한다.

이미지는 실행 가능한 애플리케이션 패키지다.

이미지 안에는 애플리케이션 코드, 실행에 필요한 바이너리, 라이브러리, 설정 등이 들어 있다.

Docker Image
= 실행에 필요한 파일 묶음
= 애플리케이션 실행 패키지

컨테이너는 이미지를 실제로 실행한 상태다.

Image → run → Container

비유하면 이미지는 프로그램 설치 파일이고, 컨테이너는 그 프로그램을 실제로 실행한 프로세스에 가깝다.

이미지만 있고 컨테이너로 실행하지 않으면 실제 서비스는 동작하지 않는다.

이미지 = 실행 전 패키지
컨테이너 = 실행 중인 인스턴스

6. Dockerfile, build, image, container 흐름

Docker 이미지는 보통 Dockerfile로 만든다.

Dockerfile은 이미지를 만들기 위한 스크립트다.

예를 들면 이런 흐름이다.

Dockerfile 작성
→ docker build
→ Docker image 생성
→ docker create / docker run
→ Container 실행

Dockerfile 안에는 이런 명령어들이 들어간다.

FROM openjdk:17
COPY app.jar app.jar
CMD ["java", "-jar", "app.jar"]

여기서 중요한 점은 이미지를 처음부터 전부 만드는 것이 아니라는 점이다. 보통 베이스 이미지에서 시작한다.

ubuntu 이미지
python 이미지
node 이미지
openjdk 이미지
nginx 이미지
postgres 이미지

이런 베이스 이미지 위에 내가 필요한 파일과 설정을 추가한다.

Docker 이미지는 여러 레이어로 구성된다. 각 명령어가 레이어를 만들고, Docker는 이 레이어를 캐싱한다.

그래서 Docker build를 할 때 캐싱이 잘 되면 빌드 속도가 빨라진다. 반대로 캐싱이 깨지면 매번 처음부터 다시 설치하는 것처럼 오래 걸릴 수 있다.

7. Docker Compose는 왜 쓸까?

컨테이너 하나만 띄울 때는 docker run으로 충분하다.

하지만 실제 서비스는 보통 하나의 컨테이너로 끝나지 않는다.

예를 들어 웹 서비스 하나를 만들더라도 다음과 같은 구성일 수 있다.

frontend
backend
database
nginx
redis

이걸 매번 하나씩 docker run으로 띄우면 너무 번거롭다.

Docker Compose는 여러 컨테이너를 하나의 설정 파일로 묶어서 실행할 수 있게 해준다.

docker compose up

이 명령어 하나로 여러 컨테이너를 동시에 띄울 수 있다.

개발과 테스트 환경에서는 Docker Compose가 굉장히 편하다. OS에 직접 설치하지 않아도 되고, 컨테이너를 지웠다가 다시 만들기도 쉽다.

8. 그런데 컨테이너가 많아지면 문제가 생긴다

처음에는 Docker Compose로 충분해 보인다.

그런데 컨테이너가 점점 많아지면 문제가 생긴다.

예를 들어 컨테이너 네 개를 돌리고 있는데 하나가 죽었다고 해보자.

그러면 사람이 직접 확인해야 한다.

어떤 컨테이너가 죽었는지 확인
어느 서버에 있는지 확인
서버에 접속
컨테이너 재시작
로그 확인
네트워크 확인

컨테이너가 몇 개 없을 때는 괜찮다.

하지만 컨테이너가 수십 개, 수백 개가 되면 사람이 직접 관리하기 어렵다.

또 다른 문제도 있다.

어느 서버에 컨테이너를 배치할 것인가?
CPU가 여유 있는 노드는 어디인가?
컨테이너가 죽으면 누가 다시 살릴 것인가?
IP가 바뀌면 어떻게 접근할 것인가?
로드밸런서는 어떻게 붙일 것인가?
볼륨은 어디에 저장할 것인가?
배포 중 장애가 나면 어떻게 롤백할 것인가?

이런 문제를 해결하기 위해 등장한 것이 컨테이너 오케스트레이션이다.

그리고 대표적인 컨테이너 오케스트레이션 도구가 쿠버네티스다.

9. 쿠버네티스란?

쿠버네티스는 컨테이너를 관리하는 시스템이다.

단순히 컨테이너를 실행하는 도구가 아니라, 컨테이너를 어디에 배치하고, 몇 개 실행하고, 죽으면 어떻게 살리고, 네트워크를 어떻게 연결하고, 배포를 어떻게 할지 관리한다.

Kubernetes
= 컨테이너 오케스트레이션 플랫폼

쿠버네티스가 해주는 대표적인 일은 다음과 같다.

컨테이너 배치
컨테이너 개수 유지
장애 발생 시 자동 복구
서비스 네트워크 제공
로드밸런싱
롤아웃 / 롤백
볼륨 연결
스케줄링
셀프 힐링

예를 들어 “nginx 컨테이너를 3개 띄워줘”라고 선언하면, 쿠버네티스는 실제 상태를 계속 확인하면서 3개가 유지되도록 관리한다.

하나가 죽으면 다시 띄운다.

이게 셀프 힐링이다.

10. 스케줄링과 빈패킹

쿠버네티스는 컨테이너를 아무 노드에나 배치하지 않는다.

어느 노드에 CPU와 메모리 여유가 있는지 보고 배치한다.

이런 역할을 하는 컴포넌트가 스케줄러다.

Scheduler
= Pod를 어느 Node에 배치할지 결정하는 컴포넌트

빈패킹이라는 개념도 있다. 자원을 최대한 효율적으로 채워 넣는 방식이다. 하지만 운영에서는 무조건 꽉 채우는 것만 좋은 것은 아니다. 너무 빽빽하게 채우면 장애나 트래픽 증가에 취약할 수 있기 때문이다.

그래서 실제 운영에서는 자원 효율성과 안정성 사이에서 균형을 맞춰야 한다.

11. CNCF와 쿠버네티스 생태계

쿠버네티스는 CNCF에 기증되었다.

CNCF는 Cloud Native Computing Foundation의 약자다. 클라우드 네이티브 기술과 오픈소스 생태계를 관리하고 지원하는 재단이라고 보면 된다.

CNCF 생태계에는 쿠버네티스뿐 아니라 수많은 오픈소스 솔루션이 있다.

예를 들면 다음과 같은 영역들이 있다.

컨테이너 런타임
네트워크
스토리지
서비스 메시
모니터링
로깅
CI/CD
보안
레지스트리

쿠버네티스를 운영하다 보면 결국 CNCF 생태계의 여러 도구들을 함께 만나게 된다.

12. Rancher는 왜 사용할까?

Rancher는 쿠버네티스 클러스터를 관리하기 위한 플랫폼이다.

쿠버네티스를 CLI로만 관리할 수도 있지만, 여러 클러스터와 노드, 파드, 워크로드를 한눈에 보고 관리하려면 UI 기반 관리 도구가 필요해진다.

Rancher는 이런 역할을 한다.

클러스터 관리
노드 관리
파드 관리
워크로드 관리
모니터링
권한 관리
멀티 클러스터 관리

특히 퍼블릭 클라우드를 쓰기 어려운 환경에서는 설치형 관리 플랫폼이 중요하다.

예를 들어 금융권이나 공공기관처럼 보안 규제가 강한 곳은 AWS, GCP 같은 퍼블릭 클라우드를 바로 쓰기 어려운 경우가 있다. 이런 경우 온프레미스나 프라이빗 클라우드 환경에 쿠버네티스를 설치하고, Rancher로 관리하는 방식이 유용하다.

Rancher에서는 관리 대상 클러스터를 다운스트림 클러스터로 본다.

Rancher UI에서 설정을 변경하면, 그 변경이 실제 쿠버네티스 클러스터에 반영된다. RKE2 같은 Rancher 계열 Kubernetes 배포판과 함께 사용되는 경우도 많다.

13. K3d는 무엇인가?

실습 환경에서는 실제 서버 여러 대를 준비하기 어렵다.

그래서 로컬에서 가볍게 쿠버네티스를 실습할 수 있는 도구를 사용한다.

그중 하나가 K3d다.

K3d는 K3s를 Docker 컨테이너 안에서 실행할 수 있게 해주는 도구다.

K3d
= Docker 위에서 가볍게 Kubernetes 클러스터를 띄우는 도구

실제 운영용 대규모 클러스터는 아니지만, 로컬 실습이나 학습용으로는 매우 편하다.

14. Kubernetes 아키텍처

쿠버네티스 클러스터는 크게 두 영역으로 나뉜다.

Control Plane
Worker Node

Control Plane은 클러스터를 관리하는 영역이다.

Worker Node는 실제 애플리케이션 컨테이너가 실행되는 영역이다.

Kubernetes Cluster

Control Plane
├── kube-apiserver
├── scheduler
├── controller-manager
└── etcd

Worker Node
├── kubelet
├── container runtime
├── CNI plugin
└── Pods

15. kube-apiserver

쿠버네티스에서 모든 요청은 kube-apiserver를 통한다.

우리가 kubectl get pod 같은 명령어를 입력하면, kubectl은 내부적으로 Kubernetes API Server에 요청을 보낸다.

kubectl get pods

이 명령은 단순히 로컬에서 실행되는 것이 아니다.

흐름은 다음과 같다.

kubectl
→ kube-apiserver
→ etcd 조회
→ 결과 반환

API란 정해진 형식의 요청과 응답 규칙이다.

kube-apiserver는 쿠버네티스 클러스터의 API 요청을 받아들이는 입구다.

중요한 점은 etcd와 직접 통신하는 핵심 컴포넌트가 kube-apiserver라는 것이다.

kube-apiserver
= Kubernetes 클러스터의 중앙 API 입구

16. etcd

etcd는 쿠버네티스의 저장소다.

쿠버네티스 클러스터의 상태 정보가 etcd에 저장된다.

예를 들어 사용자가 이런 요청을 했다고 해보자.

nginx Pod를 4개 만들어줘

이런 원하는 상태가 etcd에 저장된다.

쿠버네티스는 etcd에 저장된 원하는 상태와 현재 상태를 비교하면서 계속 맞춰간다.

Desired State: Pod 4개
Current State: Pod 3개
→ 하나 더 생성

그래서 쿠버네티스 백업에서 가장 중요한 것이 etcd 백업이다.

Kubernetes 백업의 핵심 = etcd 백업

다만 etcd는 애플리케이션 데이터베이스가 아니다. 사용자가 업로드한 파일이나 서비스 데이터까지 etcd에 저장하는 것이 아니다.

etcd는 클러스터 상태 정보를 저장하는 저장소다.

17. 왜 Control Plane 노드는 홀수 개를 권장할까?

운영 환경에서는 Control Plane 노드를 보통 3개, 5개, 7개처럼 홀수 개로 구성한다.

이유는 etcd의 쿼럼 때문이다.

etcd는 분산 저장소이고, 리더를 선출해야 한다. 여러 노드가 있을 때 과반수의 동의를 받아야 안정적으로 의사결정을 할 수 있다.

3개 중 과반수 = 2개
5개 중 과반수 = 3개
7개 중 과반수 = 4개

만약 Control Plane이 1개라면 그 노드가 죽는 순간 클러스터 관리 기능이 멈출 수 있다.

2개는 애매하다. 둘 중 하나가 죽으면 과반수를 만족하기 어렵고, 둘이 서로 다른 판단을 할 때 결정이 어려워진다.

그래서 운영 환경에서는 3개 이상의 홀수 개 Control Plane 구성이 권장된다.

Worker Node는 홀수일 필요가 없다. 실제 애플리케이션을 실행하는 노드이기 때문에 필요에 따라 1개, 2개, 10개, 100개로 늘릴 수 있다.

18. kube-scheduler

kube-scheduler는 Pod를 어느 노드에 배치할지 결정한다.

사용자가 Deployment를 만들면 쿠버네티스는 Pod를 생성해야 한다. 그런데 클러스터 안에는 여러 Worker Node가 있을 수 있다.

스케줄러는 다음과 같은 요소를 고려한다.

CPU 여유
메모리 여유
노드 상태
taint / toleration
nodeSelector
affinity
리소스 요청량

그리고 가장 적절한 노드를 선택해 Pod를 배치한다.

Pod 생성 요청
→ Scheduler가 적절한 Node 선택
→ 해당 Node의 kubelet이 Pod 실행

19. kubelet

kubelet은 각 Worker Node에 설치되는 에이전트다.

Worker Node마다 kubelet이 있어야 한다.

kubelet은 Control Plane과 통신하면서 자기 노드의 상태를 보고하고, Control Plane이 지시한 Pod를 실제로 실행되도록 만든다.

kubelet
= Worker Node의 현장 관리자

예를 들어 Control Plane이 “이 노드에 nginx Pod를 실행해”라고 결정하면, 해당 노드의 kubelet이 컨테이너 런타임에게 요청해서 컨테이너를 실행한다.

또한 kubelet은 노드와 Pod의 상태를 계속 보고한다.

노드 상태 보고
파드 상태 보고
컨테이너 실행 요청
헬스 체크

kubelet이 죽으면 해당 노드는 정상적으로 관리되지 않는다. 그래서 kubelet은 Worker Node에서 매우 중요한 컴포넌트다.

20. Container Runtime

컨테이너 런타임은 실제로 컨테이너를 실행하는 역할을 한다.

예전에는 Docker Engine이 많이 사용되었지만, 현재 Kubernetes에서는 containerd 같은 런타임이 많이 사용된다.

Container Runtime
= 컨테이너를 실제로 실행하는 컴포넌트

흐름은 다음과 같다.

kube-apiserver
→ scheduler
→ kubelet
→ container runtime
→ container 실행

즉, Kubernetes가 직접 컨테이너를 실행하는 것이 아니라, kubelet이 컨테이너 런타임에게 실행을 요청하는 구조다.

21. CNI와 네트워크 플러그인

쿠버네티스에서는 수많은 Pod가 생성된다.

Pod들은 서로 통신해야 하고, 외부와도 통신해야 한다. 이 네트워크를 구성하는 역할을 하는 것이 CNI 플러그인이다.

CNI
= Container Network Interface

대표적인 CNI 플러그인은 다음과 같다.

Calico
Canal
Cilium
Flannel

Rancher 계열에서는 Canal을 사용하는 경우가 있고, 최근에는 Cilium도 많이 주목받고 있다.

CNI는 Pod 간 통신, 네트워크 정책, 라우팅 등에 관여한다. 인프라 엔지니어가 쿠버네티스를 깊게 공부하려면 CNI 영역도 중요해진다.

22. Stateless와 Stateful

컨테이너는 기본적으로 휘발적이다.

컨테이너가 삭제되면 컨테이너 안에 저장된 데이터도 사라질 수 있다.

그래서 애플리케이션은 크게 두 종류로 나눌 수 있다.

Stateless
Stateful

Stateless는 상태를 내부에 저장하지 않는 애플리케이션이다.

예를 들어 단순 API 서버는 요청을 받고 응답만 한다. 데이터는 외부 DB나 스토리지에 저장한다.

API Server
→ DB에 데이터 저장
→ 컨테이너 자체에는 중요한 상태 없음

반면 Stateful은 상태를 가진 애플리케이션이다.

대표적으로 DB가 있다.

Database
File Storage
Message Queue

Stateful 애플리케이션을 쿠버네티스에서 운영하는 것은 더 어렵다. Pod가 다른 노드로 옮겨갔을 때 데이터는 어떻게 할 것인지, 볼륨은 어디에 붙일 것인지, 장애 시 복구는 어떻게 할 것인지 고민해야 한다.

그래서 운영에서는 중요한 데이터는 외부 DB, NAS, S3 같은 리모트 스토리지에 저장하는 구조를 많이 사용한다.

23. Kubernetes에서 볼륨이 중요한 이유

컨테이너 안에 파일을 저장하면 컨테이너가 사라질 때 데이터도 사라질 수 있다.

예를 들어 사용자가 첨부파일을 업로드했는데, 그 파일이 컨테이너 내부에만 저장되어 있다면 문제가 된다.

컨테이너가 죽고 새로 생성되면 파일이 사라질 수 있기 때문이다.

그래서 볼륨이 필요하다.

컨테이너 내부 저장
→ 컨테이너 삭제 시 데이터 유실 가능

외부 볼륨 저장
→ 컨테이너가 재생성되어도 데이터 유지 가능

싱글 노드에서는 로컬 볼륨으로 해결할 수 있지만, 노드가 여러 개가 되면 더 복잡해진다.

Pod가 Node A에서 Node B로 이동했는데 데이터가 Node A에만 있으면 문제가 된다.

그래서 NFS, NAS, S3, 외부 DB, CSI 스토리지 같은 개념이 중요해진다.

24. Rollout은 무엇인가?

롤아웃은 새로운 버전의 애플리케이션을 배포하는 과정이다.

예를 들어 nginx v1을 실행하고 있다가 v2로 바꾸는 상황을 생각할 수 있다.

nginx:v1
→ nginx:v2

쿠버네티스는 Deployment를 통해 점진적으로 새 버전을 배포할 수 있다.

문제가 생기면 이전 버전으로 되돌리는 롤백도 가능하다.

rollout
= 새 버전 배포

rollback
= 이전 버전으로 되돌리기

25. 전체 흐름 정리

쿠버네티스를 이해하려면 단순히 명령어만 외우면 안 된다.

왜 이런 기술이 나왔는지 흐름을 이해해야 한다.

온프레미스
= 물리 서버를 직접 운영
= 성능은 좋지만 확장과 자원 활용에 한계

VM
= 물리 서버를 여러 가상 서버로 나눔
= 격리성은 좋지만 무거움

컨테이너
= 애플리케이션 실행 환경을 가볍게 패키징
= 빠르게 실행, 삭제, 배포 가능

Docker
= 컨테이너 이미지를 만들고 실행하는 대표 도구

Docker Compose
= 여러 컨테이너를 개발/테스트 환경에서 쉽게 실행

Kubernetes
= 많은 컨테이너를 자동으로 배치, 복구, 확장, 관리하는 오케스트레이션 플랫폼

Rancher
= Kubernetes 클러스터를 UI 기반으로 관리하는 플랫폼

K3d
= 로컬에서 가볍게 Kubernetes를 실습하기 위한 도구

마무리

쿠버네티스는 단순히 컨테이너를 실행하는 도구가 아니다.

컨테이너가 많아졌을 때 발생하는 운영 문제를 해결하기 위한 시스템이다.

컨테이너가 죽으면 다시 살리고, 어느 노드에 배치할지 결정하고, 네트워크와 볼륨을 연결하고, 배포와 롤백을 관리한다.

즉 쿠버네티스는 다음 질문에 답하기 위해 등장했다.

컨테이너가 많아지면 누가 관리할 것인가?
컨테이너가 죽으면 누가 살릴 것인가?
어느 서버에 배치할 것인가?
IP가 바뀌면 어떻게 접근할 것인가?
데이터는 어디에 저장할 것인가?
배포 중 장애가 나면 어떻게 되돌릴 것인가?

이 질문들에 대한 답이 바로 쿠버네티스다.
