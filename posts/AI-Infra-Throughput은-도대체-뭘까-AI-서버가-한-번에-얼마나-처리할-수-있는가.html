[AI Infra] Throughput은 도대체 뭘까? AI 서버가 한 번에 얼마나 처리할 수 있는가

2026.05.07 Velog 원문

AI 인프라, 백엔드, 네트워크, GPU 추론 서버를 공부하다 보면 throughput이라는 단어가 자주 나온다.

한국어로는 보통 처리량이라고 한다.

한 줄로 말하면 이렇다.

Throughput = 단위 시간 동안 처리할 수 있는 작업량

AI inference 서버 기준으로는 보통 이렇게 이해하면 된다.

Throughput = 1초 동안 처리할 수 있는 요청 수 또는 토큰 수

1. Throughput 한 줄 정의

Throughput은 시스템이 일정 시간 동안 얼마나 많은 일을 처리할 수 있는지를 나타낸다.

예를 들어 API 서버가 1초에 요청 100개를 처리할 수 있다면 throughput은 다음과 같다.

Throughput = 100 requests/sec

LLM 서버라면 요청 수뿐만 아니라 토큰 수로도 본다.

Throughput = 500 tokens/sec

즉, throughput은 “한 명의 요청이 얼마나 빨리 끝나는가”보다는 “서버 전체가 얼마나 많은 요청을 감당할 수 있는가”에 가깝다.

2. Latency와 Throughput 차이

이 둘은 꼭 같이 봐야 한다.

Latency = 요청 하나가 끝나는 데 걸리는 시간
Throughput = 단위 시간 동안 처리한 요청 수

식당으로 비유하면 이렇다.

Latency = 손님 한 명이 주문하고 음식을 받기까지 걸린 시간
Throughput = 식당이 1시간 동안 처리한 손님 수

AI 서버로 바꾸면 이렇게 된다.

Latency = 사용자 한 명이 응답을 받기까지 걸린 시간
Throughput = 서버가 1초 동안 처리한 요청 수

표로 보면 더 명확하다.

구분 Latency Throughput
의미 요청 1개의 응답 시간 단위 시간당 처리량
관심 대상 사용자 한 명의 체감 속도 서버 전체 처리 능력
단위 ms, sec requests/sec, tokens/sec
좋다는 의미 낮을수록 좋음 높을수록 좋음
예시 응답 300ms 초당 100개 요청 처리

3. Throughput이 중요한 이유

AI 서비스를 운영할 때는 단순히 “요청 하나가 빠른가”만 보면 부족하다.

사용자가 한 명이면 latency만 봐도 된다.

하지만 실제 서비스는 여러 명이 동시에 요청한다.

사용자 A 요청
사용자 B 요청
사용자 C 요청
사용자 D 요청
...

이때 서버가 요청을 많이 처리하지 못하면 큐가 쌓인다.

요청 도착 속도 > 서버 처리 속도

그러면 어떻게 될까?

대기열 증가
응답 지연 증가
timeout 발생
서버 부하 증가
사용자 경험 저하

즉, throughput이 낮으면 요청이 몰렸을 때 latency도 같이 나빠진다.

4. AI Inference에서 Throughput 단위

AI inference에서는 throughput을 여러 방식으로 측정할 수 있다.

requests/sec
images/sec
samples/sec
tokens/sec
queries/sec

예를 들어 이미지 분류 모델이면 이렇게 볼 수 있다.

초당 이미지 500장 처리
= 500 images/sec

일반 API 서버면 이렇게 본다.

초당 요청 100개 처리
= 100 requests/sec

LLM 서버면 보통 토큰 기준도 중요하다.

초당 출력 토큰 2000개 생성
= 2000 tokens/sec

LLM은 요청마다 답변 길이가 다르기 때문에 단순 요청 수만 보면 부족하다.

짧은 답변 100개와 긴 답변 100개는 서버 부담이 다르다.

그래서 LLM에서는 tokens/sec가 중요하다.

5. Throughput은 어떻게 계산할까?

가장 단순한 계산은 이렇다.

Throughput = 처리한 작업 수 / 걸린 시간

예를 들어 10초 동안 요청 500개를 처리했다면

500 requests / 10 sec = 50 requests/sec

즉, throughput은 초당 50개 요청이다.

PyTorch inference에서도 간단히 계산할 수 있다.

import time
import torch

model.eval()

num_samples = 1000

start = time.time()

with torch.inference_mode():
    for x in data_loader:
        x = x.to(device)
        output = model(x)

if torch.cuda.is_available():
    torch.cuda.synchronize()

end = time.time()

throughput = num_samples / (end - start)

print(f"Throughput: {throughput:.2f} samples/sec")

GPU를 쓸 때는 정확한 시간 측정을 위해 torch.cuda.synchronize()를 넣는 것이 좋다.

CUDA 연산은 비동기적으로 실행될 수 있기 때문이다.

6. Batch와 Throughput의 관계

Throughput을 이해하려면 batch를 같이 봐야 한다.

Batch는 한 번에 모델에 넣는 데이터 묶음이다.

batch size = 한 번에 처리하는 입력 개수

batch size가 1이면 요청 하나씩 처리한다.

요청 1개
모델 forward 1번
응답 1개

batch size가 16이면 요청 16개를 묶어서 처리한다.

요청 16개
모델 forward 1번
응답 16개

GPU는 병렬 연산에 강하다.

그래서 batch를 키우면 GPU를 더 효율적으로 사용할 수 있고, throughput이 증가할 수 있다.

batch size 증가
→ GPU 활용률 증가
→ 초당 처리량 증가 가능

7. 그런데 Batch를 키우면 무조건 좋을까?

그렇지는 않다.

batch size를 키우면 throughput은 좋아질 수 있지만, latency는 나빠질 수 있다.

왜냐하면 요청을 batch로 묶기 위해 잠깐 기다려야 하기 때문이다.

요청 1개 도착
↓
다른 요청 기다림
↓
batch 구성
↓
모델 추론
↓
응답 반환

사용자 입장에서는 “다른 요청을 기다리는 시간”도 latency다.

그래서 이런 trade-off가 생긴다.

batch size 작음
= latency 유리
= throughput 불리

batch size 큼
= throughput 유리
= latency 불리

즉, 실시간 AI 서비스에서는 throughput만 보고 batch를 무작정 키우면 안 된다.

8. GPU 관점에서 Throughput

GPU는 일감을 크게 줄수록 효율이 잘 나오는 경우가 많다.

batch size가 너무 작으면 GPU가 충분히 바빠지지 않는다.

batch size 1
= GPU에게 일감을 조금만 줌
= GPU 활용률 낮을 수 있음

반대로 적절한 batch size를 주면 GPU가 병렬 연산을 많이 수행한다.

batch size 32
= 한 번에 많은 입력 처리
= GPU 활용률 증가
= throughput 증가 가능

하지만 너무 크게 잡으면 GPU 메모리가 부족해진다.

CUDA out of memory

그래서 throughput 최적화는 결국 이런 균형이다.

GPU 메모리가 허용하는 범위 안에서
batch size를 키워서
GPU 활용률과 처리량을 높이는 것

9. LLM에서 Throughput은 더 중요하다

LLM 추론에서는 throughput을 보통 토큰 단위로 많이 본다.

tokens/sec

LLM은 답변을 한 번에 통째로 만드는 것이 아니라 토큰을 하나씩 생성한다.

프롬프트 입력
↓
첫 번째 토큰 생성
↓
두 번째 토큰 생성
↓
세 번째 토큰 생성
↓
...

그래서 LLM 서버에서는 이런 지표를 본다.

초당 처리한 입력 토큰 수
초당 생성한 출력 토큰 수
동시 요청 수
GPU당 tokens/sec

예를 들어 서버가 1초에 출력 토큰 1000개를 생성할 수 있다면

Throughput = 1000 output tokens/sec

이 값이 높을수록 같은 GPU로 더 많은 사용자를 처리할 수 있다.

10. Throughput과 비용

Throughput은 비용과 직접 연결된다.

같은 GPU 1대로 더 많은 요청을 처리하면 비용 효율이 좋아진다.

GPU 1대
초당 100 요청 처리

보다

GPU 1대
초당 300 요청 처리

가 더 효율적이다.

즉, throughput이 높으면 같은 인프라 비용으로 더 많은 트래픽을 받을 수 있다.

AI 서비스에서는 GPU 비용이 비싸기 때문에 throughput 최적화가 중요하다.

Throughput 증가
→ GPU당 처리량 증가
→ 요청당 비용 감소 가능

11. Throughput을 높이는 방법

AI inference 서버에서 throughput을 높이는 방법은 여러 가지가 있다.

batching
dynamic batching
model optimization
quantization
mixed precision
TensorRT
ONNX Runtime
vLLM
Triton Inference Server
GPU 메모리 최적화
전처리 병렬화

하나씩 가볍게 보면 이렇다.

12. Batching

여러 요청을 묶어서 한 번에 처리하는 방식이다.

요청 1개씩 처리
→ GPU 활용률 낮을 수 있음

요청 여러 개 묶어서 처리
→ GPU 활용률 증가
→ throughput 증가 가능

Inference serving에서 가장 기본적인 throughput 최적화 방법이다.

13. Dynamic Batching

Dynamic batching은 실시간으로 들어오는 요청을 짧은 시간 동안 모아서 batch로 처리하는 방식이다.

예를 들어 5ms 정도 기다렸다가 그 안에 들어온 요청들을 묶는다.

0ms 요청 A 도착
1ms 요청 B 도착
3ms 요청 C 도착
5ms batch 구성
모델 추론

이렇게 하면 latency를 너무 많이 희생하지 않으면서 throughput을 높일 수 있다.

Triton Inference Server 같은 도구에서 자주 등장하는 개념이다.

14. Quantization

Quantization은 모델의 숫자 표현을 줄이는 최적화다.

FP32 → FP16
FP32 → INT8

숫자 표현이 가벼워지면 메모리 사용량과 연산량이 줄어들 수 있다.

그 결과 더 많은 요청을 처리할 수 있다.

모델 가벼워짐
→ 추론 속도 증가 가능
→ throughput 증가 가능

다만 정확도 손실이 생길 수 있으므로 검증이 필요하다.

15. Mixed Precision

Mixed precision은 FP32와 FP16 같은 정밀도를 섞어서 사용하는 방식이다.

GPU가 FP16 연산에 강한 경우, mixed precision을 사용하면 더 빠르게 처리할 수 있다.

FP32만 사용
→ 안정적이지만 무거움

FP16 활용
→ 더 빠르고 메모리 절약 가능

Training에서도 쓰이고, inference에서도 쓰인다.

16. 전처리 병렬화

모델 forward만 빨라도 전체 throughput이 낮을 수 있다.

왜냐하면 전처리가 병목이 될 수 있기 때문이다.

예를 들어 이미지 inference라면 이런 작업이 있다.

이미지 로딩
디코딩
resize
normalize
tensor 변환
GPU로 이동

이 작업이 느리면 GPU가 기다리게 된다.

CPU 전처리 느림
↓
GPU idle
↓
throughput 낮음

그래서 전처리를 병렬화하거나, DataLoader의 num_workers를 조정하거나, CPU 병목을 줄이는 것이 중요하다.

17. Throughput이 낮아지는 흔한 원인

Throughput이 낮을 때는 여러 원인을 의심할 수 있다.

batch size가 너무 작음
GPU 활용률이 낮음
전처리가 느림
CPU 병목
GPU 메모리 부족
모델이 너무 큼
네트워크 병목
요청 큐 처리 비효율
동기 처리 구조
불필요한 CPU-GPU 데이터 이동

특히 AI inference에서는 GPU만 보면 안 된다.

전체 파이프라인을 봐야 한다.

요청 수신
전처리
GPU 이동
모델 추론
후처리
응답 반환

이 중 어디가 느린지 구간별로 측정해야 한다.

18. FastAPI에서 Throughput을 볼 때

FastAPI로 inference API를 만들었다면 단순히 요청 하나의 latency만 보면 부족하다.

부하 테스트를 해서 초당 몇 개 요청을 처리할 수 있는지 봐야 한다.

예를 들어 이런 도구를 쓸 수 있다.

JMeter
Locust
wrk
ab
k6

측정할 때는 이런 값을 본다.

requests/sec
avg latency
p95 latency
p99 latency
error rate
GPU utilization
GPU memory usage
CPU usage

Throughput만 높고 error가 많으면 의미가 없다.

좋은 서버
= throughput 높음
+ latency 안정적
+ error rate 낮음

19. Latency와 Throughput은 같이 봐야 한다

Throughput만 높이는 것은 쉽다.

요청을 오래 모아서 큰 batch로 처리하면 된다.

하지만 그러면 사용자는 오래 기다린다.

반대로 latency만 낮추는 것도 가능하다.

요청을 하나씩 바로 처리하면 된다.

하지만 그러면 GPU 효율이 낮아지고 throughput이 떨어질 수 있다.

그래서 실제 운영에서는 둘의 균형을 맞춘다.

낮은 latency
높은 throughput
낮은 error rate
적절한 GPU 비용

이 네 가지를 같이 봐야 한다.

20. 최종 정리

Throughput은 단위 시간 동안 시스템이 처리할 수 있는 작업량이다.

throughput = 처리한 작업 수 / 걸린 시간

AI inference에서는 보통 이렇게 본다.

requests/sec
samples/sec
images/sec
tokens/sec

Latency와 비교하면 이렇게 정리할 수 있다.

Latency
= 요청 하나가 얼마나 빨리 끝나는가

Throughput
= 서버가 얼마나 많은 요청을 처리할 수 있는가

Batch와 연결하면 이렇다.

batch size 증가
→ GPU 활용률 증가 가능
→ throughput 증가 가능
→ latency 증가 가능

AI DevOps 관점에서는 throughput이 비용과도 연결된다.

throughput 높음
→ GPU당 처리량 증가
→ 요청당 비용 감소 가능

한 문장으로 정리하면 이렇다.

Throughput은 AI 서버의 처리 체력이다.

latency가 사용자 한 명의 체감 속도라면, throughput은 서버가 전체 트래픽을 버티는 힘이다.
