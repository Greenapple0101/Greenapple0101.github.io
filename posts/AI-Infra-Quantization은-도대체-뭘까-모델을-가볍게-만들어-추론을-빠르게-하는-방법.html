quantization

처음 보면 이런 생각이 든다.

모델을 양자화한다는 게 뭐지?
숫자를 줄인다는 건가?
왜 빨라지는 거지?

한 줄로 말하면 이렇다.

Quantization = 모델에서 사용하는 숫자의 표현 방식을 더 가볍게 바꾸는 최적화 기법

1. Quantization 한 줄 정의

딥러닝 모델은 결국 숫자 덩어리다.

모델 안에는 수많은 가중치가 있다.

weight
bias
activation
embedding

이 값들은 보통 부동소수점 숫자로 저장된다.

예를 들어 PyTorch 모델은 기본적으로 float32, 즉 FP32를 많이 쓴다.

FP32 = 32비트 부동소수점

그런데 FP32는 정확하지만 무겁다.

그래서 이 숫자를 더 작은 표현으로 바꾸는 것이 quantization이다.

FP32 → FP16
FP32 → INT8
FP32 → INT4

즉, 양자화는 모델의 숫자를 더 적은 비트로 표현해서 메모리 사용량과 계산량을 줄이는 방법이다.

2. 왜 Quantization이 필요할까?

AI 모델은 점점 커지고 있다.

특히 LLM이나 이미지 모델은 파라미터 수가 많다.

파라미터가 많다
= 저장해야 할 숫자가 많다
= 메모리를 많이 먹는다
= 계산량도 커진다
= 추론 비용이 비싸진다

예를 들어 모델 가중치가 FP32로 저장되어 있으면 숫자 하나당 32비트를 사용한다.

만약 INT8로 바꾸면 숫자 하나당 8비트만 사용한다.

FP32 = 32비트
INT8 = 8비트

단순 계산하면 가중치 메모리를 약 1/4 수준으로 줄일 수 있다.

32비트 → 8비트
= 4배 작아짐

그래서 quantization을 하면 이런 장점이 생길 수 있다.

모델 크기 감소
GPU 메모리 사용량 감소
CPU 메모리 사용량 감소
추론 속도 향상 가능
throughput 증가 가능
요청당 비용 감소 가능
엣지 디바이스 배포 가능성 증가

3. FP32, FP16, INT8이 뭔데?

양자화를 이해하려면 숫자 표현 방식을 알아야 한다.

딥러닝에서 자주 나오는 타입은 이렇다.

타입 의미 특징
FP32 32비트 부동소수점 정확하지만 무거움
FP16 16비트 부동소수점 더 가볍고 빠름
BF16 16비트 부동소수점 계열 학습/추론에서 안정성 좋음
INT8 8비트 정수 매우 가볍고 빠를 수 있음
INT4 4비트 정수 더 작지만 정확도 손실 위험 큼

쉽게 말하면 이렇다.

FP32 = 숫자를 정밀하게 표현
FP16 = 조금 덜 정밀하지만 더 가벼움
INT8 = 정수 기반으로 훨씬 가볍게 표현

4. 비유로 이해하기

사진 파일을 생각하면 쉽다.

원본 고화질 이미지는 용량이 크다.

원본 이미지
= 품질 좋음
= 용량 큼
= 처리 무거움

압축 이미지는 용량이 작다.

압축 이미지
= 용량 작음
= 전송 빠름
= 품질 손실 가능성 있음

Quantization도 비슷하다.

FP32 모델
= 정밀함
= 무거움

INT8 모델
= 가벼움
= 빠를 수 있음
= 정확도 손실 가능성 있음

즉, quantization은 모델의 숫자 표현을 압축하는 것에 가깝다.

5. Quantization이 줄이는 것

Quantization이 주로 줄이는 것은 세 가지다.

모델 크기
메모리 사용량
연산 비용

1. 모델 크기 감소

FP32 모델을 INT8로 바꾸면 가중치 저장 크기가 줄어든다.

FP32 weight 100MB
↓
INT8 weight 약 25MB 수준 가능

물론 실제 파일 크기는 구조와 메타데이터에 따라 달라질 수 있다.

2. 메모리 사용량 감소

모델이 GPU 메모리에 올라갈 때 더 적은 공간을 차지한다.

GPU memory 절약
↓
더 큰 batch 가능
↓
throughput 증가 가능

3. 연산 비용 감소

하드웨어가 INT8 연산을 빠르게 지원하면 추론 속도가 빨라질 수 있다.

INT8 연산 지원 GPU/CPU
↓
더 빠른 inference 가능

6. Quantization이 무조건 빠른 건 아니다

중요한 점이 있다.

Quantization = 무조건 빨라짐

은 아니다.

빨라지려면 조건이 맞아야 한다.

하드웨어가 해당 연산을 잘 지원해야 함
런타임이 quantized 연산을 잘 최적화해야 함
모델 구조가 quantization에 적합해야 함
전처리/후처리 병목이 없어야 함
batch size와 메모리 이동 비용도 맞아야 함

예를 들어 INT8로 바꿨는데 사용하는 하드웨어가 INT8 가속을 잘 못 하면 기대만큼 빨라지지 않을 수 있다.

또는 모델 forward는 빨라졌지만 전처리, 네트워크, 토큰화가 병목이면 전체 latency는 크게 안 줄어들 수 있다.

모델 추론 시간만 줄어듦
전체 서비스 latency는 별로 안 줄어듦

이럴 수도 있다.

7. Quantization의 핵심 위험: 정확도 손실

Quantization의 가장 큰 리스크는 정확도 손실이다.

숫자를 더 거칠게 표현하기 때문이다.

예를 들어 원래 값이 이런 식이라고 하자.

0.123456
0.123457
0.123458

정밀한 FP32에서는 이 차이를 표현할 수 있다.

하지만 낮은 비트로 바꾸면 이런 값들이 비슷하게 뭉개질 수 있다.

0.12
0.12
0.12

그러면 모델의 출력이 달라질 수 있다.

그래서 quantization 후에는 반드시 검증해야 한다.

원본 모델 결과
양자화 모델 결과
정확도 차이
latency 차이
throughput 차이
메모리 사용량 차이

이걸 비교해야 한다.

8. Quantization 종류

Quantization은 크게 여러 방식이 있다.

입문 단계에서는 아래 세 가지를 먼저 잡으면 된다.

Dynamic Quantization
Static Quantization
Quantization Aware Training

9. Dynamic Quantization

Dynamic Quantization은 비교적 간단한 방식이다.

주로 weight를 미리 quantization하고, activation은 실행 시점에 동적으로 처리한다.

weight = 미리 INT8로 변환
activation = inference 중 동적으로 quantization

장점은 적용이 쉽다는 것이다.

적용 쉬움
별도 calibration 데이터가 적게 필요하거나 없음
CPU inference에서 유용한 경우가 많음

단점은 static quantization보다 최적화 효과가 제한적일 수 있다.

PyTorch에서는 이런 식으로 사용할 수 있다.

import torch

quantized_model = torch.quantization.quantize_dynamic(
    model,
    {torch.nn.Linear},
    dtype=torch.qint8
)

이 코드는 Linear 레이어를 중심으로 dynamic quantization을 적용하는 예시다.

LLM이나 NLP 계열에서 Linear layer가 많기 때문에 이런 방식이 자주 언급된다.

10. Static Quantization

Static Quantization은 weight뿐만 아니라 activation까지 미리 quantization 범위를 잡는다.

이를 위해 calibration 데이터가 필요하다.

대표 입력 데이터를 모델에 몇 번 흘려보냄
↓
activation 값의 범위를 관찰
↓
scale / zero point 계산
↓
INT8 모델로 변환

장점은 더 강한 최적화가 가능하다는 점이다.

weight도 quantization
activation도 quantization
하드웨어 가속 효과 기대

단점은 과정이 더 복잡하다.

calibration 데이터 필요
모델 구조 준비 필요
정확도 검증 필수

11. Calibration이란?

Calibration은 quantization을 위해 값의 범위를 잡는 과정이다.

예를 들어 어떤 레이어의 activation 값이 보통 이 범위에 있다고 하자.

-3.2 ~ 4.8

INT8은 표현할 수 있는 값의 개수가 제한적이다.

INT8 = 대략 256개 수준의 값 표현

그래서 실제 연속적인 실수 값을 INT8 범위에 매핑해야 한다.

실수 값 범위
↓
INT8 정수 범위로 매핑

이때 필요한 것이 scale과 zero point다.

12. Scale과 Zero Point

Quantization에서는 실수 값을 정수 값으로 바꿔야 한다.

대략적인 식은 이렇게 볼 수 있다.

real_value ≈ scale × (quantized_value - zero_point)

반대로 실수를 정수로 바꿀 때는 이런 느낌이다.

quantized_value ≈ real_value / scale + zero_point

여기서 중요한 것은 이것이다.

scale = 실수 범위를 정수 범위에 맞추는 비율
zero_point = 0을 정수 범위 안에서 어디에 둘지 정하는 값

너무 수학적으로 깊게 들어가지 않아도 된다.

핵심은 이렇다.

실수 값을 낮은 비트 정수로 바꾸려면
값의 범위를 정하고
그 범위를 정수 구간에 매핑해야 한다.

13. Quantization Aware Training, QAT

QAT는 Quantization Aware Training의 약자다.

한국어로는 양자화 인지 학습이라고 한다.

말 그대로 모델이 양자화될 것을 알고 학습하는 방식이다.

일반적인 학습은 FP32 기준으로 한다.

FP32로 학습
↓
나중에 INT8로 변환

하지만 QAT는 학습 중에 quantization 효과를 흉내 낸다.

학습 중에 양자화 오차를 시뮬레이션
↓
모델이 양자화 환경에 적응
↓
최종 INT8 변환 시 정확도 손실 감소 기대

장점은 정확도 보존에 유리하다는 점이다.

단점은 학습 과정이 필요하므로 더 복잡하다.

PTQ = 학습 후 양자화
QAT = 양자화를 고려해서 다시 학습

14. PTQ와 QAT 차이

Quantization을 크게 나누면 PTQ와 QAT로 볼 수 있다.

구분 PTQ QAT
의미 Post-Training Quantization Quantization Aware Training
시점 학습 끝난 뒤 적용 학습 중 또는 재학습 중 적용
난이도 비교적 쉬움 더 복잡함
데이터 calibration 데이터 필요할 수 있음 학습 데이터 필요
정확도 보존 모델에 따라 손실 가능 보통 더 유리
사용 상황 빠른 배포 최적화 정확도 손실을 줄이고 싶을 때

정리하면 이렇다.

PTQ = 이미 학습된 모델을 나중에 압축
QAT = 압축될 것을 고려해서 모델을 학습

15. LLM에서 Quantization

LLM에서는 quantization이 정말 중요하다.

이유는 모델이 너무 크기 때문이다.

예를 들어 수십억 개 파라미터를 가진 모델은 FP16으로도 GPU 메모리를 많이 먹는다.

7B 모델
13B 모델
70B 모델

이런 모델을 더 작은 GPU에서 돌리기 위해 quantization을 사용한다.

LLM 쪽에서는 이런 표현이 자주 나온다.

FP16
BF16
INT8
INT4
GPTQ
AWQ
GGUF
bitsandbytes

입문 단계에서는 이렇게 이해하면 된다.

LLM quantization
= 큰 언어모델을 더 작은 메모리로 돌리기 위해 가중치를 낮은 비트로 압축하는 것

예를 들어 INT4 quantization을 하면 4비트 수준으로 가중치를 표현한다.

FP16 대비 훨씬 작은 메모리
↓
개인 GPU나 엣지 환경에서도 실행 가능성 증가

하지만 정확도와 응답 품질이 떨어질 수 있으므로 모델별 검증이 필요하다.

16. TensorRT에서 Quantization

TensorRT를 쓸 때도 quantization이 중요하다.

TensorRT는 FP16, INT8 같은 precision 최적화를 지원한다.

흐름은 보통 이렇다.

PyTorch 모델
↓
ONNX export
↓
TensorRT engine build
↓
FP16 또는 INT8 최적화
↓
고속 inference

FP16은 비교적 쉽게 켤 수 있다.

trtexec \
  --onnx=model.onnx \
  --saveEngine=model_fp16.engine \
  --fp16

INT8은 보통 calibration이 필요하다.

trtexec \
  --onnx=model.onnx \
  --saveEngine=model_int8.engine \
  --int8

다만 실제 INT8 적용은 모델과 TensorRT 버전, calibration 방식, 데이터 준비에 따라 더 세부 설정이 필요할 수 있다.

핵심은 이렇다.

TensorRT quantization
= NVIDIA GPU에서 inference를 빠르게 하기 위해 precision을 낮추는 최적화

17. Quantization과 Latency

Quantization은 latency를 낮출 수 있다.

모델이 가벼워짐
↓
메모리 이동 감소
↓
연산량 감소
↓
추론 시간 감소 가능

예를 들어 원래 FP32 모델의 inference latency가 50ms였는데, INT8 최적화 후 25ms로 줄어들 수 있다.

물론 항상 이렇게 줄어드는 것은 아니다.

중요한 것은 실제 측정이다.

최적화 전 latency 측정
최적화 후 latency 측정
p50 / p95 / p99 비교

평균만 보면 안 된다.

운영에서는 p95, p99도 봐야 한다.

18. Quantization과 Throughput

Quantization은 throughput도 높일 수 있다.

모델 메모리 감소
↓
더 큰 batch 가능
↓
GPU 활용률 증가
↓
초당 처리량 증가 가능

LLM에서는 특히 tokens/sec가 좋아질 수 있다.

FP16 모델: 100 tokens/sec
INT8 모델: 180 tokens/sec

이런 식으로 같은 GPU에서 더 많은 토큰을 생성할 수 있다면 비용 효율이 좋아진다.

throughput 증가
= GPU당 처리량 증가
= 요청당 비용 감소 가능

19. Quantization과 Batch

Quantization은 batch와도 연결된다.

모델이 더 작아지면 GPU 메모리에 여유가 생긴다.

그러면 batch size를 키울 수 있다.

FP32 모델
= batch size 8까지 가능

INT8 모델
= batch size 16 또는 32 가능

batch size를 키우면 throughput이 좋아질 수 있다.

하지만 batch를 너무 키우면 latency가 증가할 수 있다.

quantization
→ 메모리 절약
→ batch size 증가 가능
→ throughput 증가 가능
→ latency와 trade-off 발생 가능

결국 실무에서는 quantization, batch size, latency, throughput을 같이 봐야 한다.

검증해야 한다.

21. Quantization을 언제 쓰면 좋을까?

Quantization이 특히 유용한 상황은 이렇다.

GPU 메모리가 부족할 때
모델을 더 빠르게 서빙하고 싶을 때
동일 GPU로 더 많은 요청을 처리하고 싶을 때
엣지 디바이스에 모델을 올리고 싶을 때
LLM을 작은 GPU에서 돌리고 싶을 때
추론 비용을 줄이고 싶을 때

반대로 조심해야 하는 상황도 있다.

정확도 손실을 절대 허용하기 어려울 때
모델 구조가 quantization에 민감할 때
검증 데이터가 부족할 때
하드웨어가 quantized 연산을 잘 지원하지 않을 때
변환 후 결과 검증 파이프라인이 없을 때

22. 전체 그림으로 정리

지금까지 공부한 개념을 연결하면 이렇게 된다.

Training
= PyTorch로 모델 학습
= FP32 또는 mixed precision 사용
= 가중치 업데이트

Inference
= 학습된 모델로 예측
= forward only
= latency / throughput 중요

ONNX
= 모델을 다른 실행 환경으로 옮기기 위한 포맷

TensorRT
= NVIDIA GPU에서 inference를 최적화하는 엔진

Quantization
= 모델 숫자 표현을 낮은 비트로 줄여
  메모리와 연산 비용을 줄이는 최적화

흐름으로 보면 이렇다.

PyTorch Training
↓
model.pth
↓
ONNX export
↓
TensorRT build
↓
FP16 / INT8 quantization
↓
TensorRT engine
↓
Inference Serving
↓
latency / throughput / accuracy monitoring

23. 최종 정리

Quantization은 모델의 숫자 표현을 더 작은 비트로 바꾸는 최적화 기법이다.

FP32 → FP16
FP32 → INT8
FP16 → INT4

이렇게 하면 모델이 가벼워질 수 있다.

모델 크기 감소
메모리 사용량 감소
추론 속도 향상 가능
throughput 증가 가능
비용 절감 가능

하지만 공짜는 아니다.

정확도 손실 가능
출력 품질 저하 가능
하드웨어/런타임 의존
검증 필수

한 문장으로 정리하면 이렇다.

Quantization은 모델의 숫자를 덜 정밀하게 표현하는 대신, 더 빠르고 가볍게 추론하기 위한 기술이다.

AI DevOps 관점에서는 quantization을 단순 압축 기술로 보면 안 된다.

quantization
= inference latency를 줄이고
= throughput을 높이고
= GPU 메모리와 비용을 아끼기 위한
= 모델 배포 최적화 전략

그래서 운영형 AI 시스템에서는 quantization 후에 반드시 성능과 품질을 같이 봐야 한다.

빠른가?
가벼운가?
품질은 유지되는가?
비용은 줄었는가?

이 네 가지를 통과해야 실제 서비스에 올릴 수 있다.
