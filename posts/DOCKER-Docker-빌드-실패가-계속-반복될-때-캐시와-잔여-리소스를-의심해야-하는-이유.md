Docker로 애플리케이션을 배포하다 보면 이상한 상황을 만날 때가 있다.

분명히 코드를 고쳤다.
Dockerfile도 수정했다.
docker-compose.yml도 확인했다.
그런데 계속 같은 빌드 실패가 반복된다.

이럴 때 처음에는 보통 이렇게 생각한다.

“내 코드가 아직 잘못됐나?”
“Dockerfile 문법이 틀렸나?”
“이미지 이름이나 경로가 잘못됐나?”

물론 실제로 코드나 설정 문제가 원인일 수도 있다.
하지만 Docker 환경에서는 **이전에 만들어진 캐시, 중단된 컨테이너, 사용하지 않는 이미지, 볼륨, 네트워크 같은 잔여 리소스** 때문에 이미 고친 문제가 계속 반복되는 것처럼 보일 때가 있다.

이번 트러블슈팅은 바로 그 문제였다.

## 문제 상황

Docker 빌드가 실패했다.

처음에는 단순한 빌드 오류라고 생각했다.
그래서 코드를 수정하고 다시 빌드했다.

하지만 문제가 계속 반복됐다.

```bash
docker compose build
```

다시 실행해도 같은 오류가 발생했다.

문제는 현재 코드만 보고 판단하면 안 되는 상황이었다.
Docker가 이전 빌드 결과를 캐시로 재사용하거나, 이전 compose 실행에서 남은 컨테이너와 네트워크가 현재 실행에 영향을 주고 있었기 때문이다.

즉, 실패 원인은 단순히 “현재 코드가 틀렸다”가 아니라 다음과 같은 쪽에 가까웠다.

```text
이전 빌드 캐시
이전 이미지
이전 컨테이너
이전 네트워크
이전 볼륨
이전 compose orphan 리소스
```

이런 것들이 남아 있으면 내가 수정한 내용이 제대로 반영되지 않거나, 이전 상태와 새 상태가 섞이면서 원인을 헷갈리게 만든다.

## Docker 캐시가 문제를 만들 수 있는 이유

Docker는 빌드 속도를 빠르게 하기 위해 레이어 캐시를 사용한다.

예를 들어 Dockerfile에 이런 내용이 있다고 해보자.

```dockerfile
COPY . .
RUN ./gradlew build
```

Docker는 각 단계의 결과를 레이어로 저장한다.
그리고 다음 빌드 때 변경이 없다고 판단하면 그 단계를 다시 실행하지 않고 기존 결과를 재사용한다.

이 기능은 평소에는 매우 유용하다.

하지만 트러블슈팅 중에는 오히려 문제를 만든다.

나는 분명히 뭔가를 고쳤는데, Docker는 이전 레이어를 재사용할 수 있다.
그러면 현재 수정사항이 빌드 과정에 제대로 반영되지 않은 것처럼 보이거나, 이미 해결한 문제가 계속 남아 있는 것처럼 보인다.

그래서 빌드 실패가 반복될 때는 한 번쯤 캐시를 완전히 무시하고 빌드해볼 필요가 있다.

```bash
docker compose build --no-cache
```

이 명령어는 기존 빌드 캐시를 사용하지 않고 처음부터 다시 이미지를 빌드한다.

## orphan 컨테이너도 문제가 될 수 있다

Docker Compose를 사용하다 보면 서비스 이름을 바꾸거나 compose 파일 구조를 변경할 때가 있다.

예를 들어 예전에는 서비스 이름이 `app`이었는데, 나중에 `backend`로 바꿨다고 해보자.

```yaml
services:
  backend:
    build: .
```

이 경우 예전에 떠 있던 `app` 컨테이너가 자동으로 깔끔하게 사라지지 않을 수 있다.

이렇게 현재 compose 파일에는 더 이상 정의되어 있지 않지만, 이전 실행 결과로 남아 있는 컨테이너를 orphan container라고 볼 수 있다.

이런 컨테이너가 남아 있으면 포트 충돌, 네트워크 충돌, 이름 충돌이 발생할 수 있다.

그래서 compose 환경을 정리할 때는 단순히 down만 하는 것이 아니라 orphan 리소스까지 같이 제거하는 것이 좋다.

```bash
docker compose down --remove-orphans
```

이 명령어는 현재 compose 파일 기준으로 더 이상 필요 없는 orphan 컨테이너까지 함께 제거한다.

## system prune으로 전체 Docker 환경 정리하기

캐시와 orphan만으로 해결되지 않을 때는 Docker에 남아 있는 불필요한 리소스를 더 넓게 정리해야 한다.

이때 사용한 명령어가 다음이다.

```bash
docker system prune -a --volumes -f
```

이 명령어는 강력하다.

정리 대상은 대략 다음과 같다.

```text
사용하지 않는 컨테이너
사용하지 않는 네트워크
사용하지 않는 이미지
빌드 캐시
사용하지 않는 볼륨
```

옵션을 나눠서 보면 이렇다.

```bash
docker system prune
```

사용하지 않는 Docker 리소스를 정리한다.

```bash
-a
```

사용 중이지 않은 이미지까지 더 적극적으로 삭제한다.

```bash
--volumes
```

사용하지 않는 볼륨까지 삭제한다.

```bash
-f
```

확인 질문 없이 바로 실행한다.

다만 이 명령어는 조심해서 써야 한다.

특히 `--volumes` 옵션은 DB 데이터가 들어 있는 볼륨도 삭제할 수 있다.
개발 환경에서는 괜찮을 수 있지만, 운영 서버나 중요한 데이터가 있는 환경에서는 함부로 쓰면 안 된다.

즉, 이 명령어는 “깔끔하게 다 밀고 다시 시작하자”에 가까운 명령어다.

## 최종적으로 고정한 표준 루틴

이번에는 Docker 빌드 실패가 단순히 한 번의 재시도로 해결되는 문제가 아니었다.

중요한 건 “에러가 나면 아무렇게나 다시 해본다”가 아니라, 실패를 제거하는 절차를 표준화하는 것이었다.

그래서 다음 루틴을 고정했다.

```bash
docker compose down --remove-orphans
docker system prune -a --volumes -f
docker compose build --no-cache
```

흐름으로 보면 이렇다.

```text
1. 기존 compose 리소스 정리
2. orphan 컨테이너 제거
3. 사용하지 않는 Docker 리소스 전체 정리
4. 캐시 없이 새로 빌드
```

이렇게 하니까 같은 실패를 반복해서 만나는 일이 줄어들었다.

중요한 건 이것이 단순한 “재시도”가 아니라는 점이다.

```text
재시도:
그냥 다시 빌드해본다

절차화:
이전 상태를 제거하고, 깨끗한 상태에서 다시 빌드한다
```

이 차이가 꽤 크다.

## 이 트러블슈팅이 중요한가?

결론부터 말하면, 중요하다.

엄청 화려한 장애 대응은 아니지만, 실무에서는 꽤 중요한 유형의 트러블슈팅이다.

왜냐하면 Docker 문제는 코드 문제처럼 보이지만 실제로는 환경 문제인 경우가 많기 때문이다.

특히 다음 상황에서 중요하다.

```text
EC2 서버에서 직접 Docker Compose로 배포할 때
이미지 빌드와 컨테이너 실행을 반복할 때
서비스 이름이나 포트 설정을 자주 바꿀 때
Dockerfile을 수정했는데 변경사항이 반영되지 않는 것처럼 보일 때
이전 컨테이너나 네트워크가 남아서 충돌이 날 때
```

이런 상황에서 무작정 코드를 계속 고치면 시간이 많이 낭비된다.

사실 문제는 코드가 아니라 Docker의 이전 상태일 수 있기 때문이다.

그래서 이 트러블슈팅의 핵심은 단순히 명령어 세 개를 외우는 것이 아니다.

핵심은 이것이다.

```text
Docker 빌드/배포 문제를 볼 때 현재 코드만 보지 말고,
이전 빌드 캐시와 잔여 리소스까지 함께 의심해야 한다.
```

## 실무적으로 더 좋은 표현

포트폴리오나 이력서에 쓴다면 이렇게 표현할 수 있다.

```text
Docker Compose 기반 배포 과정에서 빌드 캐시 및 잔여 리소스로 인해 동일한 빌드 실패가 반복되는 문제를 분석하고, --no-cache 빌드, orphan 컨테이너 제거, Docker system prune 절차를 표준화하여 재현 가능한 클린 빌드 루틴을 구축했습니다.
```

조금 더 간단히 쓰면 이렇게도 가능하다.

```text
Docker 빌드 실패가 캐시 및 잔여 리소스 영향으로 반복되는 문제를 해결하기 위해 --no-cache 빌드, orphan 제거, system prune을 표준 루틴으로 정리하여 재시도가 아닌 절차 기반으로 실패를 제거했습니다.
```

이건 충분히 트러블슈팅으로 쓸 만하다.

다만 “대규모 장애를 해결했다” 느낌보다는, **개발/배포 환경의 재현성과 안정성을 높인 트러블슈팅**으로 표현하는 게 좋다.

## 정리

이번 문제는 Docker 빌드 실패가 계속 반복되는 상황이었다.

단순히 코드를 다시 고치는 방식으로 접근하면 원인을 찾기 어려웠다.
문제는 현재 코드가 아니라 Docker에 남아 있던 이전 상태일 수 있었기 때문이다.

그래서 다음 명령어들을 사용했다.

```bash
docker compose build --no-cache
docker compose down --remove-orphans
docker system prune -a --volumes -f
```

이를 통해 빌드 캐시를 무시하고, orphan 컨테이너를 제거하고, 사용하지 않는 Docker 리소스를 정리했다.

결과적으로 빌드 실패를 단순 재시도로 넘기는 것이 아니라, 깨끗한 상태에서 다시 빌드하는 절차로 정리할 수 있었다.

이 트러블슈팅의 의미는 다음 한 줄로 정리할 수 있다.

```text
Docker 문제는 코드 문제가 아니라 이전 실행 상태의 문제일 수도 있으므로, 클린 빌드 루틴을 표준화하는 것이 중요하다.
```
