[Upstage] Upstage API 기반 RAG 경험과 AI DevOps 핵심 키워드 정리

2026.05.31

1. 들어가며

업스테이지 제품을 단순히 체험해본 것이 아니라, SSAFY 과정에서 RAG 구조를 구현하며 API 호출 형태로 직접 연동해본 경험이 있다. 이 과정에서 LangChain, Embedding, 문서 분할, 벡터 검색, LLM 호출 흐름을 직접 구성했고, Upstage API를 실제 서비스 로직 안에 연결해 문서 기반 질의응답 구조를 만들어보았다.

이 경험을 통해 AI 서비스는 단순히 LLM을 호출하는 것만으로 완성되는 것이 아니라는 점을 체감했다. 좋은 답변을 만들기 위해서는 문서를 어떻게 파싱하고, 어떤 단위로 나누고, 어떤 방식으로 임베딩하고, 어떤 문맥을 검색해 LLM에 전달할 것인지가 중요하다. 즉, 모델 자체의 성능뿐 아니라 데이터 구성, 검색 품질, 파이프라인 설계, 운영 안정성이 함께 맞물려야 실제 서비스 품질이 나온다.

이 문서에서는 해당 경험에서 등장하는 핵심 키워드들을 정리하고, 이를 업스테이지 AI DevOps 직무와 어떻게 연결할 수 있는지 정리한다.

2. 핵심 구조 요약

전체 흐름

문서 기반 RAG 시스템은 일반적으로 다음 흐름으로 구성된다.

문서 수집
→ Document Parsing
→ Text Chunking
→ Embedding 생성
→ Vector DB 저장
→ 사용자 질문 입력
→ 질문 Embedding 생성
→ 유사 문서 검색
→ 검색된 문맥을 LLM에 전달

→ 응답 반환
→ 품질 평가 및 운영 모니터링
````

이 구조에서 중요한 점은 LLM이 모든 정보를 알고 있다고 가정하지 않는다는 것이다. 대신 외부 문서를 검색해 필요한 문맥을 제공하고, LLM이 그 문맥을 바탕으로 답변하게 만든다.

# 3. RAG

## RAG란?

RAG는 Retrieval-Augmented Generation의 약자로, 검색 증강 생성이라고 부른다. LLM이 답변을 생성하기 전에 외부 문서나 데이터베이스에서 관련 정보를 검색하고, 검색된 정보를 함께 제공하여 답변 품질을 높이는 구조다.

단순 LLM 호출은 사용자의 질문만 모델에 전달한다. 반면 RAG는 질문과 함께 관련 문서를 전달한다.

```text
단순 LLM 호출:
사용자 질문 → LLM → 답변

RAG:
사용자 질문 → 관련 문서 검색 → 질문 + 검색 문서 → LLM → 답변

왜 중요한가?

LLM은 학습 시점 이후의 최신 정보나 기업 내부 문서, 사내 규정, 고객별 문서 내용을 기본적으로 알지 못한다. 따라서 실제 업무 자동화에서는 외부 문서 기반으로 답변하는 구조가 필요하다.

RAG를 사용하면 다음과 같은 장점이 있다.

  • 기업 내부 문서 기반 답변 가능
  • 최신 문서 반영 가능
  • 환각 감소
  • 근거 기반 답변 생성 가능
  • 모델 재학습 없이 데이터 업데이트 가능
  • 문서 기반 업무 자동화에 적합

면접에서 말할 포인트

RAG를 구현하면서 단순히 LLM을 호출하는 것보다, 검색된 문맥을 함께 제공했을 때 답변 품질이 훨씬 안정적이라는 점을 체감했다. 특히 기업 문서 기반 서비스에서는 모델 성능뿐 아니라 문서 파싱, 청킹, 임베딩, 검색 품질이 전체 답변 품질에 직접적인 영향을 준다고 이해했다.

4. LLM

LLM이란?

LLM은 Large Language Model의 약자로, 대규모 언어 모델을 의미한다. 사용자의 자연어 입력을 이해하고, 문맥에 맞는 답변을 생성하는 모델이다.

예를 들어 사용자가 질문을 입력하면 LLM은 그 질문의 의미를 해석하고, 학습된 언어 패턴과 제공된 문맥을 바탕으로 답변을 생성한다.

RAG에서 LLM의 역할

RAG 구조에서 LLM은 모든 정보를 직접 기억하는 역할이 아니다. 검색 시스템이 찾아온 문서를 바탕으로 최종 답변을 생성하는 역할을 한다.

검색 시스템: 관련 문서 찾기
LLM: 검색된 문서를 바탕으로 자연어 답변 생성

중요한 이유

LLM은 답변 생성 능력은 뛰어나지만, 근거 없는 답변을 생성할 수 있다. 이를 환각이라고 한다. RAG에서는 LLM에게 관련 문서를 함께 제공함으로써 답변의 근거를 강화할 수 있다.

면접에서 말할 포인트

LLM 자체의 성능도 중요하지만, 실제 서비스에서는 LLM이 어떤 문맥을 받아 답변하느냐가 중요하다고 느꼈다. 따라서 AI DevOps 관점에서는 LLM API가 안정적으로 호출되는지, 지연시간과 오류율은 어떤지, 특정 배포 이후 답변 품질이 떨어지지 않았는지까지 관리해야 한다고 생각한다.

5. Upstage API

Upstage API란?

Upstage API는 업스테이지가 제공하는 AI 기능을 서비스에서 호출할 수 있도록 만든 인터페이스다. 개발자는 직접 모델을 학습하거나 운영하지 않고도, API 호출을 통해 LLM, Embedding, Document Parsing 등의 기능을 사용할 수 있다.

API 기반 사용의 의미

API 기반 사용은 단순히 웹 화면에서 데모를 체험하는 것과 다르다. 실제 서비스 코드에서 요청을 보내고, 응답을 받아 비즈니스 로직에 연결하는 것을 의미한다.

서비스 코드
→ Upstage API 호출
→ 응답 수신
→ 후처리
→ 사용자에게 결과 제공

중요한 이유

AI 모델을 직접 배포하고 운영하는 것은 복잡하다. API 형태로 제공되면 개발자는 AI 기능을 빠르게 서비스에 붙일 수 있다. 또한 기능별 API가 분리되어 있으면, 필요한 기능만 조합하여 사용할 수 있다.

면접에서 말할 포인트

Upstage API를 RAG 구조 안에서 직접 호출해보면서, AI 기능이 서비스 로직에 어떻게 연결되는지 이해할 수 있었다. 단순한 모델 체험이 아니라, 문서 처리부터 임베딩, 검색, 답변 생성까지 서비스 흐름 안에서 API를 조합해본 경험이 있다.

6. API

API란?

API는 Application Programming Interface의 약자로, 서로 다른 소프트웨어가 정해진 방식으로 기능을 주고받기 위한 인터페이스다.

예를 들어 서비스 서버가 Upstage API에 요청을 보내면, Upstage의 AI 기능이 실행되고 결과가 응답으로 돌아온다.

Client / Server
→ API Request
→ AI Service
→ API Response

API에서 중요한 요소

API를 사용할 때는 단순히 호출이 되는지만 보는 것이 아니라 다음 요소들이 중요하다.

  • 요청 형식
  • 응답 형식
  • 인증 방식
  • 에러 코드
  • 지연시간
  • 재시도 정책
  • Rate Limit
  • 비용
  • 장애 대응
  • 모니터링 가능성

면접에서 말할 포인트

AI 기능도 결국 서비스에서는 API로 호출되는 외부 의존성이다. 따라서 AI DevOps에서는 API 호출의 성공률, 지연시간, 에러율, 비용, 응답 품질을 지속적으로 관리해야 한다고 생각한다.

7. LangChain

LangChain이란?

LangChain은 LLM 기반 애플리케이션을 만들기 위한 프레임워크다. LLM 호출, 문서 로딩, 청킹, 임베딩, 벡터 검색, 프롬프트 구성, 체인 연결 등을 쉽게 구성할 수 있도록 도와준다.

RAG에서 LangChain의 역할

RAG 시스템에서는 여러 단계가 필요하다.

문서 로딩
→ 청킹
→ 임베딩
→ 벡터 저장
→ 검색
→ 프롬프트 구성
→ LLM 호출

LangChain은 이런 단계를 하나의 흐름으로 연결할 수 있게 도와준다.

중요한 이유

RAG를 직접 구현하려면 여러 모듈을 직접 연결해야 한다. LangChain을 사용하면 문서 처리, 검색, LLM 호출 흐름을 빠르게 구성할 수 있다.

면접에서 말할 포인트

LangChain을 활용해 RAG의 전체 흐름을 빠르게 구성해보았고, 그 안에서 Upstage API를 연결해 실제 문서 기반 질의응답 로직을 만들어보았다. 이를 통해 LLM 애플리케이션은 단순 API 호출이 아니라 여러 단계의 파이프라인으로 구성된다는 점을 이해했다.

8. Embedding

Embedding이란?

Embedding은 텍스트를 숫자 벡터로 변환하는 과정이다. 컴퓨터는 문장의 의미를 직접 이해하지 못하기 때문에, 문장을 의미를 담은 숫자 배열로 바꿔야 한다.

예를 들어 다음과 같은 문장이 있다고 하자.

"업스테이지는 AI 플랫폼 회사다."

이 문장은 임베딩 모델을 거치면 다음과 같은 숫자 벡터로 변환된다.

[0.12, -0.45, 0.33, ...]

이 벡터는 문장의 의미를 수치적으로 표현한 것이다.

왜 필요한가?

RAG에서는 사용자 질문과 문서 조각이 의미적으로 얼마나 비슷한지 비교해야 한다. 이를 위해 질문과 문서를 모두 벡터로 변환하고, 벡터 간 유사도를 계산한다.

사용자 질문 → Embedding
문서 조각 → Embedding
두 벡터 간 유사도 계산

면접에서 말할 포인트

Embedding은 RAG에서 검색 품질을 결정하는 핵심 요소라고 생각한다. 같은 문서라도 어떤 임베딩 모델을 쓰는지, 어떤 단위로 청킹하는지에 따라 검색되는 문맥이 달라지고, 최종 답변 품질도 달라진다.

9. Vector

Vector란?

Vector는 숫자의 배열이다. RAG에서는 텍스트의 의미를 숫자 배열로 표현한 것을 벡터라고 한다.

문장 → Embedding Model → Vector

예를 들어 비슷한 의미를 가진 문장은 벡터 공간에서도 가까운 위치에 놓이게 된다.

"문서 기반 질의응답"
"RAG 기반 Q&A"

→ 의미가 비슷하므로 벡터 거리도 가까움

중요한 이유

벡터를 사용하면 단순 키워드 검색이 아니라 의미 기반 검색이 가능하다. 예를 들어 사용자가 “계약서에서 해지 조건 알려줘”라고 질문했을 때, 문서에 “계약 종료 사유”라고 적혀 있어도 의미적으로 가까운 문서를 찾을 수 있다.

면접에서 말할 포인트

벡터 검색은 키워드가 정확히 일치하지 않아도 의미적으로 가까운 문서를 찾을 수 있다는 점에서 문서 기반 AI 서비스에 중요하다고 생각한다.

10. Vector DB

Vector DB란?

Vector DB는 임베딩된 벡터를 저장하고, 사용자 질문과 가까운 벡터를 빠르게 검색하는 데이터베이스다.

일반 데이터베이스가 정확한 값이나 조건을 기준으로 검색한다면, Vector DB는 벡터 간 거리나 유사도를 기준으로 검색한다.

문서 Chunk
→ Embedding
→ Vector DB 저장

사용자 질문
→ Embedding
→ Vector DB에서 유사한 문서 검색

중요한 이유

RAG에서는 매번 전체 문서를 LLM에 넣을 수 없다. 비용과 토큰 제한 때문이다. 따라서 질문과 관련된 문서 조각만 빠르게 찾아야 한다. 이때 Vector DB가 필요하다.

면접에서 말할 포인트

Vector DB는 RAG에서 검색 성능과 응답 속도를 좌우하는 중요한 컴포넌트라고 이해하고 있다. 운영 관점에서는 검색 지연시간, 인덱스 업데이트, 데이터 동기화, 검색 정확도까지 관리해야 한다.

11. Document Parsing

Document Parsing이란?

Document Parsing은 PDF, 이미지, 스캔 문서, 워드 문서 등 비정형 문서에서 텍스트와 구조 정보를 추출하는 과정이다.

단순히 글자를 뽑는 것만이 아니라, 표, 문단, 제목, 레이아웃, 순서 등을 LLM이 이해할 수 있는 형태로 변환하는 것이 중요하다.

왜 중요한가?

기업 환경의 문서는 대부분 깔끔한 텍스트 파일이 아니다. 계약서, 영수증, 신청서, 보고서, 스캔 PDF, 표가 포함된 문서 등 다양한 형태로 존재한다.

이런 문서를 제대로 파싱하지 못하면 이후 단계인 청킹, 임베딩, 검색, 답변 생성 품질이 모두 떨어진다.

잘못된 Parsing
→ 잘못된 Chunking
→ 잘못된 Embedding
→ 잘못된 검색
→ 부정확한 답변

면접에서 말할 포인트

Document Parsing은 RAG 품질의 시작점이라고 생각한다. 특히 업스테이지 제품에서 인상 깊었던 점은 비정형 문서를 LLM이 활용 가능한 형태로 바꾸는 기능이 API로 제공된다는 점이었다. 이는 실제 기업 문서 자동화와 직접적으로 연결된다고 느꼈다.

12. OCR

OCR이란?

OCR은 Optical Character Recognition의 약자로, 이미지나 스캔 문서 안의 글자를 인식해 텍스트로 변환하는 기술이다.

예를 들어 종이 계약서를 스캔한 PDF는 컴퓨터 입장에서는 이미지에 가깝다. OCR을 사용하면 이 이미지 안의 글자를 추출해 검색하거나 분석할 수 있다.

Document Parsing과 OCR의 관계

OCR은 문서에서 글자를 인식하는 기술이고, Document Parsing은 문서 전체의 구조를 이해 가능한 형태로 변환하는 과정이다.

OCR:
이미지 속 글자 추출

Document Parsing:
글자 + 표 + 레이아웃 + 문서 구조 추출

중요한 이유

은행, 보험, 공공기관, 법무, 회계 등 문서가 많은 업무에서는 OCR과 Document Parsing이 매우 중요하다. 문서가 제대로 디지털화되어야 이후 자동화가 가능하다.

면접에서 말할 포인트

AI DevOps 관점에서 OCR은 단순 모델 기능이 아니라 운영 품질 관리 대상이라고 생각한다. OCR 정확도, 처리 실패율, 문서 유형별 성능, 처리 지연시간, 비용을 지속적으로 모니터링해야 실제 고객 환경에서 안정적으로 사용할 수 있다.

13. Text Chunking

Text Chunking이란?

Text Chunking은 긴 문서를 일정 단위로 나누는 과정이다. LLM이나 임베딩 모델에는 입력 길이 제한이 있기 때문에, 긴 문서를 그대로 넣을 수 없다.

따라서 문서를 의미 있는 단위로 나누어 저장하고 검색해야 한다.

긴 문서
→ 문단 단위 / 섹션 단위 / 토큰 단위로 분할
→ Chunk 생성

왜 중요한가?

Chunk 크기가 너무 작으면 문맥이 부족해진다. 반대로 너무 크면 불필요한 정보가 많이 섞여 검색 품질이 떨어지고 비용이 증가한다.

Chunk가 너무 작음:
문맥 부족

Chunk가 너무 큼:
노이즈 증가, 비용 증가

적절한 Chunk:
검색 정확도와 문맥 보존의 균형

면접에서 말할 포인트

RAG를 구현하면서 청킹 전략이 답변 품질에 직접적인 영향을 준다는 점을 알게 되었다. 같은 문서라도 어떤 단위로 나누느냐에 따라 검색되는 문맥이 달라지고, 최종 답변도 달라진다.

14. Similarity Search

Similarity Search란?

Similarity Search는 사용자 질문과 의미적으로 가까운 문서 조각을 찾는 검색 방식이다. RAG에서는 질문을 임베딩하고, Vector DB에 저장된 문서 벡터와 비교해 가장 유사한 문서를 찾는다.

질문 벡터
↔ 문서 벡터들
→ 가장 가까운 문서 Top-K 검색

중요한 이유

RAG의 답변 품질은 검색된 문서의 품질에 크게 의존한다. 관련 없는 문서가 검색되면 LLM은 잘못된 문맥을 바탕으로 답변할 수 있다.

주요 개념

Top-K

가장 유사한 문서 몇 개를 가져올지 정하는 값이다.

Top-K = 3
→ 가장 유사한 문서 3개 검색

유사도 점수

질문과 문서가 얼마나 가까운지 나타내는 점수다. 일반적으로 코사인 유사도 같은 방식이 사용된다.

면접에서 말할 포인트

RAG에서 검색 품질은 답변 품질의 기반이라고 생각한다. AI DevOps 관점에서는 검색 결과의 정확도, 응답 속도, Top-K 설정, 임베딩 모델 변경에 따른 품질 변화를 관리할 필요가 있다.

15. Prompt

Prompt란?

Prompt는 LLM에 전달하는 입력이다. RAG에서는 사용자 질문만 전달하는 것이 아니라, 검색된 문서와 답변 지시사항을 함께 전달한다.

시스템 지시사항
+ 검색된 문서
+ 사용자 질문
→ LLM 답변

중요한 이유

같은 문서를 검색하더라도 프롬프트를 어떻게 구성하느냐에 따라 답변 품질이 달라질 수 있다.

예를 들어 다음과 같은 지시를 추가할 수 있다.

주어진 문서 내용만 근거로 답변하세요.
문서에 없는 내용은 모른다고 답변하세요.
답변에는 근거 문장을 포함하세요.

면접에서 말할 포인트

RAG에서는 검색뿐 아니라 프롬프트 구성도 중요하다고 생각한다. 특히 기업 문서 기반 서비스에서는 문서에 없는 내용을 추측하지 않도록 프롬프트를 설계하는 것이 필요하다.

16. Hallucination

Hallucination이란?

Hallucination은 LLM이 사실이 아닌 내용을 그럴듯하게 생성하는 현상이다. 한국어로는 환각이라고도 한다.

예를 들어 문서에 없는 내용을 마치 있는 것처럼 답변하거나, 잘못된 근거를 만들어내는 경우가 있다.

왜 발생하는가?

LLM은 확률적으로 다음 단어를 생성하는 모델이기 때문에, 질문에 대한 정확한 근거가 없더라도 자연스러운 답변을 만들어낼 수 있다.

RAG와의 관계

RAG는 환각을 줄이기 위한 대표적인 방법이다. LLM이 외부 문서를 참고하도록 만들면 근거 없는 답변을 줄일 수 있다.

하지만 RAG를 사용한다고 해서 환각이 완전히 사라지는 것은 아니다. 검색된 문서가 부정확하거나, 문맥이 부족하거나, 프롬프트가 잘못 설계되면 여전히 잘못된 답변이 나올 수 있다.

면접에서 말할 포인트

RAG는 환각을 줄이는 데 효과적이지만, 운영 환경에서는 검색 품질과 답변 품질을 지속적으로 평가해야 한다고 생각한다. 특히 문서 기반 AI 서비스에서는 답변이 문서 근거에 충실한지 평가하는 체계가 필요하다.

17. AI Service Pipeline

AI Service Pipeline이란?

AI Service Pipeline은 AI 기능이 사용자 요청을 받아 결과를 반환하기까지의 전체 처리 흐름을 의미한다.

RAG 기반 서비스에서는 다음과 같은 파이프라인이 구성된다.

사용자 요청
→ 입력 검증
→ 질문 임베딩
→ 벡터 검색
→ 문맥 구성
→ LLM 호출
→ 응답 후처리
→ 사용자 응답
→ 로그 저장
→ 모니터링

중요한 이유

AI 서비스는 단순히 모델 하나만으로 동작하지 않는다. 여러 컴포넌트가 연결된 파이프라인으로 구성된다. 따라서 한 단계에서 문제가 생기면 전체 품질에 영향을 준다.

예를 들어 다음과 같은 문제가 생길 수 있다.

  • 문서 파싱 실패
  • 임베딩 생성 실패
  • 벡터 검색 지연
  • LLM API 타임아웃
  • 응답 형식 오류
  • 비용 급증
  • 특정 문서 유형에서 품질 저하

면접에서 말할 포인트

AI 서비스는 모델 중심이 아니라 파이프라인 중심으로 봐야 한다고 생각한다. AI DevOps는 이 파이프라인이 안정적으로 동작하도록 배포, 모니터링, 장애 대응, 품질 평가 체계를 만드는 역할이라고 이해하고 있다.

18. AI DevOps

AI DevOps란?

AI DevOps는 AI 서비스를 개발 단계에서 끝내지 않고, 실제 운영 환경에서 안정적으로 배포하고 관리하는 역할이다.

일반적인 DevOps가 애플리케이션의 배포, 모니터링, 장애 대응, 자동화를 담당한다면, AI DevOps는 여기에 AI 모델과 API, 데이터 파이프라인, 품질 평가까지 함께 다룬다.

AI DevOps에서 중요한 요소

  • 배포 자동화
  • API 호출 안정성
  • Latency 관리
  • Error Rate 관리
  • Timeout 관리
  • Retry 정책
  • 비용 모니터링
  • Token Usage 관리
  • 품질 회귀 탐지
  • 모델/API 버전 변경 영향 분석
  • 문서 처리 파이프라인 안정성
  • 고객 환경별 장애 대응

왜 중요한가?

AI 기능은 일반적인 웹 API보다 변동성이 크다. 같은 질문이라도 모델 버전, 프롬프트, 검색 문서, 임베딩 모델, 청킹 전략에 따라 답변이 달라질 수 있다. 따라서 운영 환경에서는 기능이 단순히 동작하는지만 보는 것이 아니라, 품질이 유지되는지도 확인해야 한다.

면접에서 말할 포인트

AI DevOps는 AI 기능이 실제 고객 환경에서 안정적으로 동작하도록 만드는 역할이라고 이해하고 있다. 특히 RAG 서비스에서는 API 호출 성공 여부뿐 아니라, 검색 품질과 답변 품질이 유지되는지까지 관리해야 한다고 생각한다.

19. Latency

Latency란?

Latency는 사용자가 요청을 보낸 뒤 응답을 받기까지 걸리는 시간이다. 한국어로는 지연시간이라고 한다.

사용자 질문 입력
→ 서버 처리
→ API 호출
→ 답변 생성
→ 사용자에게 응답

이 전체 시간 = Latency

AI 서비스에서 중요한 이유

LLM API, Embedding API, Document Parsing API는 처리 시간이 비교적 길 수 있다. 특히 문서가 크거나 요청량이 많으면 응답이 느려질 수 있다.

Latency가 길어지면 사용자는 서비스가 느리다고 느낀다. 기업 고객 환경에서는 업무 처리 속도에도 영향을 준다.

면접에서 말할 포인트

AI DevOps에서는 평균 지연시간뿐 아니라 p95, p99 같은 지표를 함께 봐야 한다고 생각한다. 일부 요청만 매우 느려져도 실제 사용자 경험에는 큰 영향을 줄 수 있기 때문이다.

20. p95

p95란?

p95는 전체 요청 중 95%가 이 시간 안에 처리되었다는 의미다.

예를 들어 p95 latency가 2초라면, 전체 요청의 95%는 2초 이내에 응답했고, 나머지 5%는 2초보다 오래 걸렸다는 뜻이다.

평균보다 p95가 중요한 이유

평균 latency만 보면 일부 느린 요청이 가려질 수 있다. 실제 운영에서는 대부분의 요청이 괜찮아도, 일부 요청이 지나치게 느리면 사용자 불만이 생긴다.

평균 latency:
전체적인 경향 확인

p95 / p99 latency:
느린 요청과 꼬리 지연 확인

면접에서 말할 포인트

AI API는 문서 크기나 토큰 수에 따라 지연시간 편차가 커질 수 있기 때문에 평균만 보는 것은 부족하다고 생각한다. p95, p99를 함께 보면서 사용자 경험에 영향을 주는 지연을 관리해야 한다.

21. Error Rate

Error Rate란?

Error Rate는 전체 요청 중 실패한 요청의 비율이다.

Error Rate = 실패 요청 수 / 전체 요청 수

예를 들어 1,000건의 요청 중 20건이 실패했다면 Error Rate는 2%다.

AI 서비스에서 발생할 수 있는 오류

  • API 인증 실패
  • 요청 형식 오류
  • Rate Limit 초과
  • Timeout
  • 서버 오류
  • 모델 응답 실패
  • Document Parsing 실패
  • Embedding 생성 실패
  • Vector DB 검색 실패

면접에서 말할 포인트

AI 서비스에서는 API 호출 실패가 곧 사용자 기능 실패로 이어질 수 있다. 따라서 AI DevOps에서는 Error Rate를 지속적으로 모니터링하고, 장애 발생 시 어느 단계에서 실패했는지 추적할 수 있어야 한다고 생각한다.

22. Timeout

Timeout이란?

Timeout은 요청이 일정 시간 안에 완료되지 않으면 실패로 처리하는 설정이다.

예를 들어 LLM API 응답을 30초까지 기다리도록 설정했는데 30초가 지나도 응답이 없으면 Timeout으로 처리한다.

왜 필요한가?

Timeout이 없으면 하나의 요청이 계속 대기 상태로 남아 서버 자원을 잡아먹을 수 있다. 이는 전체 서비스 장애로 이어질 수 있다.

AI 서비스에서 중요한 이유

AI API는 요청 내용에 따라 응답 시간이 크게 달라질 수 있다. 문서가 크거나 토큰 수가 많으면 응답이 오래 걸릴 수 있다. 따라서 적절한 Timeout 설정이 필요하다.

면접에서 말할 포인트

AI API를 운영할 때는 Timeout을 적절히 설정하고, Timeout 발생 시 사용자에게 적절한 메시지를 반환하거나 재시도 정책을 적용해야 한다고 생각한다.

23. Retry

Retry란?

Retry는 요청이 실패했을 때 다시 시도하는 정책이다.

예를 들어 일시적인 네트워크 문제로 API 호출이 실패했다면, 일정 시간 후 다시 호출할 수 있다.

중요한 이유

모든 실패가 영구적인 실패는 아니다. 일시적인 네트워크 오류, 서버 부하, 순간적인 Timeout은 재시도로 해결될 수 있다.

주의할 점

무조건 재시도하면 오히려 장애를 키울 수 있다. 이미 서버가 부하 상태인데 모든 요청을 반복하면 트래픽이 더 증가한다.

따라서 다음과 같은 방식이 필요하다.

  • 최대 재시도 횟수 제한
  • Exponential Backoff
  • Circuit Breaker
  • 재시도 가능한 오류와 불가능한 오류 구분
  • 중복 요청으로 인한 부작용 방지

면접에서 말할 포인트

AI API 호출 실패에 대해 무조건 재시도하는 것이 아니라, 오류 유형에 따라 재시도 가능 여부를 구분해야 한다고 생각한다. 특히 운영 환경에서는 Retry가 장애를 완화할 수도 있지만, 잘못 설계하면 장애를 증폭시킬 수도 있다.

24. Token Usage

Token Usage란?

Token Usage는 LLM API를 사용할 때 입력과 출력에 사용된 토큰 수를 의미한다. LLM은 텍스트를 토큰 단위로 처리한다.

입력 토큰:
사용자 질문 + 검색된 문서 + 프롬프트

출력 토큰:
LLM이 생성한 답변

왜 중요한가?

LLM API 비용은 보통 토큰 사용량과 관련이 있다. 검색 문서를 너무 많이 넣거나, 불필요하게 긴 프롬프트를 사용하면 비용이 증가하고 지연시간도 늘어난다.

RAG에서 Token Usage가 증가하는 경우

  • Chunk 크기가 너무 큼
  • Top-K가 너무 큼
  • 프롬프트가 불필요하게 김
  • 문서 중복이 많음
  • 답변 길이 제한이 없음

면접에서 말할 포인트

RAG에서는 검색 문맥을 많이 넣는다고 항상 좋은 것이 아니라고 생각한다. 필요한 문맥만 적절히 넣어 답변 품질, 비용, 지연시간 사이의 균형을 맞추는 것이 중요하다.

25. Cost Optimization

Cost Optimization이란?

Cost Optimization은 서비스 운영 비용을 효율적으로 관리하는 것이다. AI 서비스에서는 API 호출 비용, 토큰 비용, 인프라 비용, 저장소 비용이 모두 고려되어야 한다.

AI 서비스에서 비용이 발생하는 지점

  • Document Parsing API 호출
  • Embedding API 호출
  • LLM API 호출
  • Vector DB 저장 및 검색
  • 로그 저장
  • 모니터링 시스템
  • 서버 인프라

비용 최적화 방법

  • 중복 문서 임베딩 방지
  • 캐싱 적용
  • Chunk 크기 최적화
  • Top-K 조정
  • 불필요한 프롬프트 제거
  • 요청량 모니터링
  • 문서 변경분만 재처리
  • 모델/API 선택 기준 수립

면접에서 말할 포인트

AI DevOps에서는 단순히 서비스가 동작하는지만 보는 것이 아니라, 운영 비용이 과도하게 증가하지 않도록 관리해야 한다고 생각한다. 특히 RAG에서는 토큰 사용량과 문서 처리 비용을 함께 봐야 한다.

26. Quality Regression

Quality Regression이란?

Quality Regression은 기존에는 잘 동작하던 AI 서비스의 품질이 특정 변경 이후 떨어지는 현상이다.

예를 들어 다음과 같은 변경 후 품질 회귀가 발생할 수 있다.

  • 모델 버전 변경
  • 임베딩 모델 변경
  • 프롬프트 변경
  • 청킹 전략 변경
  • Vector DB 인덱스 변경
  • 문서 파싱 방식 변경
  • 배포 환경 변경

왜 중요한가?

일반적인 소프트웨어는 테스트 케이스가 통과하면 기능이 정상이라고 볼 수 있다. 하지만 AI 서비스는 답변이 확률적이고 품질 평가가 어렵다. 따라서 기능이 동작하더라도 답변 품질이 나빠질 수 있다.

평가할 수 있는 지표

  • 답변 정확도
  • 근거 문서 일치도
  • 검색 Recall
  • Answer Similarity
  • Faithfulness
  • Hallucination Rate
  • 사용자 피드백
  • 응답 지연시간
  • 실패율

면접에서 말할 포인트

AI 서비스에서는 배포 후 API가 정상 응답한다고 해서 충분하지 않다고 생각한다. 답변 품질이 이전보다 나빠지지 않았는지 평가하는 품질 회귀 탐지 체계가 필요하다고 생각한다.

27. Monitoring

Monitoring이란?

Monitoring은 서비스의 상태를 지속적으로 관찰하는 것이다. 운영 환경에서는 서비스가 정상적으로 동작하는지 지표와 로그를 통해 확인해야 한다.

AI 서비스에서 모니터링할 항목

  • 요청 수
  • 성공률
  • Error Rate
  • Latency
  • p95 / p99 latency
  • Timeout 발생 수
  • Retry 횟수
  • Token Usage
  • API 비용
  • 문서 처리 실패율
  • 검색 결과 품질
  • 답변 품질
  • 사용자 피드백

중요한 이유

AI 서비스는 여러 외부 API와 데이터 파이프라인에 의존한다. 따라서 문제가 발생했을 때 어느 단계에서 문제가 생겼는지 빠르게 파악해야 한다.

면접에서 말할 포인트

AI DevOps에서는 모델이나 API 자체만 보는 것이 아니라, 사용자 요청이 들어와 답변이 나가기까지 전체 파이프라인을 모니터링해야 한다고 생각한다.

28. Observability

Observability는 시스템 내부 상태를 외부 지표, 로그, 트레이스를 통해 이해할 수 있는 능력이다.

Monitoring이 “문제가 있는지 보는 것”이라면, Observability는 “왜 문제가 생겼는지 추적할 수 있는 것”에 가깝다.

29. Service Integration

Service Integration이란?

Service Integration은 외부 API나 모듈을 기존 서비스 로직에 연결하는 과정이다.

Upstage API를 사용할 때도 단순히 API를 호출하는 것에서 끝나는 것이 아니라, 기존 백엔드 로직과 연결해야 한다.

사용자 요청
→ 백엔드 서버
→ Upstage API 호출
→ 결과 후처리
→ DB 저장
→ 사용자 응답

중요한 이유

AI API는 독립적으로 존재하는 기능이 아니라, 실제 서비스 안에 들어가야 가치가 생긴다. 따라서 인증, 예외 처리, 응답 포맷, 장애 대응, 비용 관리까지 함께 고려해야 한다.

면접에서 말할 포인트

Upstage API를 실제 서비스 흐름 안에 연결해보면서, AI 기능을 서비스에 붙이는 과정에서 API 사용성, 응답 구조, 예외 처리, 운영 안정성이 중요하다는 점을 이해했다.

30. Business Document Automation

Business Document Automation이란?

Business Document Automation은 기업에서 사용하는 다양한 문서 업무를 AI로 자동화하는 것을 의미한다.

예를 들어 다음과 같은 업무가 있다.

  • 계약서 요약
  • 청구서 정보 추출
  • 신청서 검토
  • 내부 규정 질의응답
  • 고객 문서 분류
  • 보험 서류 처리
  • 금융 문서 분석
  • OCR 기반 데이터 입력 자동화

왜 업스테이지와 연결되는가?

업스테이지의 Document Parsing, OCR, Embedding, LLM API는 문서 기반 업무 자동화와 연결된다. 특히 기업 환경에서는 문서가 많고, 사람이 직접 확인해야 하는 반복 업무가 많다.

면접에서 말할 포인트

업스테이지를 단순 생성형 AI API 회사가 아니라, 실제 비즈니스 문서를 처리하고 자동화할 수 있는 AI 플랫폼 회사로 이해하고 있다. 특히 문서 기반 자동화는 RAG, OCR, Document Parsing, LLM이 함께 연결되어야 구현 가능하다고 생각한다.

31. API Response Structure

API Response Structure란?

API Response Structure는 API 호출 후 반환되는 응답의 구조를 의미한다.

예를 들어 응답에는 보통 다음과 같은 정보가 포함된다.

{
  "result": "...",
  "status": "success",
  "usage": {
    "input_tokens": 100,
    "output_tokens": 50
  }
}

중요한 이유

응답 구조가 일관적이면 기존 서비스에 연동하기 쉽다. 반대로 응답 형식이 복잡하거나 자주 바뀌면 후처리 로직이 불안정해진다.

AI 서비스에서 응답 구조가 중요한 이유

AI API의 응답은 단순 문자열이 아닐 수 있다. 토큰 사용량, 처리 결과, 에러 정보, 메타데이터 등을 함께 다뤄야 한다.

면접에서 말할 포인트

API 응답 구조가 명확하고 일관적이면 서비스 연동과 예외 처리가 쉬워진다. AI DevOps 관점에서도 응답 메타데이터를 활용해 비용, 지연시간, 실패 원인을 모니터링할 수 있다고 생각한다.

32. Developer Experience

Developer Experience란?

Developer Experience는 개발자가 제품이나 API를 사용할 때 느끼는 사용성이다. 문서가 명확한지, API 호출이 직관적인지, 오류 메시지가 이해하기 쉬운지 등이 포함된다.

중요한 이유

AI API가 아무리 성능이 좋아도 개발자가 사용하기 어렵다면 서비스에 적용하기 힘들다. 반대로 API가 명확하고 일관적이면 빠르게 기능을 붙이고 실험할 수 있다.

면접에서 말할 포인트

Upstage API를 사용하면서 개발자 관점에서 기능이 잘 분리되어 있고, API 응답 구조가 비교적 명확해 서비스 로직에 연결하기 쉽다고 느꼈다. 이는 실제 고객사나 개발팀이 AI 기능을 도입할 때 중요한 장점이라고 생각한다.

33. Model Performance와 Service Quality

Model Performance란?

Model Performance는 모델 자체의 성능을 의미한다. 예를 들어 LLM의 답변 능력, 임베딩 모델의 의미 표현 능력, OCR 모델의 인식 정확도 등이 해당된다.

Service Quality란?

Service Quality는 실제 사용자가 경험하는 서비스 품질이다. 모델 성능뿐 아니라 응답 속도, 안정성, 비용, 사용성, 장애 대응, 데이터 품질까지 포함한다.

차이점

Model Performance:
모델 자체가 얼마나 잘하는가

Service Quality:
실제 서비스에서 얼마나 안정적으로 가치를 제공하는가

중요한 이유

모델 성능이 좋아도 서비스 품질이 나쁠 수 있다. 예를 들어 답변은 정확하지만 너무 느리거나, 비용이 너무 높거나, 장애가 자주 발생하면 좋은 서비스라고 보기 어렵다.

면접에서 말할 포인트

AI 서비스는 모델 성능만으로 평가할 수 없다고 생각한다. 실제 운영 환경에서는 지연시간, 에러율, 비용, 품질 회귀, 사용자 피드백까지 함께 관리해야 한다.

34. Data Pipeline

Data Pipeline이란?

Data Pipeline은 데이터가 수집되어 처리되고, 저장되고, 서비스에 사용되기까지의 흐름이다.

RAG에서는 문서가 다음과 같은 파이프라인을 거친다.

문서 업로드
→ Parsing
→ Cleaning
→ Chunking
→ Embedding
→ Vector DB 저장
→ 검색
→ LLM 답변에 활용

중요한 이유

RAG 서비스 품질은 데이터 파이프라인 품질에 크게 의존한다. 잘못 파싱된 문서, 중복된 문서, 오래된 문서, 잘못 청킹된 문서는 답변 품질을 떨어뜨린다.

면접에서 말할 포인트

RAG를 구현하면서 데이터 파이프라인이 모델 호출만큼 중요하다는 점을 느꼈다. AI DevOps에서는 이 파이프라인이 안정적으로 동작하고, 데이터 변경이 검색 품질에 잘 반영되는지 관리해야 한다고 생각한다.

35. Deployment

Deployment란?

Deployment는 개발한 서비스를 실제 실행 환경에 배포하는 과정이다.

AI 서비스에서는 일반 백엔드 배포뿐 아니라 다음 요소도 함께 고려해야 한다.

  • API Key 관리
  • 환경 변수 관리
  • 모델/API 버전 관리
  • 프롬프트 버전 관리
  • Vector DB 연결
  • 외부 API 장애 대응
  • 배포 후 품질 검증

중요한 이유

AI 서비스는 배포 이후에도 품질이 변할 수 있다. 코드가 같아도 외부 API 버전이나 문서 데이터가 바뀌면 결과가 달라질 수 있다.

면접에서 말할 포인트

AI 서비스 배포에서는 기능 정상 여부뿐 아니라, 배포 이후 답변 품질과 API 호출 안정성까지 확인해야 한다고 생각한다. 기존 DevOps 경험을 AI API 운영 환경에 연결해 기여하고 싶다.

36. CI/CD

CI/CD란?

CI/CD는 코드를 자동으로 빌드, 테스트, 배포하는 흐름이다.

  • CI: Continuous Integration, 지속적 통합
  • CD: Continuous Delivery 또는 Continuous Deployment, 지속적 배포

AI 서비스에서 CI/CD가 필요한 이유

AI 서비스도 일반 서비스처럼 빠르게 변경되고 배포된다. 하지만 AI 서비스는 코드 테스트뿐 아니라 품질 평가도 필요하다.

예를 들어 다음 항목을 CI/CD에 포함할 수 있다.

  • API 호출 테스트
  • 프롬프트 변경 테스트
  • RAG 검색 결과 테스트
  • 답변 품질 평가
  • Latency 측정
  • Error Rate 확인
  • 비용 추정
  • Regression Test

면접에서 말할 포인트

AI DevOps에서는 CI/CD 파이프라인 안에 단순 코드 테스트뿐 아니라 RAG 품질 평가와 API 성능 검증도 포함할 수 있다고 생각한다. 이를 통해 배포 이후 품질 회귀를 줄일 수 있다.

37. Evaluation

Evaluation이란?

Evaluation은 AI 서비스의 결과가 얼마나 좋은지 평가하는 과정이다.

일반 소프트웨어는 정답이 명확한 경우가 많지만, LLM 답변은 자연어이기 때문에 평가가 어렵다.

RAG에서 평가할 수 있는 항목

  • 검색된 문서가 질문과 관련 있는가
  • 답변이 문서 내용에 근거하는가
  • 답변이 정확한가
  • 답변이 불필요하게 길지 않은가
  • 문서에 없는 내용을 지어내지 않았는가
  • 이전 버전보다 품질이 떨어지지 않았는가

대표 지표

  • Retrieval Recall
  • Answer Similarity
  • Faithfulness
  • Context Precision
  • Context Recall
  • Hallucination Rate
  • Human Evaluation

면접에서 말할 포인트

AI 서비스 운영에서는 정답/오답만 보는 테스트로는 부족하다고 생각한다. RAG에서는 검색 품질과 답변 품질을 나누어 평가하고, 배포 전후 품질 회귀를 탐지하는 체계가 필요하다.

38. Answer Similarity

Answer Similarity란?

Answer Similarity는 생성된 답변이 기준 답변과 의미적으로 얼마나 유사한지 측정하는 지표다.

단순히 문자열이 같은지를 보는 것이 아니라, 의미가 비슷한지를 평가한다.

예시

기준 답변:

업스테이지는 문서 기반 AI 자동화를 위한 API를 제공한다.

생성 답변:

업스테이지는 기업 문서를 처리하고 자동화할 수 있는 AI API를 제공한다.

두 문장은 표현은 다르지만 의미가 비슷하다. 이런 경우 Answer Similarity가 높게 나올 수 있다.

중요한 이유

LLM 답변은 매번 같은 문장으로 나오지 않을 수 있다. 따라서 단순 문자열 비교보다 의미 기반 평가가 필요하다.

면접에서 말할 포인트

LLM 서비스 평가는 정답 문장과 완전히 일치하는지만 보면 부족하다고 생각한다. 의미적으로 같은 답변인지 평가하는 Answer Similarity 같은 지표가 필요하다.

39. Faithfulness

Faithfulness란?

Faithfulness는 LLM의 답변이 제공된 문서 내용에 충실한지를 평가하는 지표다.

즉, 답변이 검색된 문서에 근거하고 있는지를 보는 것이다.

중요한 이유

RAG에서는 LLM이 검색된 문서를 바탕으로 답변해야 한다. 문서에 없는 내용을 추가로 만들어내면 신뢰도가 떨어진다.

예시

검색 문서에는 “A 상품의 환불 기간은 7일”이라고 되어 있는데, 답변에서 “환불 기간은 14일”이라고 하면 Faithfulness가 낮다.

면접에서 말할 포인트

문서 기반 AI 서비스에서는 답변이 그럴듯한 것보다 문서 근거에 충실한지가 더 중요하다고 생각한다. 따라서 Faithfulness 같은 지표를 통해 환각을 줄이고 신뢰도를 관리해야 한다.

40. Retrieval Recall

Retrieval Recall이란?

Retrieval Recall은 질문에 답하기 위해 필요한 문서가 검색 결과에 포함되었는지를 평가하는 지표다.

중요한 이유

RAG에서 LLM은 검색된 문서만 보고 답변한다. 필요한 문서가 검색되지 않으면 LLM이 아무리 좋아도 정확한 답변을 만들기 어렵다.

좋은 검색
→ 필요한 문서 포함
→ 좋은 답변 가능

나쁜 검색
→ 필요한 문서 누락
→ 답변 품질 저하

면접에서 말할 포인트

RAG 품질을 평가할 때 최종 답변만 보는 것이 아니라, 그 전에 검색이 잘 되었는지도 봐야 한다고 생각한다. 검색 단계의 Recall이 낮으면 답변 품질도 함께 떨어질 가능성이 크다.

41. API Key

API Key란?

API Key는 외부 API를 사용할 때 인증을 위해 사용하는 키다. Upstage API 같은 외부 서비스를 호출할 때도 API Key를 사용한다.

중요한 이유

API Key는 인증 정보이기 때문에 코드에 직접 노출하면 안 된다. GitHub에 올라가거나 로그에 찍히면 보안 문제가 생길 수 있다.

관리 방법

  • 환경 변수로 관리
  • .env 파일은 Git에 올리지 않기
  • Secret Manager 사용
  • 권한 최소화
  • 주기적 교체
  • 로그에 출력하지 않기

면접에서 말할 포인트

외부 AI API를 운영 환경에 붙일 때는 API Key 관리가 중요하다고 생각한다. 특히 GitHub나 로그에 노출되지 않도록 환경 변수와 Secret 관리 체계를 사용해야 한다.

42. Rate Limit

Rate Limit이란?

Rate Limit은 일정 시간 동안 API를 호출할 수 있는 횟수 제한이다.

예를 들어 1분에 100번까지만 호출할 수 있도록 제한될 수 있다.

중요한 이유

Rate Limit을 초과하면 API 호출이 실패할 수 있다. 사용자가 많아지거나 배치 작업이 동시에 실행되면 제한에 걸릴 수 있다.

대응 방법

  • 요청량 모니터링
  • Queue 사용
  • Backoff 적용
  • 캐싱
  • Batch 처리
  • 호출량 제한
  • 사용자별 요청 제한

면접에서 말할 포인트

AI API는 외부 서비스 의존성이 있기 때문에 Rate Limit을 고려해야 한다고 생각한다. 운영 환경에서는 요청량을 제어하고, 제한 초과 시 graceful하게 대응하는 구조가 필요하다.

43. Caching

Caching이란?

Caching은 자주 사용하는 결과를 저장해두고 재사용하는 방식이다.

AI 서비스에서는 동일한 문서 처리 결과, 임베딩 결과, 자주 묻는 질문의 답변 등을 캐싱할 수 있다.

중요한 이유

Caching을 사용하면 다음 효과가 있다.

  • 응답 속도 개선
  • API 호출 비용 감소
  • 외부 API 부하 감소
  • Rate Limit 대응
  • 사용자 경험 개선

RAG에서 캐싱 가능한 대상

  • 문서 Parsing 결과
  • 문서 Embedding 결과
  • 질문 Embedding 결과
  • 검색 결과
  • 자주 묻는 질문 답변
  • LLM 응답

면접에서 말할 포인트

RAG 서비스에서는 매번 모든 단계를 새로 수행하면 비용과 지연시간이 커질 수 있다. 따라서 변경되지 않는 문서의 Parsing, Embedding 결과는 캐싱하거나 재사용하는 것이 중요하다고 생각한다.

44. Logging

Logging이란?

Logging은 서비스에서 발생한 이벤트와 오류를 기록하는 것이다.

AI 서비스에서 필요한 로그

  • 요청 ID
  • 사용자 질문
  • 검색된 문서 ID
  • LLM 응답
  • API 호출 성공/실패
  • 에러 메시지
  • Latency
  • Token Usage
  • 비용 추정
  • 재시도 횟수

주의할 점

AI 서비스 로그에는 민감한 문서 내용이나 개인정보가 포함될 수 있다. 따라서 로그를 남길 때는 보안과 마스킹을 고려해야 한다.

면접에서 말할 포인트

AI 서비스에서는 로그를 통해 장애 원인뿐 아니라 품질 문제도 추적할 수 있다고 생각한다. 다만 문서 기반 서비스는 개인정보나 민감정보가 포함될 수 있기 때문에 로그 정책을 신중하게 설계해야 한다.

45. Security

Security가 중요한 이유

AI 서비스는 기업 내부 문서나 고객 데이터를 다루는 경우가 많다. 따라서 보안이 매우 중요하다.

고려해야 할 요소

  • API Key 보호
  • 문서 접근 권한 관리
  • 개인정보 마스킹
  • 로그 보안
  • 네트워크 보안
  • 저장 데이터 암호화
  • 전송 구간 암호화
  • 사용자별 권한 분리

RAG에서의 보안 문제

RAG에서는 사용자 질문에 따라 문서를 검색해 답변한다. 이때 사용자가 접근 권한이 없는 문서까지 검색되면 보안 사고가 될 수 있다.

따라서 검색 단계에서 권한 필터링이 필요하다.

면접에서 말할 포인트

문서 기반 AI 서비스에서는 정확도만큼 보안도 중요하다고 생각한다. 사용자가 볼 수 있는 문서만 검색되도록 권한 기반 Retrieval을 설계해야 한다.

46. Customer Environment

Customer Environment란?

Customer Environment는 실제 고객사가 서비스를 사용하는 환경을 의미한다. 고객마다 네트워크, 보안 정책, 문서 형식, 트래픽 패턴, 인프라 구성이 다를 수 있다.

중요한 이유

AI 서비스는 내부 개발 환경에서 잘 동작하더라도 고객 환경에서는 다른 문제가 발생할 수 있다.

예를 들어 다음 문제가 생길 수 있다.

  • 고객사 방화벽 문제
  • API 접근 제한
  • 문서 포맷 차이
  • 대용량 문서 처리 문제
  • 보안 정책 충돌
  • 네트워크 지연
  • 특정 시간대 트래픽 집중

면접에서 말할 포인트

AI DevOps는 제품이 고객 환경에서도 안정적으로 동작하도록 지원하는 역할이라고 생각한다. 특히 고객사마다 문서 형식과 운영 환경이 다르기 때문에, 장애 원인을 빠르게 분석하고 안정화하는 역량이 중요하다고 생각한다.

47. Upstage 제품을 바라보는 관점

단순 생성형 AI API가 아닌 이유

업스테이지 제품은 단순히 LLM 답변을 생성하는 API에 그치지 않는다고 느꼈다. Document Parsing, Embedding, LLM API처럼 문서 기반 AI 서비스에 필요한 기능들이 분리되어 있고, 이를 조합해 RAG 구조를 만들 수 있기 때문이다.

AI 플랫폼으로서의 의미

플랫폼이라는 것은 여러 기능을 조합해 실제 서비스를 만들 수 있어야 한다. 업스테이지 API는 문서 처리, 임베딩, 답변 생성이 각각 기능으로 제공되기 때문에 개발자가 필요한 흐름을 구성할 수 있다.

비즈니스 문서 자동화와의 연결

기업 환경에서는 문서 기반 업무가 많다. 계약서, 보고서, 신청서, 고객 문의, 내부 규정 등 다양한 문서를 AI가 이해하고 답변할 수 있다면 업무 자동화 가치가 크다.

면접에서 말할 포인트

업스테이지를 단순한 생성형 AI API 회사가 아니라, 실제 비즈니스 문서를 처리하고 자동화할 수 있는 AI 플랫폼 회사로 이해하고 있다. 특히 AI DevOps 관점에서는 이런 API들이 고객 환경에서 안정적으로 호출되고, 비용과 지연시간이 관리되며, 답변 품질이 유지되도록 만드는 일이 중요하다고 생각한다.

답변 구조

면접용 답변 예시

업스테이지 제품은 단순히 데모로 체험한 것이 아니라, SSAFY 과정에서 RAG 구조를 구현하면서 API 형태로 직접 연동해본 경험이 있습니다. 문서를 일정 단위로 나누고, 임베딩을 생성한 뒤, 사용자 질문과 유사한 문서를 검색해 LLM에 전달하는 흐름을 구성했습니다.

이 과정에서 단순히 LLM에 질문을 던지는 것과, 검색된 문맥을 함께 제공하는 것은 답변 품질에서 큰 차이가 있다는 점을 체감했습니다. 결국 AI 서비스의 품질은 모델 자체 성능뿐 아니라, 문서를 어떻게 파싱하고, 어떻게 청킹하고, 어떤 기준으로 검색하고, 어떤 문맥을 LLM에 전달하느냐에 크게 좌우된다고 느꼈습니다.

업스테이지 제품에서 인상 깊었던 점은 Embedding, Document Parsing, LLM API처럼 RAG 시스템에 필요한 기능들이 잘 분리되어 있다는 점이었습니다. 특히 비정형 문서를 LLM이 활용 가능한 형태로 바꾸는 Document Parsing 기능은 실제 기업 문서 자동화와 직접적으로 연결된다고 생각했습니다.

저는 업스테이지를 단순한 생성형 AI API 회사라기보다, 실제 비즈니스 문서를 처리하고 자동화할 수 있는 AI 플랫폼 회사로 이해하고 있습니다. AI DevOps 관점에서는 이런 API들이 운영 환경에서 안정적으로 호출되고, latency, error rate, timeout, retry, token usage, 비용, 품질 회귀가 관리되도록 만드는 일이 중요하다고 생각합니다. 제가 경험한 RAG 구현과 배포·모니터링 경험을 연결해서, 실제 고객 환경에서 AI 서비스가 안정적으로 동작하도록 기여하고 싶습니다.
